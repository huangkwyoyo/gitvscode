# MySQL 迁移 Doris 项目边界与规范

> 文档版本：v1.0  
> 日期：2026-05-31  
> 适用范围：MySQL → Apache Doris 数据迁移项目

---

## 一、项目范围边界

### 1.1 纳入迁移

- MySQL 源库中所有活跃 ODS/DWD/DWS/ADS 层数据表（含 DDL 与全量+增量数据）
- ETL 加工 SQL（含函数改写、存储过程等价转换）
- 调度任务中依赖 MySQL 的节点配置迁移

### 1.2 不纳入迁移

- 已下线/废弃超过 6 个月的表及任务
- 与 Doris 不适配的特殊字段（如 `bid_text` 等二进制大对象类型），经业务方确认后可剔除
- MySQL 特有的存储引擎层物理文件

### 1.3 待评审项

- 历史分区数据保留周期超出 Doris 集群存储容量的表
- 复杂存储过程且改写方案不明确的模块（需逐项评估）

## 二、Doris 建表规范

### 2.1 VC（虚拟集群）选择

| 场景 | VC选择 |
|------|--------|
| 默认情况 | VC1 |
| 独立资源隔离需求 | VC2 |
| 跨集群双写 | VC1 & VC2 |

> 参数化配置，按表归属数据库自动路由。

### 2.2 表名称规范

**原则**：Doris 限制总表数量（表个数 × 分区数 × 分桶数有上限），禁止每天一张表或每月一张表的命名方式。

| 规范 | 示例 |
|------|------|
| 日增量表 | `mysql.tab_xxx_d` |
| 月增量表 | `mysql.tab_xxx_m` |
| 全量表 | `mysql.tab_xxx_f` |

**补充要求**：

1. **设置分区保留周期**——历史分区无用后自动删除，避免元数据膨胀；
2. **小数据量表不建议建分区**——数据量较小的表直接使用非分区表。

### 2.3 分桶键及分桶数规范

| 规范项 | 要求 |
|--------|------|
| 分桶键选择 | 选择**离散度高的 1 个或多个字段**，避免数据倾斜 |
| 分桶个数 | 结合表数据量和集群 BE 节点数合理设置，避免过多或过少 |

## 三、SQL 语法差异对照（MySQL → Doris）

以下为已验证的 23 条差异及等价改写方案。所有 ETL SQL 迁移必须逐条对照：

| 序号 | MySQL 写法 | Doris 写法 | 风险 | 备注 |
|------|-----------|-----------|------|------|
| 1 | 库名大写 | 库名小写 | 🔴 高 | 全局替换 |
| 2 | `decode(col, v1, r1, v2, r2, default)` | `case when col=v1 then r1 when col=v2 then r2 else default end` | 🟡 中 | |
| 3 | `to_number(attr_value)` | `cast(attr_value as int)` 或 `attr_value * 1` | 🟡 中 | |
| 4 | `to_char(date_col, 'yyyymm')` | `DATE_FORMAT(date_col, '%Y%m')` | 🔴 高 | 格式串语法不同，注意 `%Y%m` vs `yyyymm` |
| 5 | `sysdate()` | `now()` | 🔴 高 | |
| 6 | `merge` / `update` | 两步 `insert` 替代 | 🔴 高 | 参考 `d_dept_d` 表实现范式 |
| 7 | `insert overwrite` | **禁止使用** | 🔴 高 | 存在不可预期的结果 |
| 8 | `full join` | **禁止使用** | 🔴 高 | 存在不可预期的结果，改用 `left join` + `union` + `right join` |
| 9 | `\|\|` 双竖线拼接 | `concat()` | 🟡 中 | |
| 10 | 特殊字段 `bid_text` | 不可直接 Stream Load | 🔴 高 | 须预处理或剔除 |
| 11 | 存储过程 | **不支持** | 🔴 高 | 需改写为调度工具脚本 |
| 12 | `nvl(col, 'default')` | 类型须严格一致 | 🟡 中 | decimal 场景：① nvl 填充 decimal 值；② 或用 `cast` 转 varchar 比较 |
| 13 | 整数除法 | 分子 × `1.000000` | 🟡 中 | 保证商结果精度为 decimal |
| 14 | `substr(str, 0, n)` | `substr(str, 1, n)` | 🔴 高 | Doris 从 **1** 开始计数 |
| 15 | `instr(col, 'x', 1, 1)` | `instr(col, 'x')` | 🟡 中 | **须严格测试数据一致性** |
| 16 | `delete ... or ...` | 拆为多次 `delete` | 🔴 高 | 明细表不支持 `or` |
| 17 | `months_between(d1, d2)` | `timestampdiff(month, d2, d1)` | 🟡 中 | 注意参数顺序 |
| 18 | 关键字作字段名 | 用反引号 `` `col` `` 包裹 | 🟡 中 | |
| 19 | 关联条件含子查询 | **不支持** | 🔴 高 | 改写为 CTE 或临时表 |
| 20 | 明细表 `delete ... where` | 仅支持 `and` + 比较符（`=, >, <, >=, <=, !=, in, not in`） | 🔴 高 | 不支持 `or`、`like`、`not like` |
| 21 | 主键表 `delete ... where` | 支持 `or`、`like`、`not like` | 🟡 中 | 主键表能力更全面 |
| 22 | `full join` 结果异常 | 显式拆写为 `left join` + `union` + `right join` | 🔴 高 | |
| 23 | 大查询内存溢出 | 调大 BE 内存参数，分批执行 | 🔴 高 | `fc_inc_fee_d` 已出现过 OOM |

## 四、数据入库方式

| 方式 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| **DataOs 直入** | 单文件 ≤ 2GB | 组件少，无中间文件，配置方便 | 大文件慢 |
| **文件落地 + Stream Load** | 任意大小 | 速度快，保留中间文件可校验 | 组件多，配置繁琐 |
| **curl Stream Load** | 批量/超大文件 | 速度最快，可脚本化 | 需手动管理 label 唯一性 |

## 五、质量保障

### 5.1 数据校验标准

| 校验层级 | 方法 | 通过标准 |
|---------|------|---------|
| 一级：行数 | `select count(*)` 两端对比 | 完全一致 |
| 二级：汇总值 | 各数值字段 `sum()` 对比 | 差值 < 0.01% |
| 三级：抽样明细 | 随机 100 行全字段比对 | 100% 一致 |

### 5.2 校验原则

1. SQL 改写后**必须严格测试数据一致性**（尤其涉及 `instr`、`substr`、`nvl` 等高风险函数）；
2. 迁移期间 MySQL 源表保留**至少 30 天只读**以支持回退；
3. 校验不通过的表标注差异原因，走评审决策。

## 七、风险与应对

| 风险 | 应对措施 |
|------|---------|
| Doris OOM | 调大 BE 内存参数；大表分批导入 |
| `full join` 结果异常 | 全局禁用，强制改写 |
| `insert overwrite` 数据丢失 | 全局禁用，改用 `truncate` + `insert` |
| 存储过程不支持 | 逐项评估，优先迁移核心链路 |
| 隐式类型转换 Bug | 所有 `nvl`/类型转换须显式处理并验证 |
| 表数量超限 | 合并小表，合理规划分区策略 |
| 特殊字段 load 失败 | 提前识别并预处理或剔除 |
