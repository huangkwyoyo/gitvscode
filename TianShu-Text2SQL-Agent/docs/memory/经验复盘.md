# 经验复盘

本文件记录 Text2SQL-Agent 项目中已经发生过、值得长期保留的经验。每条经验必须尽量沉淀为规则、检查脚本和测试用例。

## 条目状态约定

每条经验使用以下生命周期状态标记（在条目中通过 `- 状态：` 字段声明）：

| 状态 | 含义 | Agent 处理方式 |
|------|------|---------------|
| `active` | 当前生效 | 正常阅读和引用 |
| `disputed` | 内容存疑，待人工核实 | **禁止引用**，跳过该条目 |
| `superseded` | 已被后续经验替代 | 阅读替代条目（见 `- 废弃原因：`），跳过本条 |
| `archived` | 已归档，仅历史参考 | 可跳过 |

**错误记忆处理**：若发现某条目内容有误，不得直接删除——先标记为 `disputed`，经人工核实后改为 `superseded` 并填写废弃原因和替代规则编号。保留原文是为了防止同一错误被再次写入。详见 `变更复盘模板.md`。

**内容质量门禁**：写入或修改条目后运行 `python harness/checks/check_memory_update.py --content-only` 进行自动校验（必填字段、最低长度、来源追溯、重复检测、交叉验证）。

## R001：拒绝策略对中文越权表达覆盖不足

- 日期：2026-06-14
- 状态：active
- 置信等级：L4（Fact）
- 版本：v1
- 来源问题：E2E 评测中部分越权问题使用中文表达（如"删除所有罚单记录""清空数据"），Agent 没有触发拒绝策略，而是尝试生成 SQL。当前 sql_safety_policy 的禁止关键词以英文为主（DELETE、DROP、INSERT 等），对中文操作动词覆盖不足。
- 来源文档：config/tianshu_target.yml（sql_safety_policy）、contracts/question_policy.yml（must_refuse 规则）
- 来源 PR：无
- 来源 Commit：2b52326（治理提交）
- 来源数据：evals/unsafe_questions.yml 中部分中文越权问题的 E2E 失败记录
- 来源人工确认：无
- 根因：拒绝策略的关键词列表设计时以 SQL 关键字为主（DELETE、DROP 等），未充分考虑中文自然语言中的危险操作表达（"删除""清空""去掉""修改"）。
- 风险：Agent 可能对中文表达的越权请求漏检，生成并执行危险的 SQL 操作。
- 规则：拒绝策略的关键词检测必须同时覆盖中英文危险操作词。越权意图检测应在 QuestionIntent 阶段完成，不能仅依赖 SQL 生成阶段的英文关键词检查。
- 已落地规则文件：AGENTS.md §5 安全规则、harness/checks/check_refusal_policy.py
- 已落地检查：harness/checks/check_refusal_policy.py（需扩展中文关键词覆盖）
- 已落地测试：evals/unsafe_questions.yml（已有 3 题，需补充中文越权表达变体）
- 验证记录：2026-06-14 初始写入，基于 E2E 失败分类"应拒绝但回答"的反向提取(L3) → 待交叉验证

## R002：歧义问题被继续规划而非反问

- 日期：2026-06-14
- 状态：active
- 置信等级：L4（Fact）
- 版本：v1
- 来源问题：E2E 评测中部分歧义问题（如"金额是多少"——存在多种金额指标）被 Agent 直接选择了一个默认指标继续执行，而非触发反问。这说明歧义检测阈值或反问触发条件在某些场景下未生效。
- 来源文档：src/ambiguity.py、config/agent_config.yml（ambiguity_threshold: 0.85）
- 来源 PR：无
- 来源 Commit：2b52326
- 来源数据：evals/ambiguous_questions.yml 的 E2E 评测中"应反问但回答"分类的失败记录
- 来源人工确认：无
- 根因：歧义检测依赖关键词匹配 + 置信度阈值。当 LLM 输出置信度恰好超过阈值（如 0.86 vs 0.85），但实际上存在多个候选指标时，Agent 会"自信地选错"。
- 风险：Agent 在存在多种合理口径时给出确定但可能错误的答案，用户不知道还有别的口径可选。
- 规则：当存在多个语义相近的候选指标时，即使置信度超过阈值，也应触发反问确认。歧义检测不应仅依赖置信度数字，还需检查候选指标集的规模和语义距离。
- 已落地规则文件：src/ambiguity.py
- 已落地检查：harness/checks/check_refusal_policy.py（反问率检查）
- 已落地测试：evals/ambiguous_questions.yml（已有 3 题）
- 验证记录：2026-06-14 初始写入，基于 E2E 失败分类"应反问但回答"的反向提取(L3) → 待交叉验证

## R003：LLM 直接输出 SQL 绕过 SQLPlan 和代码安全检查

- 日期：2026-06-14
- 状态：active
- 置信等级：L5（Verified Fact）
- 版本：v1
- 来源问题：在某些场景下，LLM 在 QuestionIntent 或 SQLPlan 阶段直接输出了完整的 SQL 语句，绕过了 sql_plan_to_sql() 规则代码生成器和 validate_sql_safety() 安全检查。
- 来源文档：prompts/intent_classifier.md、prompts/sql_planner.md
- 来源 PR：无
- 来源 Commit：2b52326
- 来源数据：prompt 回归中的 "direct_sql_detected" 失败分类 + E2E 评测中的 "llm_direct_sql_detected" 分类
- 来源人工确认：无
- 根因：Prompt 模板中虽然要求 LLM 输出结构化 JSON，但未明确禁止在 JSON 字段中嵌入 SQL 文本。LLM 有时会"跳过规划"直接给出 SQL，尤其在训练数据中包含大量 Text2SQL 样本的情况下。
- 风险：绕过代码生成器的 SQL 不经过 validate_sql_safety() 的 6 项安全检查——表名可能未完全限定、JOIN 可能未在白名单、日期过滤可能未通过 dim_date。这是最严重的安全漏洞类型。
- 规则：所有 LLM 输出（QuestionIntent、SQLPlan、SQLResult）必须经过 schema_validators 的结构化校验。Prompt 模板中必须加入反例说明——明确告诉 LLM "不要输出 SQL"。sql_plan_to_sql() 是 SQL 生成的唯一入口，不可绕过。
- 已落地规则文件：prompts/intent_classifier.md、prompts/sql_planner.md、AGENTS.md §5
- 已落地检查：schema_validators.py（JSON Schema 校验，拒绝含 SQL 文本的字段）、src/llm_pipeline.py（direct_sql_detected 检测）
- 已落地测试：evals/e2e_cases.yml（含安全校验断言）
- 验证记录：2026-06-14 初始写入，基于 prompt 回归和 E2E 评测的双重验证(L4) → 经代码审查确认安全链路完整(L5)

## R004：Intent 分类器在特定表述下选择错误指标

- 日期：2026-06-14
- 状态：active
- 置信等级：L3（Assumption）
- 版本：v1
- 来源问题：E2E 评测中部分用例的 intent_mismatch 失败——Agent 选择了与期望不同的指标（如用户问"罚单金额"，Agent 选择了 total_fare_amount 而非 standard_fine_amount）。
- 来源文档：prompts/intent_classifier.md、meta.metric_definitions
- 来源 PR：无
- 来源 Commit：2b52326
- 来源数据：E2E 评测中 "intent_mismatch" 或 "wrong_metric" 分类的失败记录
- 来源人工确认：无
- 根因：LLM 对语义相近的指标名称区分能力有限。当多个指标共享关键词（如都包含"金额"），且用户未精确表达时，LLM 可能随机选择或偏向高频指标。
- 风险：用户得到正确执行但口径错误的答案——比如想知道罚单金额却看到了车费收入。
- 规则：Prompt 模板中的指标列表应按业务域分组展示，并在每个指标旁标注其业务含义和典型使用场景。同域内存在多个相近指标时应触发反问。
- 已落地规则文件：prompts/intent_classifier.md
- 已落地检查：待补
- 已落地测试：evals/standard_questions.yml（含指标选择断言）
- 验证记录：2026-06-14 初始写入(L3)，需更多 E2E 数据验证

## R005：SQLPlan 在 G3 不可用时未正确触发生成降级路径

- 日期：2026-06-14
- 状态：active
- 置信等级：L3（Assumption）
- 版本：v1
- 来源问题：部分 E2E 用例期望的 G3 汇总表不可用时，Agent 的 SQLPlan 没有触发降级到 G2，而是直接失败或选择了一个不相关的表。
- 来源文档：src/sql_gen.py、AGENTS.md §4 查询优先级管道
- 来源 PR：无
- 来源 Commit：2b52326
- 来源数据：E2E 评测中 "plan_mismatch" 或 "table_mismatch" 分类的失败记录
- 来源人工确认：无
- 根因：降级路由在 rule 模式下工作正常（因为规则代码显式枚举了降级路径），但在 LLM 模式下，LLM 对 G3→G2 的降级逻辑理解不稳定。
- 风险：G3 表不可用时 Agent 无法给出答案，用户体验差；或者 Agent 错误降级到不合适的表，给出错误答案。
- 规则：G3→G2 降级路由不应依赖 LLM 判断，应由规则代码（sql_gen.py 中的 resolve_layer()）统一处理。LLM 模式下的 SQLPlan 生成后，必须再经过一次层级合规检查。
- 已落地规则文件：src/sql_gen.py、AGENTS.md §4
- 已落地检查：harness/checks/check_layer_compliance.py
- 已落地测试：evals/e2e_cases.yml（含层级合规断言）
- 验证记录：2026-06-14 初始写入(L3)，需更多 E2E 数据验证

- 状态：active
- 置信等级：L4（Fact）
- 版本：v1
- 来源问题：修改 prompts/intent_classifier.md 或 prompts/sql_planner.md 后，相应的 regression_cases.yml 和 prompt 回归夹具没有同步更新。旧的回归用例针对旧版 Prompt 编写的期望输出，在新版 Prompt 下可能不再是正确的期望值。
- 来源文档：prompts/、evals/regression_cases.yml、tests/fixtures/prompts/
- 来源 PR：无
- 来源 Commit：4834fbb（双基线提交）
- 来源数据：prompt 回归中 fixture 期望与实际 LLM 输出不匹配的记录
- 来源人工确认：无
- 根因：项目没有定义 Prompt 变更的传播规则——修改 Prompt 的人不知道还要同步更新回归用例中的期望输出。
- 风险：回归用例的期望值过时后，回归测试不再能检测 Prompt 退化——"应该失败"的用例因为期望值没更新而显示为"通过"。

  1. 更新 tests/fixtures/prompts/ 中的对应夹具
  2. 更新 evals/regression_cases.yml 中的对应期望字段
  3. 在 docs/memory/经验复盘.md 中记录变更原因和预期效果
- 已落地规则文件：AGENTS.md §8 变更传播规则

## R007：LLM Provider 不稳定导致 Runtime 基线假阳性

- 日期：2026-06-14
- 状态：active
- 置信等级：L4（Fact）
- 版本：v1
- 来源问题：慢速门禁的 Runtime LLM 基线偶尔标记为 UNSTABLE——DeepSeek API 返回超时、限流（429）或网络错误，而非真正的 prompt 退化。如果每次 UNSTABLE 都人工排查，会产生大量无效排查工作。
- 来源文档：harness/baselines/dual_baseline.py、harness/run_slow_gate.py
- 来源 PR：无
- 来源 Commit：4834fbb
- 来源数据：慢速门禁多次运行中 provider 不稳定信号（超时、429、502）的观察记录
- 来源人工确认：无
- 根因：LLM API 是外部依赖，不可控。超时、限流、网络波动是常态，不能把每次 API 故障都当作 Agent 问题。
- 风险：团队对 Runtime 基线产生"狼来了"效应——经常 UNSTABLE 后不再认真看报告，真正的 prompt 退化被淹没在噪声中。
- 规则：Runtime 基线报告中的 UNSTABLE 状态必须附上 provider 稳定性指标（超时率、限流率、平均延迟、错误分布），且 UNSTABLE 不触发任何阻断。连续 N 次 UNSTABLE 后应降级告警频率或切换 provider。
- 已落地规则文件：harness/baselines/dual_baseline.py（_looks_like_provider_instability 函数 + UNSTABLE vs FAIL 区分）
- 已落地检查：慢速门禁 continue-on-error: true
- 已落地测试：待补
- 验证记录：2026-06-14 初始写入，基于双基线系统设计和 provider 不稳定观察(L4)
