# 经验复盘

本文件记录 Text2SQL-Agent 项目中已经发生过、值得长期保留的经验。每条经验必须尽量沉淀为规则、检查脚本和测试用例。

## 条目状态约定

每条经验使用以下生命周期状态标记（在条目中通过 `- 状态：` 字段声明）：

| 状态 | 含义 | Agent 处理方式 |
|------|------|---------------|
| `active` | 当前生效 | 正常阅读和引用 |
| `disputed` | 内容存疑，待人工核实 | **禁止引用**，跳过该条目 |
| `superseded` | 已被后续经验替代 | 阅读替代条目（见 `- 废弃原因：`），跳过本条 |
| `archived` | 已归档，仅历史参考 | 可跳过 |

**错误记忆处理**：若发现某条目内容有误，不得直接删除——先标记为 `disputed`，经人工核实后改为 `superseded` 并填写废弃原因和替代规则编号。保留原文是为了防止同一错误被再次写入。详见 `变更复盘模板.md`。

**内容质量门禁**：写入或修改条目后运行 `python harness/checks/check_memory_update.py --content-only` 进行自动校验（必填字段、最低长度、来源追溯、重复检测、交叉验证）。

## R001：拒绝策略对中文越权表达覆盖不足

- 日期：2026-06-14
- 状态：active
- 置信等级：L4（Fact）
- 版本：v1

- 来源文档：config/tianshu_target.yml（sql_safety_policy）、contracts/question_policy.yml（must_refuse 规则）
- 来源 PR：无

- 风险：Agent 可能对中文表达的越权请求漏检，生成并执行危险的 SQL 操作。
- 规则：拒绝策略的关键词检测必须同时覆盖中英文危险操作词。越权意图检测应在 QuestionIntent 阶段完成，不能仅依赖 SQL 生成阶段的英文关键词检查。

## R002：歧义问题被继续规划而非反问

- 日期：2026-06-14
- 状态：active
- 置信等级：L4（Fact）
- 版本：v1
- 来源问题：E2E 评测中部分歧义问题（如"金额是多少"——存在多种金额指标）被 Agent 直接选择了一个默认指标继续执行，而非触发反问。这说明歧义检测阈值或反问触发条件在某些场景下未生效。
- 来源文档：src/ambiguity.py、config/agent_config.yml（ambiguity_threshold: 0.85）
- 来源 PR：无
- 来源 Commit：2b52326
- 来源数据：evals/ambiguous_questions.yml 的 E2E 评测中"应反问但回答"分类的失败记录
- 来源人工确认：无
- 根因：歧义检测依赖关键词匹配 + 置信度阈值。当 LLM 输出置信度恰好超过阈值（如 0.86 vs 0.85），但实际上存在多个候选指标时，Agent 会"自信地选错"。
- 风险：Agent 在存在多种合理口径时给出确定但可能错误的答案，用户不知道还有别的口径可选。
- 规则：当存在多个语义相近的候选指标时，即使置信度超过阈值，也应触发反问确认。歧义检测不应仅依赖置信度数字，还需检查候选指标集的规模和语义距离。
- 已落地规则文件：src/ambiguity.py
- 已落地检查：harness/checks/check_refusal_policy.py（反问率检查）
- 已落地测试：evals/ambiguous_questions.yml（已有 3 题）

## R003：LLM 直接输出 SQL 绕过 SQLPlan 和代码安全检查

- 日期：2026-06-14
- 状态：active
- 置信等级：L5（Verified Fact）
- 版本：v1
- 来源问题：在某些场景下，LLM 在 QuestionIntent 或 SQLPlan 阶段直接输出了完整的 SQL 语句，绕过了 sql_plan_to_sql() 规则代码生成器和 validate_sql_safety() 安全检查。
- 来源文档：prompts/intent_classifier.md、prompts/sql_planner.md
- 来源 PR：无
- 来源 Commit：2b52326
- 来源数据：prompt 回归中的 "direct_sql_detected" 失败分类 + E2E 评测中的 "llm_direct_sql_detected" 分类
- 来源人工确认：无
- 根因：Prompt 模板中虽然要求 LLM 输出结构化 JSON，但未明确禁止在 JSON 字段中嵌入 SQL 文本。LLM 有时会"跳过规划"直接给出 SQL，尤其在训练数据中包含大量 Text2SQL 样本的情况下。
- 风险：绕过代码生成器的 SQL 不经过 validate_sql_safety() 的 6 项安全检查——表名可能未完全限定、JOIN 可能未在白名单、日期过滤可能未通过 dim_date。这是最严重的安全漏洞类型。
- 规则：所有 LLM 输出（QuestionIntent、SQLPlan、SQLResult）必须经过 schema_validators 的结构化校验。Prompt 模板中必须加入反例说明——明确告诉 LLM "不要输出 SQL"。sql_plan_to_sql() 是 SQL 生成的唯一入口，不可绕过。
- 已落地规则文件：prompts/intent_classifier.md、prompts/sql_planner.md、AGENTS.md §5
- 已落地检查：schema_validators.py（JSON Schema 校验，拒绝含 SQL 文本的字段）、src/llm_pipeline.py（direct_sql_detected 检测）
- 已落地测试：evals/e2e_cases.yml（含安全校验断言）
- 验证记录：2026-06-14 初始写入，基于 prompt 回归和 E2E 评测的双重验证(L4) → 经代码审查确认安全链路完整(L5)

## R008：跨表多指标在 Phase 1 被错误反问——IR 结构不够用

- 日期：2026-06-16
- 状态：active
- 置信等级：L5（Verified Fact）
- 版本：v1
- 来源问题：用户问"每天行程数和受伤人数"，Phase 1 的 `_check_multi_metric_merge_conditions()` 检测到 `trip_count`（dws_daily_trip_summary）和 `persons_injured`（dws_daily_crash_summary）的 planning_table 不一致，返回 `NEED_CLARIFICATION` 反问用户"请分别查询"。但用户的跨表问题是自然的、合理的——不应该因为系统架构限制而迫使改变提问方式。
- 来源文档：src/ir.py（AgentResponse）、src/agent.py（_plan_query_multi_metric、_check_multi_metric_merge_conditions）
- 来源 PR：无
- 来源 Commit：Phase 2A→2D 实现提交
- 来源数据：Phase 2A 测试 `test_cross_table_two_metrics_detected` 验证了 Phase 1 的跨表反问行为；Phase 2B 测试 `test_two_metrics_cross_table_is_multi_plan` 验证了拆分后的多计划结构
- 来源人工确认：无
- 根因：Phase 1 的三层 IR 有一个隐藏假设——`AgentResponse.plan: Optional[SQLPlan]` 是一对一的。当多个指标落在不同表时，单 SQLPlan 无法表达"查两张表各取所需"的语义。架构设计时未考虑跨表多指标场景，IR 结构本身不够用。
- 风险：如果不在 IR 层面解决，跨表多指标问题会被永久反问——用户被迫将一个问题拆成两个问题分别提问，体验差。更坏的情况是，LLM 可能"偷偷"只回答置信度最高的那个指标，静默丢失其他指标的信息。
- 规则：IR 结构应支持"一对多"映射——`AgentResponse.plans: list[UnifiedResponse]` 使单次问答可包含多个独立查询计划。每个子计划（UnifiedResponse）内部仍是完整的三层链路（SubIntent → SQLPlan → SQLResult），保持安全校验的完整性。旧字段 `plan` / `result` 保持不变以确保向后兼容。
- 已落地规则文件：src/ir.py（SubIntent、UnifiedResponse、AgentResponse.plans、AgentResponse.is_multi_plan、Strategy.UNSUPPORTED_MULTI_PLAN）
- 已落地检查：每个子 SQLPlan 独立走 `plan.validate()` + `validate_sql_safety()`（src/agent.py ask() Phase 2C 执行路径）
- 已落地测试：tests/test_mvp_agent.py TestPhase2BMultiPlanPlanning（5 测试）、TestPhase2CSerialExecution（5 测试）、TestPhase2DResultFusion（4 测试）
- 验证记录：2026-06-16 初始写入，基于 Phase 2A→2D 完整实现 + 35 测试全部通过(L5)

## R009：模板式融合对多计划结果的信息展示不充分——需要 LLM 融合但需安全后校验

- 日期：2026-06-16
- 状态：active
- 置信等级：L4（Fact）
- 版本：v1
- 来源问题：Phase 2D 实现的 `fuse_results()` 模板式融合能够正确列出每个子计划的统计摘要（行数、合计、日均、前3行样例），但用户需要自行对比两个子结果。例如"每天行程数和受伤人数"的融合回答列出 trip_count 和 persons_injured 各自的数据，但没有关联解读（如"1月份行程量在 76万~110万之间波动，受伤人数在 50~78 之间"）。
- 来源文档：src/explainer.py（fuse_results、_lookup_zh_names、_find_numeric_columns）
- 来源 PR：无
- 来源 Commit：Phase 2D 实现提交
- 来源数据：Phase 2D 测试 `test_chinese_answer_mentions_split` 验证了模板融合的拆分说明；Phase 2D 测试 `test_fuse_results_no_llm_no_join` 验证了无 LLM 特征
- 来源人工确认：无
- 根因：模板融合是确定性代码，它只能按固定模板描述"每个子结果各自怎么样"，无法做跨结果的对比分析。这是因为跨结果分析需要"理解"两个数据序列之间的关系（哪个更大、趋势是否一致、是否有关联），这超出了模板的能力边界。
- 风险：如果后续引入 LLM 融合，LLM 可能"编造"子结果中不存在的数据关系（幻觉），或"微调"数值使其看起来更合理（数值篡改）。LLM 也可能在融合时引入因果断言（"因为行程减少所以事故减少"）——这在技术上缺乏依据。
- 规则：模板融合是安全基线，不可移除。LLM 融合必须：
  1. 保留模板融合作为输出格式的兜底（当 LLM 融合失败或后校验不通过时自动回退）
  2. LLM 融合输出必须经过数值一致性后校验——LLM 引用的任何数值必须与 SQLResult 中的原始值一致
  3. LLM 的 Prompt 必须包含反例约束——"只描述已提供的数据，不要推断因果关系，不要修改任何数值"
- 已落地规则文件：src/explainer.py（fuse_results 模板融合——已完成）；LLM 融合待 Phase 3B
- 已落地检查：fuse_results 无 LLM 调用（当前阶段验证通过）；LLM 融合的数值一致性后校验待 Phase 3B
- 已落地测试：tests/test_mvp_agent.py TestPhase2DResultFusion（4 测试）
- 验证记录：2026-06-16 初始写入，基于 Phase 2D 模板融合实现的局限性分析(L4)

- 根因：Text2SQLAgent 最初设计为 CLI/REPL 单用户交互，没有考虑 HTTP 并发场景。Agent 内部的状态变量和单一 DuckDB 连接在多请求并发时不是线程安全的。
- 风险：并发 ask() 可能造成 DuckDB 查询互相干扰、状态串扰、连接竞争。每请求创建 Agent 又会造成连接生命周期失控和重复加载成本。

## R011：API 错误脱敏是防御深度——500 响应不能返回 traceback/SQL/数据库路径

- 日期：2026-06-20
- 状态：active
- 置信等级：L4（Fact）
- 版本：v1
- 来源问题：Phase 6B 建设 REST API 时，必须确保未预期异常（500）不会向客户端泄露 traceback、SQL、数据库路径或内部状态信息。这些信息可被攻击者用于侦察。
- 来源文档：src/api/errors.py（unhandled_exception_handler / validation_exception_handler）
- 来源 PR：无
- 来源 Commit：Phase 6B 实现
- 来源数据：Phase 6B 安全测试 test_500_no_traceback（验证 500 不含 Traceback）、test_answer_no_sql_keywords（验证不含 INSERT/DELETE 等）
- 来源人工确认：无
- 根因：FastAPI 默认的异常处理会返回 traceback（当 debug=True），且错误消息可能包含内部路径和模块名。裸异常的 str() 可能包含数据库路径、表名等敏感信息。
- 风险：500 响应泄露 traceback 可暴露文件系统路径、使用的库版本、代码结构。泄露数据库路径可帮助攻击者定位数据文件。泄露 SQL 可暴露业务逻辑和数据结构。
- 规则：所有 500/503 响应必须使用安全错误 envelope（ApiErrorResponse）。错误消息必须是用户友好的中文模板，不包含任何内部信息。expose_internal_errors 默认必须为 false。unhandled_exception_handler 必须拦截所有未被捕获的异常。

- 已落地检查：tests/test_api_security.py::TestNoTracebackLeak、TestNoSQLLeak、TestNoDbPathLeak
- 已落地测试：tests/test_api_security.py（7 项安全测试）
