# 风险清单

本文件记录当前仍需持续关注的 Text2SQL-Agent 项目风险。风险不等于错误，但需要在进入下一阶段前确认是否已有规则、检查和测试覆盖。

## 高风险

| 风险编号 | 风险 | 影响 | 当前防线 |
|---|---|---|---|
| RISK-001 | LLM 直接输出 SQL 绕过 SQLPlan 和代码安全检查 | Agent 生成未经安全校验的 SQL——可能访问非 Gold 表、使用未核准 JOIN、绕过 dim_date 过滤 | schema_validators.py（JSON Schema 校验）、sql_plan_to_sql() 作为 SQL 唯一生成入口、prompt 反例说明 |
| RISK-002 | Prompt 修改后回归用例未同步更新 | 回归测试期望值过时，无法检测 Prompt 退化——"应该失败"的用例显示为"通过" | AGENTS.md §8 变更传播规则、Memory Gate 检查 |
| RISK-003 | LLM Provider 不稳定导致 Runtime 基线假阳性 | 团队对慢速门禁产生"狼来了"效应，真正的问题被淹没在 API 噪声中 | _looks_like_provider_instability() 检测 + UNSTABLE vs FAIL 区分 + continue-on-error |
| RISK-004 | 拒绝策略对中文越权表达覆盖不足 | Agent 可能对"删除""清空""修改"等中文危险操作漏检 | check_refusal_policy.py、sql_safety_policy 关键词列表 |
| RISK-005 | 歧义检测在置信度边界失效 | Agent 在多个候选指标存在时"自信地选错"，用户不知道还有其他口径 | ambiguity.py 阈值逻辑、反问模板 |
| RISK-006 | E2E 评测用例覆盖不全，新功能缺少评测 | 新加的 Prompt 规则、反问策略、安全规则没有对应的评测断言 | evals/ 五类评测体系、Memory Gate 变更检测 |

## 中风险

| 风险编号 | 风险 | 影响 | 当前防线 |
|---|---|---|---|
| RISK-007 | DuckDB SQL 方言与标准 SQL 差异 | LLM 可能生成 MySQL/PostgreSQL 方言的 SQL，DuckDB 无法执行 | sql_plan_to_sql() 规则代码（不依赖 LLM 生成 SQL）、DuckDB 方言适配 |
| RISK-008 | TianShu 数据仓库 schema 变更后 Agent 未感知 | Agent 使用的表/字段/指标可能已过期或改名 | resolver.py 启动时动态发现（读 information_schema + meta.metric_definitions） |
| RISK-009 | 指标口径漂移——TianShu 修改了指标定义但 eval 用例期望值未更新 | E2E 评测出现假阳性失败——Agent 正确但期望值过期 | 双基线：Source（代码正确）+ Runtime（AI 稳定），分离工程质量和 LLM 质量 |
| RISK-010 | LLM 模型更新后 Prompt 兼容性未知 | 新模型可能对同一 Prompt 产生不同的输出格式或推理逻辑 | 慢速门禁（真实 LLM 回归）、prompt 回归报告 |
| RISK-011 | CI 快速门禁全部用 Mock——Mock 与真实 LLM 行为不同 | Mock 模式下通过但真实 LLM 模式下失败的"漏检" | 慢速门禁用真实 LLM（仅观察不阻断）作为兜底 |
| RISK-012 | pre-commit hook 被 `--no-verify` 频繁绕过 | 门禁形同虚设，关键变更未检查即合入 | pre-commit 引导式阻断（展示失败原因和修复步骤，降低绕过动机） |
| RISK-013 | 记忆系统条目被错误写入——AI 把推测当成事实 | 下游 Agent 基于错误记忆做出错误决策，错误认知被固化为规则 | Memory Gate 内容质量门禁（必填字段/来源追溯/交叉验证）、proposed → active 冷却期、disputed/superseded 生命周期 |

## 新增风险（Phase 2 完成后暴露）

| 风险编号 | 风险 | 影响 | 当前防线 | 目标阶段 |
|----------|------|------|----------|:--------:|
| RISK-018 | 并行执行时 DuckDB 连接竞争——多线程共享单连接可能导致崩溃或数据错乱 | 并行多计划执行时结果不可靠，串行不受影响 | ← 尚无（当前串行执行无此风险）| 3A |
| RISK-019 | 并行执行超时后线程未正确回收 | 线程泄露，积累后影响后续查询性能甚至耗尽资源 | ← 尚无 | 3A |
| RISK-020 | LLM 融合时篡改或编造数值——即使 Prompt 要求忠实于数据，LLM 仍可能"创造性"修改 | 用户得到看似合理但数值错误的融合答案 | ← 尚无（当前模板融合无此风险）| 3B |
| RISK-021 | LLM 融合 JSON 解析失败导致整个多计划回答不可用 | 多计划查询全部执行成功但因融合环节失败而无法返回给用户 | ← 尚无（需模板融合作为 fallback）| 3B |
| RISK-022 | 多结果按 date merge 时 grain 不一致导致错误对齐 | trip 表按 trip_date，crash 表按 crash_date——某天有行程无事故时如何对齐？错误对齐产生虚假对比 | ← 尚无 | 3C |
| RISK-023 | 跨域校验误报导致用户对系统失去信任 | 正常业务波动被标记为"异常"，用户产生"狼来了"效应后忽视真正异常 | ← 尚无 | 3D |
| RISK-024 | LLM 在跨域因果解释中编造因果链——将时间巧合断言为因果关系 | LLM 生成"因为行程量下降所以事故减少"，但二者可能都是天气变差导致的共同结果 | ← 尚无 | 4 |
| RISK-025 | 图表生成时中文乱码或日期格式错误 | 图表本应直观展示趋势，乱码后反而比纯文本更难理解 | ← 尚无 | 5 |

## 新增风险（Phase 6B 完成后发现）

| 风险编号 | 风险 | 影响 | 当前防线 | 目标阶段 |
|----------|------|------|----------|:--------:|
| RISK-026 | REST API 的 AgentRuntime 使用单 Agent + 异步锁串行化，吞吐量上限为 ~1 QPS | 高并发场景下请求排队，超过 2 秒返回 429；无法水平扩展 | asyncio.Lock + 429 限流 + Phase 6C 固定窗口 Rate Limiter | ✅ 6C |
| RISK-027 | DuckDB 长查询无法在 Python 线程中真实中断 | 恶意或异常查询（如全表扫描大表）永久占用 worker 线程，后续请求全部 429 | ThreadPoolExecutor 单 worker + DuckDB connect_timeout 兜底 | ⚠️ 持续关注 |
| RISK-028 | API 错误 500 脱敏可能掩盖真实问题——运维人员无法从客户端响应中诊断 | 开发/运维排查困难，需登录服务器查看日志 | expose_internal_errors=false + Phase 6C 审计记录 error_code | ✅ 6C |
| RISK-029 | is_online 判定依赖 DuckDB 连接状态——网络抖动可能误判 offline | 间歇性 503 影响服务可用性，健康检查不稳定 | is_connected 属性 + 健康检查轮询 | ⚠️ 持续关注 |
| RISK-030 | 本地 127.0.0.1 绑定假设网络层已隔离——如果被反向代理暴露到公网则无认证保护 | 任何人可调用 /v1/ask 执行查询 | Phase 6C 本地令牌认证（X-TianShu-Token + hmac.compare_digest）| ✅ 6C |
