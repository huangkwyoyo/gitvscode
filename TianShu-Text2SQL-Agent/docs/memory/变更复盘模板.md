# 变更复盘模板

每次发现新的问题后，按本模板写入 `经验复盘.md`，再判断是否需要更新规则、脚本和测试。

## 模板

```md
## RXXX：规则标题

- 日期：YYYY-MM-DD
- 状态：proposed（提案）| active（活跃）| disputed（存疑）| superseded（已替代）| archived（归档）
- 置信等级：L5（Verified Fact）| L4（Fact）| L3（Assumption）| L2（Hypothesis）| L1（Human Reviewed）
- 版本：v1（初始版本号；同编号修订时递增为 v2, v3...）
- 来源问题：[具体描述发生了什么，如何发现的——必填，≥20 字]
- 来源文档：[关联的提示模板 / 配置文件 / 设计文档路径，如无可写"无"]
- 来源 PR：[关联的 PR 编号，如无可写"无"]
- 来源 Commit：[关联的 git commit hash，如无可写"无"]
- 来源数据：[关联的 E2E 评测结果 / prompt 回归报告 / LLM raw output，如无可写"无"]
- 来源人工确认：[确认人姓名 + 确认日期 + 确认内容，如无可写"无"]
- 根因：[为什么发生]
- 风险：[如果不修复会导致什么后果]
- 规则：[沉淀出的可执行规则]
- 已落地规则文件：[对应的 AGENTS.md / prompts / 检查脚本文件]
- 已落地检查：[对应的 check_*.py 脚本]
- 已落地测试：[对应的 test_*.py 用例或 evals YAML]
- 废弃原因：（仅在状态为 superseded 时填写）为何废弃、被哪条规则替代
- 验证记录：
  - YYYY-MM-DD：初始写入，基于[证据来源] → [初始置信等级]
  - YYYY-MM-DD：[交叉验证 / 人工审核] → [调整后置信等级]（如有调整）
```

## 字段说明

| 字段 | 必填 | 说明 |
|------|:----:|------|
| 日期 | ✅ | 记录写入日期，非问题发生日期 |
| 状态 | ✅ | 初始为 `proposed`（提案）；经 Harness 门禁或人工确认后转为 `active`；若事后发现错误则改为 `disputed` 或 `superseded` |
| 置信等级 | ✅ | **初始写入时不得高于 L3（Assumption）**——除非有 E2E 评测多轮数据或 prompt 回归报告等多源交叉验证支撑 |
| 版本 | ✅ | 初始为 `v1`；同编号修订时递增并在废弃原因中说明变更内容 |
| 来源问题 | ✅ | 具体描述发生了什么、如何发现，≥ 20 字 |
| 来源文档 | 建议 | 关联的 prompt 模板 / 配置文件 / 设计文档路径 |
| 来源 PR | 建议 | 关联的 PR 编号 |
| 来源 Commit | 建议 | 关联的 git commit hash |
| 来源数据 | 建议 | 关联的 E2E 评测结果 / prompt 回归报告 / LLM raw output |
| 来源人工确认 | 建议 | 确认人姓名 + 确认日期 + 确认内容 |
| 根因 | ✅ | 解释"为什么"，而非重复"是什么" |
| 规则 | ✅ | 必须可执行，不能是模糊的"以后注意" |
| 废弃原因 | 条件必填 | 状态为 superseded 时必填 |
| 验证记录 | ✅ | 记录置信等级的初始设定和每次变更，形成可追溯的置信演化链 |

## Text2SQL-Agent 专属触发条件

以下场景**必须**触发经验复盘写入：

| 触发场景 | 复盘重点 | 关联文件 |
|---------|---------|---------|
| Prompt 模板修改（intent_classifier / sql_planner / sql_generator / explainer） | 为什么改、改了什么、预期效果、需要更新的回归用例 | prompts/*.md → evals/regression_cases.yml |
| LLM 输出模式异常（direct_sql、schema_validation_failed 等） | 是 Prompt 问题还是模型漂移、修复方案 | schema_validators.py |
| 安全策略调整（新增/修改禁止关键词、JOIN 白名单变更） | 为什么加这条规则、阻止了哪类攻击或误用 | sql_gen.py、check_sql_readonly.py |
| IR 数据结构变更（QuestionIntent / SQLPlan / SQLResult 字段变化） | 向下兼容性、对已有评测的影响 | ir.py、evals/*.yml |
| Agent 行为边界变更（反问策略、降级策略、置信度阈值） | 为什么调整阈值、预期对反问率/准确率的影响 | agent_config.yml、ambiguity.py |
| CI 门禁调整（新增/删除/修改检查项） | 为什么加/撤检查、对阻断标准的影响 | harness/、.github/workflows/ci.yml |
| E2E 评测中出现新的失败模式（不在已有 failure_triage 分类中） | 新失败模式的特征、推荐修复路径 | failure_triage.py |
| LLM Provider 切换或模型参数调整 | 切换原因、预期对输出质量的影响 | agent_config.yml |

## 状态生命周期

```text
proposed ──→ active ──→ disputed ──→ superseded ──→ archived
   │            │           │（经人工核实     │（被新规则     │（超过有效期
   │            │           │  确认确有错误）   │  替代）       │  且已无参考价值）
   │            │           │                │              │
   │            └───────────┴────────────────┴──────────────┘
   │                         所有路径最终可归档
   │
   └── 冷却期（默认 24 小时）
        新写入的记忆必须经过 Harness 门禁或人工确认后才能转为 active。
        冷却期内 Agent 可读取但不作为决策的唯一依据。
```

- **proposed**：新写入的记忆，待确认中——**Agent 可读取但不得作为决策的唯一依据**
- **active**：当前生效，每次相关任务前应查阅
- **disputed**：内容存疑，等待人工核实——**Agent 不得引用 disputed 条目作为决策依据**
- **superseded**：已被后续经验或规则替代，保留原文 + 标注废弃原因和替代规则编号
- **archived**：超过有效期或已无参考价值

## 内容质量门禁（自动校验）

写入或修改记忆条目后，运行内容质量校验：
```bash
python harness/checks/check_memory_update.py --content-only
```

校验项：
1. 必填字段完整性（日期、来源问题、根因、风险、规则、置信等级、版本、状态）
2. 结构化来源字段至少填写 1 项（来源文档/PR/Commit/数据/人工确认）
3. 条目最低长度（≥ 8 行）
4. 重复检测（与已有条目正文完全相同的复制粘贴）
5. 来源追溯（来源问题描述不得短于 20 字）
6. 交叉验证（条目中引用的表名、指标名是否在 meta.metric_definitions 或 information_schema 中真实存在）

## 写入前交叉验证（防止错误记忆）

在写入记忆前，必须：
1. **与 IR 定义交叉验证**：记忆中的事实是否与 `src/ir.py` 中的 dataclass 定义一致？若冲突，以 ir.py 为准。
2. **与已有规则交叉验证**：新增规则是否与已有规则矛盾？若矛盾，需在条目中明确说明冲突点和处理方式（替换/并存/待仲裁）。
3. **来源标记**：条目中必须至少包含 1 个结构化来源（来源文档/PR/Commit/数据/人工确认）。仅有自由文本"来源问题"且无结构化来源的条目在内容质量门禁中视为警告。
4. **置信等级自评**：初始写入时置信等级不得高于 L3——除非有 E2E 多轮评测数据或 prompt 回归报告等硬证据支撑。禁止将基于推测的结论标为 L4/L5。
5. **冷却期**：新写入的记忆默认状态为 `proposed`（非 `active`）。需经至少 1 次 Harness 门禁通过或人工确认后才能转为 `active`。

## 发现错误记忆后的修复流程

核心步骤：
1. 将错误条目标记为 `disputed`，说明存疑原因，降级置信等级至 L2 或更低
2. 运行 `python harness/checks/check_memory_update.py --content-only` 获取自动检测的下游影响清单
3. 核实后改为 `superseded`，填写废弃原因和替代规则编号，创建新版条目（如有修正后规则）
4. 检查并修正所有下游污染（AGENTS.md 引用、检查脚本、测试），在变更 commit message 中注明关联的 R 编号
5. **不得直接删除错误条目**——保留原文是为防止同一错误被再次写入，也是置信演化链的一部分

## 判断标准

如果问题只影响一次性 Prompt 文案，可以只写复盘。

如果问题会影响 Agent 行为边界、安全检查、评测标准、Prompt 模板或 IR 结构，必须进一步沉淀为：

1. AGENTS.md 规则
2. harness/checks 检查脚本
3. tests 回归测试或 evals YAML
