# evals 覆盖缺口审计报告

> 审计日期：2026-06-17
> 审计范围：`evals/*.yml`（5 个文件，共 39 个用例）对 6 个新模块的能力和安全边界覆盖
> 审计层级：evals（Agent 行为回归）vs pytest（单元测试）vs harness（静态代码检查）
> 状态：只审计，不出实现

---

## 审计方法论

本审计区分三个层次：

| 层次 | 用途 | 覆盖对象 | 运行环境 |
|------|------|---------|---------|
| **evals** | Agent 端到端行为回归 | 意图→计划→SQL→执行→融合→展示的完整链路 | LLM mock + DuckDB |
| **pytest** | 代码单元行为验证 | 单个函数/类的输入输出 | 纯 Python，无 LLM |
| **harness** | 静态代码安全扫描 | 源码中的模式/引用/结构 | 静态分析，无执行 |

**核心原则**：pytest 覆盖代码逻辑 ≠ evals 覆盖 Agent 行为。一个函数的输入输出正确，不代表它在 Agent 管道的真实上下文中正确工作。

## 逐模块覆盖分析

### 1. PlanExecutor

| 能力/边界 | evals 覆盖 | pytest 覆盖 | harness 覆盖 |
|-----------|-----------|------------|-------------|
| 每个 SQLPlan 独立生成 SQL | ✅ e2e answer×10, downgrade×5 | ✅ test_execute_one_generates_sql_and_returns_result | ✅ check_plan_executor_safety |
| 每个 SQLPlan 独立 validate_sql_safety | ✅ e2e safety boundary×1 (SQL注入检测) | ✅ test_execute_one_trace_on_safety_failure | ✅ 源码中 validate_sql_safety 引用检查 |
| offline 模式不执行 DuckDB | ❌ 无 eval | ✅ test_execute_one_offline_mode_blocked | ✅ 源码中 offline 引用检查 |
| read_only 不被绕过 | ❌ 无 eval | ❌ 无显式测试 | ✅ 源码中 read_only 引用检查 |
| safety_check_passed 在 ExecutionTrace 中传递 | ❌ 无 eval | ✅ test_execute_one_sets_execution_trace | ✅ check_plan_executor_safety |

**结论**：PlanExecutor 的基本通路有 evals 覆盖（所有 answer 用例都经过它），但**安全边界退化场景**（offline bypass, read_only bypass）只有 pytest + harness 覆盖，缺少 evals 端到端验证。

**关键缺口**：
- GAP-1: 无人验证如果 validate_sql_safety 在真实 Agent 管道中被跳过会怎样
- GAP-2: 无人验证 offline 模式在 mock 场景下的行为与真实场景一致

### 2. Execution Strategy

| 能力/边界 | evals 覆盖 | pytest 覆盖 | harness 覆盖 |
|-----------|-----------|------------|-------------|
| 并发默认关闭 | ✅ 所有 evals 默认串行 | ✅ test_default_strategy_is_serial | ✅ check_execution_strategy_safety |
| ThreadPool 不共享 DuckDB connection | ❌ 无 eval | ✅ test_no_shared_connection | ✅ 源码中 executor_factory 检查 |

| 串行/并发结果一致 | ❌ 无 eval | ❌ 无显式对比测试 | ❌ |

**结论**：Execution Strategy 的**核心安全边界只在 pytest 层验证**。evals 完全没有覆盖并发执行场景，因为 evals 使用 mock 模式时执行策略被简化。

**关键缺口**：
- GAP-3: 无 evals 验证"即使配置错误打开了并行，连接也不会共享"
- GAP-4: 无 evals 验证"并行模式下 3 个 plan 的执行结果与串行完全一致"

## 不建议现在做的内容

1. **不新增 mock LLM 注入测试**：需要修改 eval runner 支持 `mock_llm_output_override`，这属于 runner 改动，超出本轮范围
2. **不新增并发 evals**：并发行为依赖真实多线程环境，mock 模式下难以可靠模拟
3. **不新增 chart spec visual regression**：图表类型选择的可视化验证需要 screenshot diff，基础设施不成熟
4. **不修改 eval runner**：本轮只审计，不实现
5. **不将 harness check 能覆盖的静态约束降级为 evals**：如 HTML/JS 禁止、LLM 导入检测等，harness 更高效
6. **不新增"真实 LLM"的 evals**：仅建议 mock 模式下的新用例
