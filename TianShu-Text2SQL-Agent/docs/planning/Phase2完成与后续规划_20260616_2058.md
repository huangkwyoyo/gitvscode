## 三、Phase 3 规划：并行 + 融合 + 对齐

### 3A：并行执行基础设施

**目标**：在不破坏安全约束的前提下，让多计划并行执行。

**核心挑战**：

| 挑战 | 说明 | 解决思路 |
|------|------|----------|
| DuckDB 线程安全 | 当前使用单连接 `read_only=True`。多线程共享单连接可能不安全 | 验证 DuckDB 的 `duckdb.connect(read_only=True)` 是否支持多线程读；如不支持，每个线程创建独立连接 |
| 连接池管理 | 每个线程需要一个独立连接，但不能无限创建 | 限制最大连接数（如 `min(len(plans), 4)`），用完即关 |
| 错误隔离 | 一个子计划失败不应影响其他子计划 | `ThreadPoolExecutor.submit()` + `Future.result()` 捕获异常 |

**最小实现路径**：

```python
# agent.py 中的串行循环替换为：
from concurrent.futures import ThreadPoolExecutor, as_completed

def _execute_plans_parallel(self, unified_responses, forbidden_kw, ...):
    max_workers = min(len(unified_responses), 4)
    with ThreadPoolExecutor(max_workers=max_workers) as executor:
        futures = {
            executor.submit(self._execute_single_plan, ur, ...): i
            for i, ur in enumerate(unified_responses)
        }
        for future in as_completed(futures):
            i = futures[future]
            try:
                unified_responses[i] = future.result(timeout=30)
            except Exception as exc:
                unified_responses[i].result = SQLResult(
                    sql="", error=f"并行执行异常: {exc}"
                )
```

**新增风险**：

| 风险编号 | 风险 | 说明 |
|----------|------|------|
| RISK-018 | 并行执行导致 DuckDB 连接竞争 | 如果多线程共享单连接，可能产生 `database is locked` 或数据错乱 |
| RISK-019 | 并行执行超时后线程未正确回收 | 一个 SQL 超时不应阻塞其他 SQL 的结果返回 |

### 3B：LLM 结果融合

**目标**：用 LLM 替代当前模板式 `fuse_results()`，生成更自然的中文融合回答。

**核心挑战**：

| 挑战 | 说明 | 解决思路 |
|------|------|----------|
| LLM 幻觉 | LLM 可能"编造"子结果中不存在的数据关系 | Prompt 中明确要求"只描述已提供的数据，不要推断因果关系" |
| 数值篡改 | LLM 可能在融合时"微调"数值使其看起来更合理 | 要求 LLM 在回答中引用原始数值，且与 SQLResult 中的值一致 |
| 跨域表述 | 不同业务域的指标放在一起解释时容易产生误导性关联 | 分域陈述，先各自摘要再统一收尾 |

**最小实现路径**：

```python
def fuse_results_llm(question, unified_responses, llm_client):
    # 1. 构造融合 Prompt，包含所有子结果的摘要
    # 2. LLM 生成自然语言融合解释
    # 3. 后校验：检查 LLM 输出中的数值是否与原始结果一致
    # 4. 如校验失败，回退到模板融合
```

**新增风险**：

| 风险编号 | 风险 | 说明 |
|----------|------|------|
| RISK-020 | LLM 融合时篡改或编造数值 | 即使 Prompt 要求忠实于数据，LLM 仍可能"创造性"地修改 |
| RISK-021 | LLM 融合的 JSON 解析失败导致整个多计划回答不可用 | 需要模板融合作为 fallback |

### 3C：多结果按 date 自动 merge

**目标**：将多个子结果的日期列对齐，形成统一的日期序列，便于对比分析。

**核心挑战**：

| 挑战 | 说明 | 解决思路 |
|------|------|----------|
| grain 不一致 | trip 表按 `trip_date`，crash 表按 `crash_date`。同一日期可能在一张表有数据、另一张没有 | 使用 FULL OUTER JOIN 语义，缺失日期填充 NULL |
| 日期范围不一致 | 不同子查询的日期范围可能不同（取决于各表的数据覆盖） | 明确 merge 策略：取交集 / 取并集 / 以第一个结果为准 |
| 指标名冲突 | 多个子结果可能有相同的列名（如都叫 "count"） | 子结果列名自动添加表名前缀 |

**最小实现路径**：

```python
def merge_by_date(unified_responses):
    # 1. 收集所有子结果的 (date, value) 对
    # 2. 按 date 做 outer join
    # 3. 返回合并后的 DataFrame
    # 4. 标注缺失日期的指标
```

### 3D：复杂跨域校验

**目标**：跨业务域多指标查询时，检测结果中的语义异常。

**核心挑战**：

| 挑战 | 说明 | 解决思路 |
|------|------|----------|
| 异常定义 | 什么算"异常"缺乏业务标准，不同场景阈值不同 | 先做简单的同比/环比检测，超过 N 个标准差标记 |
| 误报 | 跨域数据因外部因素（天气、节假日、政策）天然会有联动变化 | 标记为"值得关注"而非"异常"，由用户判断 |

## 六、新增经验复盘条目

以下条目应按 [[变更复盘模板]] 写入 [[经验复盘]]：

### R008：跨表多指标在 Phase 1 被错误反问——IR 结构不够用

- **来源问题**：用户问"每天行程数和受伤人数"，Phase 1 返回 `UNSUPPORTED_MULTI_METRIC` 反问用户分别查询——但用户的跨表问题是自然的、合理的
- **根因**：Phase 1 的 IR 隐藏假设"一次问答 = 一个 SQLPlan"，`AgentResponse.plan` 是 `Optional[SQLPlan]` 而非列表
- **规则**：IR 结构应支持"一对多"映射——`AgentResponse.plans: list[UnifiedResponse]` 使单次问答可包含多个独立查询计划
- **已落地**：Phase 2B SubIntent / UnifiedResponse / AgentResponse.plans / Strategy.UNSUPPORTED_MULTI_PLAN

### R009：模板式融合对多计划结果的信息展示不充分

- **来源问题**：当前 `fuse_results()` 只列出每个子计划的统计摘要，用户需要自行对比两个结果
- **根因**：模板式融合无法做跨结果的对比分析（如"trip_count 日均 85万，persons_injured 日均 70人"的关联解读）
- **规则**：模板融合是安全基线——LLM 融合必须保留模板作为 fallback，且 LLM 输出必须经过数值一致性后校验
- **已落地**：Phase 2D fuse_results() 模板融合（已完成）；LLM 融合待 Phase 3B

## 八、关键设计决策待确认

以下决策需要在进入 Phase 3A 之前明确：

| 决策点 | 选项 A | 选项 B | 推荐 |
|--------|--------|--------|:--:|
| 并行粒度 | 按 SQLPlan 并行（每个计划一个线程） | 按 SQL 并行（所有 SQL 提交到一个线程池） | A |
| DuckDB 连接 | 每线程一个独立连接 | 共享单连接 + 读写锁 | A（更安全） |
| LLM 融合 fallback | 自动回退到模板（透明） | 返回错误让用户重试 | A（用户体验更好） |
| date merge 策略 | FULL OUTER JOIN（取并集） | INNER JOIN（取交集） | A（避免丢失信息） |
| 跨域校验告警 | 仅在回答末尾标注"值得关注" | 单独弹出一个警告框 | A（减少干扰） |
