## 三、Phase 3 规划（修订版）

### Phase 0：文档债清理（优先级最高，预计 1 轮迭代）

**目标**：消除 §〇 中识别的 5 处文档缺口，确保后续设计建立在准确文档基础上。

| 缺口 | 修复动作 | 影响范围 |
|:--:|---------|---------|
| 1 | `text2sql_current_pipeline.md` 新增 §3.5 多计划执行路径 + 更新 §1 工作流图 | pipeline 文档 |
| 2 | `text2sql_engineering_glossary.md` 新增 6 个 Phase 2 术语（§41-§46）| 术语表 |
| 3 | `prompt_regression.md` 新增 2 个 failure_category | 回归文档 |
| 4 | `llm_integration_design.md` + `llm_integration_phase2a.md` 添加 disambiguation 块 | 设计文档 |
| 5 | `docs/README.md` 开发日志更新 | 索引文档 |

### Phase 3A：并行执行基础设施

**目标**：在不破坏安全约束的前提下，让多计划并行执行。

**核心挑战**（同 v1，新增文档同步要求）：

| 挑战 | 说明 | 解决思路 |
|------|------|----------|
| DuckDB 线程安全 | 单连接 `read_only=True` 是否支持多线程读 | 如不支持，每线程独立连接 |
| 连接池管理 | 连接数可控 | `min(len(plans), 4)` 上限 |
| 错误隔离 | 子计划间不互相影响 | ThreadPoolExecutor + Future.result() 捕获异常 |
| **文档同步**（新增） | glossary §18 "只读 DuckDB" 描述单连接假设——并行化后条目过时 | 同步更新 glossary §18 |

**新增风险**：RISK-018（连接竞争）、RISK-019（线程泄露）——已在 v1 中记录。

### Phase 3B：LLM 结果融合

**调整为两步**：

**3B-1：LLM 融合 Prompt 设计**（可与 3A 并行）
- 设计融合 Prompt 模板（`prompts/fuse_results.md`）
- 定义 LLM 输入/输出 JSON schema
- 反幻觉约束：只描述已提供数据 + 不推断因果 + 不修改数值
- 设计 failure_category：`fusion_hallucination`、`fusion_number_mismatch`

**3B-2：数值一致性后校验 + 模板 fallback**（依赖 3B-1）
- 实现后校验函数：提取 LLM 输出中的所有数值，与 SQLResult 比对
- 实现自动回退：后校验失败 → 输出模板融合结果 + 标注"LLM 融合未通过校验，以下为模板总结"

**新增风险**：RISK-020（数值篡改）、RISK-021（JSON 解析失败）——已在 v1 中记录。

### Phase 3C：多结果按 date 自动 merge

**目标**：将多个子结果的日期列对齐。

同 v1 设计，新增：
- 在 glossary 中补充 "grain 不一致" 术语定义（3C 设计前完成）
- merge 策略文档化到 `text2sql_current_pipeline.md` 的 §3.5

### Phase 3D：复杂跨域校验

**调整为两步**：

**3D-1：单域同比/环比检测**（可与 3B 并行）
- 对单个子结果做同比/环比计算
- 超过 N 个标准差标记为"值得关注"
- 不依赖 3C（不需要跨表 merge）

**3D-2：跨域联合异常检测**（依赖 3C + 3D-1）
- 多个子结果按 date 对齐后，检测跨域联动异常
- 需要业务规则定义（什么算异常）
