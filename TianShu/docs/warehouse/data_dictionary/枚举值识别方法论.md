# 枚举值识别方法论

> **定位**：本文档定义如何系统性地识别数据仓库中需要解释中文含义的枚举值字段（含状态码、标志位、分类代码）。是 `bronze_enum_values.md` 的生成规则来源。
> **适用范围**：Bronze / Silver / Gold 所有层级的表和视图。

---

## 一、什么是"枚举值"

在数据仓库语境中，**枚举值（Enumeration Value）** 泛指以下三类字段：

| 类型 | 典型形态 | 例子 | 中文习惯叫法 |
|---|---|---|---|
| **标志位** | `0`/`1`、`Y`/`N`、`YES`/`NO` | `is_waisu = 1` 表示外宿 | Flag、标志位 |
| **状态码** | 固定几个英文短语或缩写 | `Status = 'Approved'` | 状态码、Status |
| **分类代码** | 大写字母缩写或纯数字 | `Type = 'PDR'`、`payment_type = 2` | 枚举值、代码、Code |

**共同特征**：字段取值**来自一个有限的集合**，每个值代表一种**业务状态或分类**，字面不一定自解释，需要通过数据字典补充中文含义。

**典型例子**：

| 字段 | 取值样例 | 类型 | 为什么需要解释 |
|---|---|---|---|
| `is_waisu` | `0`, `1` | 标志位 | `1` 代表外宿还是内宿？不查字典不知道 |
| `Type` | `PDR`, `VDR`, `HDR` | 分类代码 | 三个大写缩写，字面无法猜出含义 |
| `VendorID` | `1`, `2`, `6`, `7` | 分类代码 | 数字代码，对应具体供应商 |
| `payment_type` | `1`, `2`, `3`, `4` | 分类代码 | 数字代表支付方式类别 |
| `Status` | `Denied`, `Under Review` | 状态码 | 可读但中文用户需要翻译 |
| `Active` | `YES`, `NO` | 标志位 | 含义明确，但注明后更规范 |

**不是枚举值的例子**：

| 字段 | 取值样例 | 为什么不需要解释 |
|---|---|---|
| `trip_distance` | `1.5`, `23.7` | 连续数值，不是分类 |
| `street_name` | `BROADWAY`, `5TH AVE` | 专有名称，字面自解释 |
| `collision_id` | `4418392` | 唯一标识符，不是分类 |
| `fare_amount` | `15.50`, `42.00` | 金额度量，不是分类 |

## 二、识别规则

### 规则 1：事实表 — 去重率判断

**适用场景**：行数 > 10万的事实表（如 tripdata、crash、parking 等）

**算法**：

```
code_signal = distinct_count(column) / total_rows

if code_signal < 0.01 (即 1%) and distinct_count between 2 and 200:
    → 该字段很可能是枚举值字段
    → 进入人工审核流程
```

**阈值说明**：
- **上限 200**：超过 200 种取值通常不是分类代码（除非是车牌号等外部编码，车牌号另行判断）
- **下限 2**：只有 1 种取值无分类意义
- **1% 阈值**：如果 100 万行中一个字段只有不到 1 万种取值，说明大量重复，分类可能性高

**验证示例**（来自 Bronx 层实际扫描）：

| 表 | 行数 | 字段 | Distinct | 信号 | 判定 |
|---|---|---|---|---|---|
| yellow_tripdata | 11,077,206 | VendorID | 4 | 0.00004% | ✅ 分类代码 |
| yellow_tripdata | 11,077,206 | RoutecodeID | 7 | 0.00006% | ✅ 分类代码 |
| parking_violations | 9,582,412 | violation_code | 100 | 0.001% | ✅ 分类代码 |
| parking_violations | 9,582,412 | plate_type | 84 | 0.0009% | ✅ 分类代码 |
| crash_merged | 1,655,065 | collision_id | ≈1,655,065 | ≈100% | ❌ 唯一标识，非枚举值 |
| crash_merged | 1,655,065 | borough | 5 | 0.0003% | ✅ 分类代码 |

### 规则 2：维度表 — 全表扫描

**适用场景**：行数 ≤ 10万的快照/维度表（如 new_driver_applications、taxi_zone、vehicle/driver 快照表）

**方法**：直接执行 `SELECT DISTINCT` 获取所有取值，人工逐列审核。

**判断标准**（满足任一条即为枚举值）：
1. 值匹配正则 `^[A-Z][A-Z0-9_-]{1,20}$`（大写字母开头的缩写/代码格式）→ **分类代码**
2. 值为纯数字且 ≤ 200 种（数字编码）→ **分类代码**
3. 值虽可读但属于行业术语（如 `Medallion`、`Street Hail`）→ **状态码/分类代码**
4. 值为 `Y`/`N`、`YES`/`NO`、`0`/`1` 且 distinct ≤ 4 → **标志位**

**验证示例**（来自 Bronze 层实际扫描）：

| 表 | 行数 | 字段 | Distinct | 枚举值类型 |
|---|---|---|---|---|
| new_driver_applications | 4,076 | Type | 3 (HDR/PDR/VDR) | 分类代码 |
| new_driver_applications | 4,076 | Status | 5 (英文短语) | 状态码 |
| new_driver_applications | 4,076 | Drug Test | 3 (Complete/Needed/Not Applicable) | 状态码 |
| taxi_zone_lookup | 265 | Borough | 8 (含EWR) | 分类代码 |
| fhv_active_vehicles | 104,420 | VEH | 9 (BEV/HYB/CNG...) | 分类代码 |
| active_vehicles | 119,207 | Fuel Type | 7 (英文短语) | 分类代码 |
| active_vehicles | 119,207 | WAV | 3 (YES/NO/Pilot) | 标志位 |
| medallion_authorized | 10,547 | Medallion Type | 2 | 分类代码 |

### 规则 3：排除规则（避免误判）

以下类型即使满足规则1或规则2的阈值条件，也**不应**作为枚举值处理：

| 字段类型 | 例子 | 排除理由 |
|---|---|---|
| 唯一标识符 | `License Number`, `collision_id` | 每行不同，不是分类 |
| 日期/时间 | `issue_date`, `crash_time` | 时间序列，不是分类 |
| 金额/度量 | `fare_amount`, `total_amount` | 连续数值，不是分类 |
| 姓名/名称 | `Owner Name`, `Base Name` | 专有名词，字面自解释（但缩写名仍需解释） |
| 车牌/VIN | `DMV License Plate Number`, `VIN` | 外部编码，唯一标识 |

## 四、审核状态标记规范

| 标记 | 含义 | 适用条件 |
|---|---|---|
| `✅ 已确认` | 含义来自官方数据字典或权威公开文档 | 查到了 NYC TLC/DOT/DMV 官方文档或数据集说明页 |
| `⚠️ 推断，待官方确认` | 基于上下文、行业知识和字段命名合理推断 | 官方未公开该字段的枚举值映射，但通过上下文可合理推断 |
| `⚠️ Human Review` | 完全无法推断 | 代码含义无公开资料，需向数据提供方确认 |

## 五、与项目其他规范的衔接

- **`bronze_enum_values.md`**：本方法论的实际产出，按表列出所有枚举值字段的中文含义
- **`README.md`（本目录）**：数据字典总览，引用本方法论
- **`database_design/`**：枚举值识别结果如涉及表结构变更，需同步更新 database_design
- **质量检查**：可在 `scripts/quality/` 中增加自动化脚本，基于本方法论的规则 1 定期扫描新增字段
