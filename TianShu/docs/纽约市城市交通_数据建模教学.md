# 数据建模教学：从 ER 模型到 Data Lakehouse 分层

本文结合 `D:\ProgramData\Datawarehouse\纽约市城市交通` 下的数据（已导入 DuckDB 数据库 `nyc_transport`），讲清楚数据建模从浅到深的几种常见方法。

> **环境说明**：数据库文件位于 `D:/ProgramData/Datawarehouse/纽约市城市交通/nyc_transport.duckdb`。启动 DuckDB CLI 时需指定文件路径：
> ```bash
> duckdb "D:/ProgramData/Datawarehouse/纽约市城市交通/nyc_transport.duckdb"
> ```
> 为方便使用，可设置别名（Linux/Mac 的 `~/.bashrc` 或 Windows 的 PowerShell Profile）：
> ```bash
> alias nyc-db='duckdb "D:/ProgramData/Datawarehouse/纽约市城市交通/nyc_transport.duckdb"'
> ```

```text
ER模型（实体关系模型）
↓
维度建模（事实表 + 维表）
↓
星型模型（Star Schema）
↓
雪花模型（Snowflake Schema）
↓
Bronze → Silver → Gold → Meta（Data Lakehouse 分层）
```

这几个概念不是互相替代的关系，而是面向不同阶段、不同目标的建模思想。

- ER 模型更偏"业务实体和关系"。
- 维度建模更偏"面向分析和指标"。
- 星型模型和雪花模型是维度建模的两种典型结构。
- Bronze/Silver/Gold/Meta 是 Data Lakehouse（数据湖仓）工程落地时的分层体系。

> **补充说明**：传统数据仓库常使用 ODS → DWD → DWS → ADS 分层。本文数据库 `nyc_transport` 实际采用了业界更现代的 **Data Lakehouse 分层模型**（Bronze / Silver / Gold / Meta），它融合了数据湖的灵活性和数据仓库的治理能力。两种分层的对应关系会在第 6 节详细说明。

对本次纽约市城市交通数据来说，最合适的思路是：

```text
源表宽表集合（原始 CSV / Parquet / SHP 文件）
↓
🥉 Bronze（青铜层 / 原始数据层）— 原始数据接入，保持原貌
↓
🥈 Silver（白银层 / 清洗数据层）— 清洗、标准化、去重、统一字段
↓
🥇 Gold（黄金层 / 业务数据层）— 面向业务的高度聚合宽表与汇总表
↓
📊 Meta（元数据层）— 数据质量监控、血缘追踪、中文语义字典
```

其中事故数据适合保留局部 ER 子模型，出行、罚单、车辆、司机等主题适合进入维度建模和星型模型。

## 2. ER 模型：先把业务实体和关系讲清楚

ER 是 Entity Relationship 的缩写，中文常叫"实体关系模型"。

它关心三个问题：

1. 有哪些实体？
2. 每个实体有什么属性？
3. 实体之间是什么关系？

例如在本次数据里，可以识别出这些实体：

| 实体（中文） | 来源表（英文） | 来源表（中文） | 关键字段 |
|---|---|---|---|
| 行程 | TLC 行程 parquet | TLC 行程数据 | 上车时间、下车时间、上车区域、下车区域、费用、距离 |
| 出租车区域 | `taxi_zone_lookup` | 出租车区域字典表 | `LocationID`（区域ID）、`Borough`（行政区）、`Zone`（区域名） |
| 车辆 | `active_vehicles` | 活跃车辆注册表 | `License Number`（驾照号）、VIN（车架号）、车牌、燃料、年份 |
| 司机 | FHV 活跃司机、SHL 活跃司机 | 活跃司机表 | `License Number`（驾照号）、姓名、类型、到期日期 |
| 事故 | `crash_merged` | 交通事故主表 | `collision_id`（事故编号）、事故时间、经纬度、伤亡人数 |
| 事故人员 | `crash_person_all` | 事故人员明细表 | `unique_id`（人员记录编号）、`collision_id`（事故编号）、人员类型、伤害程度 |
| 停车罚单 | `parking_violations_all` | 停车罚单全量表 | `summons_number`（罚单编号）、车牌、违章代码、开票日期 |
| 司机申请 | 新司机申请状态表 | 新司机申请状态表 | `App No`（申请编号）、申请日期、状态 |
| TIF 支付 | TIF 支付表 | TIF 牌照支付记录表 | 牌照号、支付日期、支付金额 |

ER 模型最适合表达"事故"和"事故人员"这种关系：

```text
crash_merged（交通事故主表）
├─ collision_id（事故编号）      -- 主键（待治理）
├─ crash_date（事故日期）
├─ crash_time（事故时间）
├─ latitude（纬度）
├─ longitude（经度）
└─ ...

crash_person_all（事故人员明细表）
├─ unique_id（人员记录编号）      -- 主键（待治理）
├─ collision_id（事故编号）       -- 外键，关联事故表
├─ person_type（人员类型）
├─ person_injury（伤害程度）
└─ ...
```

理想情况下，它应该是：

```text
一场事故 crash_merged.collision_id（事故编号）
↓ 一对多
多名事故人员 crash_person_all.collision_id（事故编号）
```

也就是：

```text
事故表（crash_merged） 1 ─── N 事故人员表（crash_person_all）
```

但本次数据审查发现：

- `crash_merged.collision_id`（事故编号）当前不唯一。
- `crash_person_all.unique_id`（人员记录编号）当前不唯一。
- 两表的 `collision_id`（事故编号）覆盖不完全。

这说明当前源表还不能直接当成干净 ER 模型，需要先做 Silver 层治理：

```text
源表 crash_merged（交通事故主表）
↓ 去重、确认粒度、生成代理键
silver.crash_detail（事故明细标准表）

源表 crash_person_all（事故人员明细表）
↓ 去重、确认人员记录粒度、生成代理键
silver.crash_person_detail（事故人员明细标准表）
```

然后再建立清晰关系：

```text
silver.crash_detail.collision_id（事故编号）
↓
silver.crash_person_detail.collision_id（事故编号）
```

### ER 模型适合什么

ER 模型适合：

- 讲业务实体。
- 说明主键、外键。
- 表达一对一、一对多、多对多关系。
- 支撑业务系统或明细数据治理。

### ER 模型不擅长什么

ER 模型不适合直接服务复杂分析。例如用户问：

> 2026 年 Q1 每个区域的行程量、事故量、罚单量分别是多少？

如果只用 ER 模型，查询会很复杂。这个时候就需要维度建模。

## 3. 维度建模：从"实体关系"转向"分析问题"

维度建模不是从实体出发，而是从分析问题出发。

它先问：

1. 分析的对象是什么？
2. 指标是什么？
3. 从哪些角度看指标？

维度建模里有两个核心概念：

| 概念 | 含义 | 本次数据示例 |
|---|---|---|
| 事实表（Fact Table） | 记录业务事件或可度量行为 | 行程、事故、罚单、支付、申请 |
| 维表（Dimension Table） | 描述分析角度 | 日期、区域、车辆、司机、基地、违章代码 |

举例：分析"行程量"。

事实表是行程：

```text
fact_trips（行程事实表）
├─ trip_id（行程ID）              -- 主键，全局唯一
├─ pickup_at（上车时间）           -- 事实度量
├─ dropoff_at（下车时间）          -- 事实度量
├─ pickup_location_id（上车区域ID） -- 关联维表
├─ dropoff_location_id（下车区域ID）-- 关联维表
├─ trip_distance（行程距离）       -- 指标
├─ fare_amount（基础费用）         -- 指标
├─ total_amount（总费用）          -- 指标
└─ trip_source（行程来源）         -- 退化维度：yellow / green / fhv / fhvhv
```

维度表是分析角度：

```text
dim_date（日期维表）        -- 时间角度：日、周、月、季度、财年
dim_taxi_zone（区域维表）   -- 空间角度：区域、行政区、服务区
dim_vehicle（车辆维表）     -- 车辆角度：车型、燃料、年份、WAV
dim_base（基地维表）        -- 基地角度：基地名称、类型、地址
```

用户问：

> 每天每个行政区有多少行程？

维度建模会把问题拆成：

```text
指标：行程数 COUNT(*)
事实：fact_trips（行程事实表）
时间维度：dim_date（日期维表）
空间维度：dim_taxi_zone（区域维表）
```

这比从一堆源表临时拼字段更清晰，也更适合 Agent 自动生成查询。

## 4. 星型模型：最适合分析和 Agent 问数

星型模型是维度建模最常见的形式。

它的结构像星星：

```text
          dim_date（日期维表）
             |
dim_taxi_zone（区域维表） -- fact_trips（行程事实表） -- dim_base（基地维表）
             |
        dim_vehicle（车辆维表）
```

中间是事实表，周围是维表。

以本次数据为例，出行主题可以这样设计：

```text
fact_trips（行程事实表）
├─ trip_id（行程ID）                    -- 主键
├─ trip_source（行程来源）              -- 退化维度：yellow / green / fhv / fhvhv
├─ pickup_at（上车时间）
├─ dropoff_at（下车时间）
├─ pickup_date_key（上车日期键）        -- 外键，关联 dim_date
├─ pickup_location_id（上车区域ID）      -- 外键，关联 dim_taxi_zone
├─ dropoff_location_id（下车区域ID）     -- 外键，关联 dim_taxi_zone
├─ dispatching_base_num（调度基地编号）  -- 外键，关联 dim_base
├─ trip_distance（行程距离/英里）
├─ fare_amount（基础费用/美元）
├─ total_amount（总费用/美元）
├─ passenger_count（乘客数）
└─ quality_flags（质量标记位）

dim_date（日期维表）
├─ date_key（日期键）                    -- 主键，YYYYMMDD 格式
├─ date（日期）
├─ year（年）
├─ quarter（季度）
├─ month（月）
├─ week（周）
└─ day_of_week（星期几）

dim_taxi_zone（出租车区域维表）
├─ location_id（区域ID）                -- 主键
├─ borough（行政区）                     -- Manhattan / Brooklyn / Queens / Bronx / Staten Island
├─ zone（区域名称）                      -- 如"Upper East Side"
└─ service_zone（服务区）               -- Yellow Zone / Boro Zone

dim_base（基地维表）
├─ base_no（基地编号）                   -- 主键
├─ base_name（基地名称）
├─ base_type（基地类型）
└─ base_address（基地地址）
```

用户问：

> 2026 年 Q1 曼哈顿的 HVFHV 出行量是多少？

星型模型的思路是：

```text
fact_trips（行程事实表）
↓ pickup_location_id（上车区域ID）
dim_taxi_zone（出租车区域维表）
↓ borough = 'Manhattan'（行政区 = 曼哈顿）
```

再按 `trip_source = 'fhvhv'` 过滤。

### 为什么星型模型适合 Agent

因为 Agent 最怕的是：

- 不知道表和表怎么连。
- 不知道哪个字段是指标。
- 不知道哪些字段能过滤。
- 不知道哪个字段中文含义是什么。

星型模型把这些问题变简单：

```text
事实表回答"发生了什么"
维表回答"从什么角度看"
指标来自事实表
过滤和分组多来自维表
```

所以，对中文问数 Agent 来说，星型模型比原始宽表更安全。

## 5. 雪花模型：把维表继续拆细

雪花模型是星型模型的进一步规范化。

星型模型中，维表通常比较宽：

```text
dim_vehicle（车辆维表 / 宽表形态）
├─ vehicle_key（车辆键）                -- 主键
├─ license_no（驾照号）
├─ plate_no（车牌号）
├─ vin（车架号）
├─ vehicle_make（车辆品牌）
├─ vehicle_model（车辆型号）
├─ vehicle_year（车辆年份）
├─ fuel_type（燃料类型）
├─ wav_flag（无障碍车辆标识）
├─ base_no（基地编号）
├─ base_name（基地名称）
└─ base_type（基地类型）
```

如果继续拆，就变成雪花模型：

```text
dim_vehicle（车辆维表 / 雪花形态）
├─ vehicle_key（车辆键）                -- 主键
├─ license_no（驾照号）
├─ plate_no（车牌号）
├─ vin（车架号）
├─ vehicle_model_key（车型键）          -- 外键，关联 dim_vehicle_model
├─ fuel_type_key（燃料类型键）          -- 外键，关联 dim_fuel_type
└─ base_key（基地键）                   -- 外键，关联 dim_base

dim_vehicle_model（车型维表）
├─ vehicle_model_key（车型键）          -- 主键
├─ vehicle_make（车辆品牌）
├─ vehicle_model（车辆型号）
└─ vehicle_year（年份）

dim_fuel_type（燃料类型维表）
├─ fuel_type_key（燃料类型键）          -- 主键
└─ fuel_type_name（燃料类型名称）

dim_base（基地维表）
├─ base_key（基地键）                   -- 主键
├─ base_no（基地编号）
├─ base_name（基地名称）
└─ base_type（基地类型）
```

这就是雪花模型——维表被进一步规范化，形成多层关联。

### 星型和雪花怎么选

| 模型 | 优点 | 缺点 | 本次建议 |
|---|---|---|---|
| 星型模型（Star Schema） | 简单、查询快、Agent 容易理解 | 维表可能有重复描述 | 优先使用 |
| 雪花模型（Snowflake Schema） | 更规范、冗余更少 | Join 更多，Agent 更容易连错 | 只在维表很复杂时使用 |

对本次数据，我建议：

- 出行、罚单、事故分析层优先使用星型模型。
- 车辆、基地、违章代码这类维表可以局部雪花化。
- 面向 Agent 的查询层尽量保持星型，减少多跳 Join。

## 6. Data Lakehouse 分层：Bronze → Silver → Gold → Meta

前面的 ER、维度建模、星型、雪花都是"模型思想"。

Bronze / Silver / Gold / Meta 是"工程落地分层"——它是业界标准的 **Data Lakehouse（数据湖仓）** 架构，由 Databricks 等厂商推广，融合了数据湖的灵活存储和数据仓库的严格治理。`nyc_transport` 数据库完美遵循了这套分层模型。

### 6.0 与传统数仓分层的对比

在深入之前，先理清两种分层体系的对应关系：

| Data Lakehouse 分层 | 传统数仓分层 | 核心职责 | 数据特征 |
|---|---|---|---|
| 🥉 **Bronze**（青铜层 / 原始数据层） | ODS（原始数据层） | 原始接入，保持源数据原貌 | 未经处理、可能含脏数据、格式不统一 |
| 🥈 **Silver**（白银层 / 清洗数据层） | DWD（明细标准层） | 清洗、标准化、去重、统一字段 | 高质量、字段统一、可直接用于分析 |
| 🥇 **Gold**（黄金层 / 业务数据层） | DWS（汇总服务层）+ ADS（应用层） | 面向业务的高度聚合 | 宽表或汇总表，直接供 BI/看板使用 |
| 📊 **Meta**（元数据层） | —（数仓中通常散落各处） | 数据质量监控、血缘追踪、中文语义 | "关于数据的数据" |
| 🏠 **Main**（默认主层） | —（数据库默认 Schema） | DuckDB 默认 Schema，当前工作空间 | 开发调试时的临时落脚点 |

**关键区别**：
- 传统数仓分层强调"逐层加工"，层与层之间是强依赖的流水线；
- Data Lakehouse 分层强调"逐层提纯"，Bronze 可以同时被多个 Silver 任务消费，Silver 也可以灵活组合成不同 Gold 视图；
- Lakehouse 多了一个独立的 **Meta 层**，把数据质量、血缘、语义字典统一管理起来，而不是散落在各层。

### 6.1 🥉 Bronze 层（青铜层 / 原始数据层）

Bronze 只做一件事：**把源数据接进来，保持原貌**。

在 `nyc_transport` 数据库中，Bronze 层已包含以下表：

| 英文表名 | 中文表名 | 数据域 | 来源路径 |
|---|---|---|---|
| `bronze.yellow_tripdata_2026q1` | 黄色出租车行程（2026 Q1） | 出行域 | Parquet 导入 |
| `bronze.green_tripdata_2026q1` | 绿色出租车行程（2026 Q1） | 出行域 | Parquet 导入 |
| `bronze.fhv_tripdata_2026q1` | FHV 网约车行程（2026 Q1） | 出行域 | Parquet 导入 |
| `bronze.fhvhv_tripdata_2026q1` | FHVHV 高端网约车行程（2026 Q1） | 出行域 | Parquet 导入 |
| `bronze.parking_violations_all` | 停车罚单全量表 | 监管合规域 | CSV 导入 |
| `bronze.crash_merged` | 交通事故主表 | 安全域 | CSV 导入 |
| `bronze.crash_person_all` | 事故人员明细表 | 安全域 | CSV 导入 |
| `bronze.active_vehicles` | 活跃车辆注册表 | 资产域 | CSV 快照 |
| `bronze.fhv_active_vehicles` | FHV 活跃车辆快照 | 资产域 | CSV 快照 |
| `bronze.fhv_active_drivers` | FHV 活跃司机快照 | 供给域 | CSV 快照 |
| `bronze.shl_active_drivers` | SHL 活跃司机快照 | 供给域 | CSV 快照 |
| `bronze.medallion_authorized_vehicles` | Medallion 授权车辆表 | 监管合规域 | CSV 导入 |
| `bronze.taxi_zone_lookup` | 出租车区域字典表 | 空间地理域 | CSV 导入 |
| `bronze.tif_medallion_payments` | TIF 牌照支付记录表 | 监管合规域 | CSV 导入 |
| `bronze.new_driver_applications` | 新司机申请状态表 | 供给域 | CSV 导入 |

Bronze 层的设计原则：

- **不做复杂清洗**——数据直接从 CSV/Parquet 导入，字段名、类型尽量保持原样。
- **可追溯**——任何时候都能回到 Bronze 层查看"原始数据长什么样"。
- **可复现**——清洗逻辑出了问题，直接从 Bronze 重跑即可，不需要重新拉取外部数据。
- **保留原始字段**——即使某些字段看起来没用，也先留着，后续可能会有分析需求。

> 你之前看到的 `yellow_tripdata_2026q1`（黄色出租车行程）就在这一层。它的字段名还是 TLC 官方的原始命名（如 `tpep_pickup_datetime`、`PULocationID`），这正是 Bronze 层的特征——**数据保持原样，不做任何修改，可能包含脏数据或格式不统一的情况**。

### 6.2 🥈 Silver 层（白银层 / 清洗数据层）

Silver 层开始做**标准化和治理**。这是整个 Lakehouse 最关键的一层——数据质量的高低，直接决定了 Gold 层分析的可靠性。

以行程表为例，需要把四张不同源表统一成一张标准明细表：

```text
silver.trip_detail（行程明细标准表）
├─ trip_id（行程ID）                    -- 主键，全局唯一
├─ trip_source（行程来源）              -- yellow / green / fhv / fhvhv
├─ pickup_at（上车时间）                -- 统一为 TIMESTAMP
├─ dropoff_at（下车时间）               -- 统一为 TIMESTAMP
├─ pickup_location_id（上车区域ID）      -- 统一字段名
├─ dropoff_location_id（下车区域ID）     -- 统一字段名
├─ base_no（调度基地编号）
├─ passenger_count（乘客数）
├─ distance_miles（行程距离/英里）       -- 统一单位
├─ fare_amount（基础费用/美元）
├─ total_amount（总费用/美元）
├─ is_time_anomaly（是否时间异常）       -- 质量标记
├─ is_location_missing（是否位置缺失）   -- 质量标记
├─ quality_flags（综合质量标记位）
└─ source_table（来源表）               -- 溯源：来自哪张 Bronze 表
```

Silver 层的核心工作是**统一口径**：

| 原始字段（Bronze 层） | 标准化字段（Silver 层） | 中文字段名 | 说明 |
|---|---|---|---|
| `tpep_pickup_datetime` | `pickup_at` | 上车时间 | Yellow 出租车的时间字段 |
| `lpep_pickup_datetime` | `pickup_at` | 上车时间 | Green 出租车的时间字段 |
| `pickup_datetime` | `pickup_at` | 上车时间 | FHV/FHVHV 的时间字段 |
| `PULocationID` / `PUlocationID` | `pickup_location_id` | 上车区域ID | 不同表的区域字段大小写不一致 |
| `DOLocationID` / `DOlocationID` | `dropoff_location_id` | 下车区域ID | 同上 |
| `trip_distance` / `trip_miles` | `distance_miles` | 行程距离（英里） | 统一为英里 |

停车罚单也需要在 Silver 层治理：

```text
silver.parking_violation_detail（停车罚单明细标准表）
├─ violation_id（罚单ID）               -- 主键，代理键（Surrogate Key）
├─ summons_number（罚单编号）
├─ plate_no（车牌号）
├─ issue_date（开票日期）
├─ violation_code（违章代码）
├─ violation_description（违章描述）
├─ street_name（街道名称）
├─ fiscal_year（财年）
├─ is_duplicate_summons（是否重复罚单） -- 质量标记
└─ source_row_hash（来源行哈希）        -- 溯源字段
```

事故数据也要在 Silver 层治理：

```text
silver.crash_detail（事故明细标准表）
├─ collision_id（事故编号）             -- 主键，自然键
├─ collision_id（原始事故编号）
├─ crash_at（事故时间）                  -- 统一为 TIMESTAMP
├─ borough（行政区）
├─ zip_code（邮编）
├─ latitude（纬度）
├─ longitude（经度）
├─ persons_injured（受伤人数）
├─ persons_killed（死亡人数）
├─ is_duplicate_collision（是否重复事故）-- 质量标记
└─ source_row_hash（来源行哈希）         -- 溯源字段
```

Silver 层的设计原则：

- **数据质量更高**——去重、去空、格式统一、异常标记。
- **字段类型统一**——不同来源的同类数据使用一致的字段名和类型。
- **可复现的清洗逻辑**——每条 Silver 数据都能通过 `source_row_hash`（来源行哈希）追溯到对应的 Bronze 原始行。
- **日常分析和报表的基础**——大部分分析师直接使用 Silver 层数据做分析。

### 6.3 🥇 Gold 层（黄金层 / 业务数据层）

Gold 层存放**高度聚合、面向业务需求**的数据。这一层的数据形态通常是：

- **宽表（Wide Table）**：把多张 Silver 表 Join 在一起，方便 BI 工具直接使用。
- **汇总表（Summary Table）**：按时间、区域等维度预聚合。

例如：

```text
gold.daily_zone_trip_summary（每日各区域行程汇总表）
├─ date（日期）                         -- 维度
├─ pickup_location_id（上车区域ID）
├─ borough（行政区）                    -- 从 taxi_zone 维表 Join 得到
├─ trip_source（行程来源）
├─ trip_count（行程数）                 -- 指标
├─ total_distance（总距离/英里）        -- 指标
├─ total_fare（总费用/美元）            -- 指标
└─ avg_fare（平均费用/美元）            -- 指标

gold.monthly_violation_summary（月度罚单汇总表）
├─ month（月份）                        -- 维度
├─ violation_code（违章代码）
├─ violation_count（罚单数）            -- 指标
├─ unique_plate_count（去重车牌数）     -- 指标
└─ top_street_name（最高频街道）        -- 指标，待确认中文名

gold.zone_safety_summary（区域安全汇总表）
├─ borough（行政区）                    -- 维度
├─ crash_count（事故数）                -- 指标
├─ persons_injured（受伤人数）          -- 指标
├─ persons_killed（死亡人数）           -- 指标
└─ crash_density_level（事故密度等级）  -- 指标
```

Gold 层也可以包含面向 Agent 的应用视图：

```text
gold.agent_transport_overview（Agent 交通总览视图）
```

为 Agent 提供一口查询的总览数据：

- 总行程量
- 总罚单量
- 总事故量
- 活跃车辆数
- 活跃司机数
- 高风险区域

Gold 层的设计原则：

- **面向业务需求**——每张表都直接对应一个明确的业务问题或看板。
- **直接供 BI 工具使用**——Tableau、Power BI、Metabase 等直接连接 Gold 层。
- **查询性能优先**——预聚合、预 Join，避免在展示层做复杂计算。
- **数据口径统一**——所有高管看板和对外报表使用同一套 Gold 数据，确保"一个真相（Single Source of Truth）"。

### 6.4 📊 Meta 层（元数据层）

Meta 层存放"**关于数据的数据**"。它是数据治理的中枢，也是 Lakehouse 架构区别于传统数仓的一大特色——传统数仓中元数据通常散落在各层，而 Lakehouse 将其独立成一个 Schema 统一管理。

在 `nyc_transport` 中，Meta 层已经包含：

| 英文表名 | 中文表名 | 用途 |
|---|---|---|
| `meta.bronze_quality_summary` | Bronze 数据质量汇总表 | 监控 Bronze 层各表的数据质量指标 |
| `meta.table_catalog` | 表目录 | 登记所有表的中英文名、数据域、用途 |
| `meta.field_semantics` | 中文字段字典 | 每个字段的中文含义、数据类型、业务说明 |
| `meta.table_relationships` | 表关系说明 | 哪些表可以关联、用什么字段关联 |
| `meta.indicator_definitions` | 指标口径定义 | "行程量"怎么算、"平均费用"怎么算 |
| `meta.quality_rules` | 质量规则 | 哪些字段不允许为空、哪些数值有合理范围 |
| `meta.data_lineage` | 数据血缘 | Bronze → Silver → Gold 的加工链路 |

Meta 层也用于告诉 Agent 哪些问题可以问、哪些不能问：

| 问题类型 | 是否支持 | 原因 | 对应的 Meta 表 |
|---|---|---|---|
| 每日行程趋势 | ✅ 支持 | `silver.trip_detail`（行程明细标准表）有标准时间字段 | `meta.field_semantics`（中文字段字典） |
| 某司机完成了哪些行程 | ❌ 不支持 | 行程表没有司机级强外键 | `meta.table_relationships`（表关系说明） |
| 某区域事故和出行是否相关 | ⚠️ 支持但需说明口径 | 事故经纬度与 taxi zone 需要空间映射 | `meta.indicator_definitions`（指标口径定义） |
| 罚单按车牌统计 | ⚠️ 支持但需去重 | `summons_number`（罚单编号）当前不唯一 | `meta.quality_rules`（质量规则） |

Meta 层的设计原则：

- **数据质量监控**——定期检查各层数据是否符合质量标准（如你之前看到的 `bronze_quality_summary`）。
- **血缘追踪**——知道 Gold 层的每个数字是从哪条 Bronze 记录一步步算出来的。
- **中文语义字典**——让 Agent 和中文用户能理解每个字段的业务含义。
- **统一口径**——"什么是活跃车辆"、"什么是有效行程"，定义清楚，全员一致。

### 6.5 🏠 Main 层（默认主层）

Main 是 DuckDB 默认自带的 Schema。它的特点是：

- **默认工作空间**——当你不指定 Schema 时，DuckDB 自动使用 `main`。
- **开发和调试的临时落脚点**——可以在 Main 层创建临时表、测试视图、验证查询逻辑。
- **不建议存放正式数据**——正式数据应归属于 Bronze / Silver / Gold / Meta 四个层。
- **当前 `current = true`**——说明你现在正处于 Main 空间中。

在日常使用中的典型场景：

```sql
-- 在 Main 层做临时分析，验证逻辑
CREATE OR REPLACE TABLE main.tmp_zone_test AS
SELECT * FROM bronze.yellow_tripdata_2026q1  -- 黄色出租车行程
WHERE "PULocationID" = 237  -- 上东区（Upper East Side）
LIMIT 1000;

-- 验证通过后，清理临时表
DROP TABLE main.tmp_zone_test;
```

> **提示**：日常查询和开发习惯使用 Main 层做实验，但最终交付的分析结果应写回 Gold 层或 Meta 层。Main 层的临时表不会被纳入数据血缘追踪，也不应被 BI 工具直接引用。

## 8. 为什么不要一上来就做 Gold

因为当前还有几个基础治理问题：

| 问题 | 影响 |
|---|---|
| `parking_violations_all.summons_number`（罚单编号）不唯一 | 罚单事实表不能直接以它为主键 |
| `crash_merged.collision_id`（事故编号）不唯一 | 事故事实粒度需要确认 |
| `crash_person_all.unique_id`（人员记录编号）不唯一 | 事故人员明细需要去重或生成代理键 |
| 事故表与事故人员表 `collision_id`（事故编号）覆盖不完全 | 不能默认完整一对多 |
| TLC 行程存在异常时间 | 趋势分析需要质量标记 |
| FHV 行程上车区域缺失较高 | 空间分析需要降级说明 |
| 司机与行程缺少强外键 | 不能回答单司机行程明细 |

如果跳过 Silver 层直接做 Gold，Agent 很容易产生错误结论。

正确顺序是 Lakehouse 的逐层提纯逻辑：

```text
Bronze（青铜层 / 保真）→ Silver（白银层 / 标准化）→ Gold（黄金层 / 聚合）→ Meta（元数据层 / 治理）
```

---

## 9. 最终怎么理解这几类模型

可以用一句话理解：

```text
ER模型（实体关系模型）：讲清楚业务对象之间是什么关系。
维度建模：讲清楚分析什么指标、从哪些角度分析。
星型模型（Star Schema）：把一个事实表和多个维表组织成最容易分析的结构。
雪花模型（Snowflake Schema）：把复杂维表继续拆细，减少冗余但增加Join复杂度。
Bronze/Silver/Gold/Meta：把数据从原始接入到业务服务，逐层提纯落地。
```

结合本次数据，最实用的答案是：

```text
事故域（安全域）：先做 ER 子模型
出行/罚单/支付/申请：做事实表（Fact Table）
区域/车辆/司机/基地/违章代码：做维表（Dimension Table）
分析层：优先星型模型（Star Schema）
工程层：Bronze → Silver → Gold → Meta
Agent层：中文语义层（Meta）+ 指标口径 + 问数模板
```

---

## 10. 面向 Agent 的最终目标

最终不是为了"建很多表"，而是为了让 Agent 能安全回答问题。

一个成熟的 Agent 数据底座应该具备：

| 能力 | 当前 Meta 表（✅ 已建成） | 待建 Meta 表（🔄） | 说明 |
|---|---|---|---|
| 知道有哪些表 | `meta.source_tables` / `meta.v_bronze_tables` | — | 英文表名 + 数据域 + 行数 + 列数 |
| 知道表有哪些字段 | `meta.source_columns` / `meta.v_bronze_columns` | — | 英文字段名 + 数据类型 + 序号 |
| 知道字段中文含义 | — | `meta.field_semantics` | 英文字段名 → 中文字段名 + 业务含义 |
| 知道哪些数据有质量问题 | `meta.bronze_missing_matrix`、`meta.bronze_key_checks`、`meta.v_bronze_quality_issues` | — | 缺失率、候选键唯一性、问题汇总 |
| 知道整体质量概况 | `meta.bronze_quality_summary` | — | 每表的问题键数 + 高缺失字段数 |
| 知道哪些表能关联 | — | `meta.table_relationships` | 关联键、关联类型、关联口径 |
| 知道哪些字段能做指标 | — | `meta.indicator_definitions` | 指标名、计算公式、单位 |
| 知道哪些问题不能回答 | — | `meta.agent_question_templates` | 支持/不支持/有条件支持 + 原因 |

所以，纽约市城市交通数据最终应形成：

```text
DuckDB 数据库 nyc_transport
├─ 🥉 bronze（青铜层 / 原始数据层）— 原始接入，保持原貌
├─ 🥈 silver（白银层 / 清洗数据层）— 去重、标准化、统一字段
├─ 🥇 gold（黄金层 / 业务数据层）— 聚合宽表、BI 看板、Agent 视图
├─ 📊 meta（元数据层）— 质量监控、血缘追踪、中文语义字典
└─ 🏠 main（默认主层）— 开发调试、临时分析
```

这套 Data Lakehouse 架构既适合数据分析，也适合数据开发，更适合后续构建中文数据 Agent。每一层职责清晰、边界明确，数据从 Bronze 到 Gold 逐层提纯，Meta 层贯穿始终提供治理能力。

## 附录：DuckDB CLI 快速上手

### 启动与切换 Schema

```sql
-- 查看当前所在 Schema
SELECT current_schema;

-- 查看所有 Schema
SELECT schema_name FROM information_schema.schemata;

-- 切换到不同层
SET schema = 'bronze';   -- 进入青铜层（原始数据层）
SET schema = 'silver';   -- 进入白银层（清洗数据层）
SET schema = 'gold';     -- 进入黄金层（业务数据层）
SET schema = 'meta';     -- 进入元数据层
SET schema = 'main';     -- 回到默认主层
```

### 跨 Schema 查询（推荐）

不需要频繁切换 Schema，直接使用全限定表名即可：

```sql
-- 在 main 层直接查询 bronze 层数据
SELECT count(*) FROM bronze.yellow_tripdata_2026q1;  -- 黄色出租车行程

-- 跨层关联查询
SELECT
    b.count AS bronze_rows,     -- Bronze 层行数
    m.quality_score             -- 质量评分
FROM meta.bronze_quality_summary m  -- 数据质量汇总表
JOIN (
    SELECT 'yellow_tripdata_2026q1' AS table_name, count(*) AS count
    FROM bronze.yellow_tripdata_2026q1
) b ON m.table_name = b.table_name;
```

### 常用探索命令

```sql
-- 查看某层的所有表
SELECT table_name
FROM information_schema.tables
WHERE table_schema = 'bronze';

-- 查看表结构
DESCRIBE bronze.yellow_tripdata_2026q1;  -- 黄色出租车行程

-- 查看数据质量汇总
SELECT * FROM meta.bronze_quality_summary;  -- Bronze 数据质量汇总表
```

### 导出数据

```sql
-- 导出查询结果到 Parquet
COPY (SELECT * FROM gold.daily_zone_trip_summary)  -- 每日各区域行程汇总表
TO 'daily_summary.parquet' (FORMAT PARQUET);

-- 导出到 CSV
COPY (SELECT * FROM silver.trip_detail LIMIT 1000)  -- 行程明细标准表
TO 'trip_sample.csv' (HEADER, DELIMITER ',');
```
