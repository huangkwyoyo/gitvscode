# 经验复盘

本文件记录 TianShu 项目中已经发生过、值得长期保留的经验。每条经验必须尽量沉淀为规则、检查脚本和测试用例。

## R001：Silver 不得凭空新增 Bronze 不存在的金额字段

- 日期：2026-06-07
- 来源问题：`silver.parking_violation_detail` 曾规划 `fine_amount`、`payment_amount`、`amount_due` 等金额字段，但 Bronze 表 `parking_violations_all` 实际没有这些字段。
- 根因：规划时根据业务直觉补齐了“罚单应有金额”的字段，没有强制回查 Bronze schema 和数据库设计文档。
- 风险：Silver 层会出现虚假来源字段，Gold 指标和 Text2SQL 会继续放大错误。
- 规则：Silver 字段必须可追溯到 Bronze，或明确标注为 `derived` 并填写派生逻辑、可信等级和审核状态。
- 已落地检查：`scripts/quality/check_silver_dictionary.py`
- 已落地测试：`tests/test_silver_dictionary.py::test_parking_violation_no_fake_amount_fields`

## R002：DuckDB 禁用 `DATE::INT`

- 日期：2026-06-07
- 来源问题：`dim_date` 规划中曾使用 `DATE::INT` 生成 `date_key`，但 DuckDB 不支持 `DATE` 直接转 `INT`。
- 根因：规划文档中的 SQL 片段没有经过目标数据库方言验证。
- 风险：建表脚本运行失败，日期维表无法生成。
- 规则：DuckDB 中日期键必须使用 `strftime(date_value, '%Y%m%d')::INTEGER`。
- 已落地检查：`scripts/quality/check_dangerous_patterns.py`

## R003：稳定主键不得使用无序 `ROW_NUMBER() OVER ()`

- 日期：2026-06-07
- 来源问题：`trip_detail.trip_id` 曾计划使用无序行号生成代理键。
- 根因：没有区分“临时行号”和“可重复构建的稳定主键”。
- 风险：重跑 Silver 后主键变化，下游 Gold、语义层和缓存结果不可追溯。
- 规则：稳定主键必须使用自然键、复合键、稳定哈希，或带明确排序依据的代理键。
- 已落地检查：`scripts/quality/check_dangerous_patterns.py`

## R004：枚举值不得硬编码

- 日期：2026-06-07
- 来源问题：`driver_application_detail.status` 曾按规划写成 4 类，但实际 Bronze distinct 值可能更多。
- 根因：没有先用 `SELECT DISTINCT` 验证枚举全集。
- 风险：状态清洗丢失真实值，审批状态分析口径错误。
- 规则：枚举值必须来自 Bronze 实际 distinct、官方字典或人工审核结果，不得只凭样例硬编码。
- 已落地规则：`docs/warehouse/silver/AGENTS.md`

## R005：字段数字典、Markdown 和设计文档必须一致

- 日期：2026-06-07
- 来源问题：`trip_detail`、`crash_detail`、`crash_person_detail` 曾出现规划字段数与 Excel 字段数不一致。
- 根因：Markdown、生成脚本和 Excel 之间没有统一一致性检查。
- 风险：后续自动建表、字段字典和 Agent 元数据目录会产生偏差。
- 规则：字段数必须在数据库设计文档、Markdown 规划、Excel 字段字典之间一致。
- 已落地检查：`scripts/quality/check_silver_dictionary.py`
- 已落地测试：`tests/test_silver_dictionary.py::test_field_count_matches_plan`

## R006：金额字段必须使用 `DECIMAL`

- 日期：2026-06-07
- 来源问题：金额字段如果使用 `DOUBLE` 或 `FLOAT`，会引入浮点精度问题。
- 根因：未提前统一金额类型规范。
- 风险：财务、罚款、支付、收入类指标出现精度误差。
- 规则：金额字段统一使用 `DECIMAL`，推荐 `DECIMAL(12,2)`。
- 已落地测试：`tests/test_silver_dictionary.py::test_amount_fields_use_decimal`

## R007：MD5 代理键在 8,000 万行规模下可能碰撞

- 日期：2026-06-08
- 来源问题：`silver.trip_detail` 仅用 (trip_source, pickup_at, dropoff_at, pickup_location_id, dropoff_location_id, base_no) 做 MD5，实际出现了 86,171 次碰撞。
- 根因：同一秒内同一基地有多辆车在同一对区域之间接客，MD5 输入数据区分度不够。
- 风险：主键不唯一，下游 JOIN 会产生笛卡尔积，Gold 指标翻倍。
- 规则：MD5 代理键必须包含一个保证唯一的成分。对行程类数据，建议使用 `MD5(trip_source + '|' + CAST(source_row_no AS VARCHAR))`，其中 source_row_no 为带 ORDER BY 的 ROW_NUMBER。
- 已落地检查：`build_silver_duckdb.py` 每个建表函数末尾的 `GROUP BY pk HAVING n>1` 校验。
- 已落地规则：`AGENTS.md` §13.6 强制质量门禁。

## R008：DuckDB TRY_CAST 无法解析 ISO 日期格式拼接时间

- 日期：2026-06-08
- 来源问题：`crash_detail.crash_at` 全表 165 万行全部为 NULL。Bronze 数据中 `crash_date` 格式为 `2021-09-11T00:00:00.000`（ISO 格式），`crash_time` 为 `2:39`。直接 `crash_date || ' ' || crash_time` 拼接后 TRY_CAST 无法解析。
- 根因：ISO 格式日期字符串含 `T` 分隔符和时间部分 `00:00:00`，导致拼接后出现两个时间组件。
- 风险：事故时间维表完全失效，所有基于时间的分析（趋势、时段、季节性）不可用。
- 规则：DuckDB 中拼接日期和时间时，先确认日期列的格式。如为 ISO 格式，应先 `LEFT(date_col, 10)` 截取日期部分再拼接。**所有 TIMESTAMP 类型转换后必须立即验证 NULL 率**，不能假设 TRY_CAST 一定成功。
- 已落地检查：`build_silver_duckdb.py` 中 `build_crash_detail` 函数对 NULL crash_at 的显式统计。

## R009：DuckDB ROW_NUMBER 不能引用同层 SELECT 的别名

- 日期：2026-06-08
- 来源问题：多次出现 `Binder Error: Column "xxx" referenced that exists in the SELECT clause - but this column cannot be referenced before it is defined`。
- 根因：`ROW_NUMBER() OVER (ORDER BY alias_name)` 中 `alias_name` 是同一层 SELECT 的别名，DuckDB 不允许此引用。parking 表因 ORDER BY 引用原始列名恰好通过，tif/application 表因引用别名而失败。
- 风险：建表失败，浪费时间定位。
- 规则：`ROW_NUMBER() OVER (ORDER BY ...)` 中的排序列必须使用**源表原始列名**（如 `"License Number"`），不能使用同层别名。如果必须使用别名，先放到 CTE 中再 ROW_NUMBER。
- 已落地：`build_silver_duckdb.py` 所有 ROW_NUMBER 调用已统一为源列名或 CTE 模式。

## R010：DuckDB INSERT OR REPLACE 需要唯一约束

- 日期：2026-06-08
- 来源问题：`write_meta_comments` 首次运行时 `INSERT OR REPLACE INTO meta.table_comments` 报 `Binder Error: There are no UNIQUE/PRIMARY KEY constraints that refer to this table`。
- 根因：MySQL/SQLite 兼容语法在 DuckDB 中需要显式唯一约束才能使用 `OR REPLACE`。
- 风险：Meta 注释写入失败，后续建表进度被阻塞。
- 规则：DuckDB 中 `INSERT OR REPLACE` → `DELETE WHERE ...; INSERT INTO ...`。所有 Meta 写入操作先删后插。
- 已落地：`build_silver_duckdb.py` 中 `write_meta_comments` 已改为 DELETE + INSERT 模式。

## R011：建表完成后未同步更新关联文档

- 日期：2026-06-08
- 来源问题：Silver 11 张表全部建成后，`sql/silver/README.md`、`sql/meta/README.md`、`PROJECT_STATUS.md` 等文件仍保持建表前的状态。
- 根因：项目没有定义"变更传播规则"——即改完 X 之后还要改哪些 Y。
- 风险：协作者看到 `sql/silver/README.md` 还以为 Silver 是空的；`PROJECT_STATUS.md` 写"准备进入 Silver"而实际已完成。
- 规则：`AGENTS.md` §13 定义了完整的变更传播清单（11 类文件）和执行顺序。
- 已落地规则：`AGENTS.md` §13 变更传播规则 + §13.6 强制质量门禁。

## R012：AGENTS 规则不是自动触发器，必须有 Memory Gate 执行

- 日期：2026-06-09
- 来源问题：`AGENTS.md` 已写入变更传播规则，但修复 Silver 构建问题后，项目不会自动把经验写入 `docs/memory`。
- 根因：`AGENTS.md` 是规则声明，不会读取 git diff、不会判断关键路径变更，也不会阻止提交。
- 风险：Agent 或研发知道“应该写复盘”，但实际可能漏写，导致同类问题下次继续出现。
- 规则：关键路径发生变更时，必须由脚本检查是否同步更新核心记忆文件。
- 已落地检查：`scripts/quality/check_memory_update.py`
- 已接入门禁：`scripts/quality/run_all_checks.py`

## R013：Silver 建成后必须启用实表强校验

- 日期：2026-06-09
- 来源问题：Silver 建成后，字段字典、数据库设计和 DuckDB 实表一度出现字段数漂移：`parking_violation_detail`、`tif_payment_detail`、`crash_detail`、`crash_person_detail` 不一致。
- 根因：建表前 Harness 允许跳过 Silver 实表检查，但 Silver 建成后没有默认启用 `--require-silver-tables`。
- 风险：Gold 层可能基于错误字段结构继续建模，文档、字典、实表三套事实源分裂。
- 规则：项目阶段进入 `post_silver_build` 后，schema 一致性检查必须启用 Silver 实表强校验。
- 已落地检查：`harness/config/harness_targets.yml` 增加 `stage: post_silver_build`；`run_all_checks.py` 根据阶段自动追加 `--require-silver-tables`。

## R014：TRY_CAST 静默失败必须用格式化解析和空值检查兜底

- 日期：2026-06-09
- 来源问题：`driver_application_detail.app_date`、`driver_detail.expiration_date`、`tif_payment_detail.total_payment_amount`、`base_detail.total_dispatched_trips` 等字段曾全表或大比例 NULL。
- 根因：DuckDB `TRY_CAST` 遇到美国日期格式、带 `$` 和逗号的金额、千位分隔整数时会返回 NULL，不会报错。
- 风险：表能成功建成，但关键日期、金额字段不可用，Gold 指标会基于空值产生错误结论。
- 规则：非 ISO 日期必须使用 `try_strptime` 指定格式；货币字符串必须先去除 `$` 和逗号再转 `DECIMAL`；千位分隔整数必须先 `regexp_replace(col, ',', '', 'g')` 再转 `BIGINT`；建表后必须运行 `check_silver_null.py`。
- 已落地修复：`scripts/silver/build_silver_duckdb.py`
- 已落地检查：`scripts/quality/check_silver_null.py`

## R015：Gold 建表前必须先做设计门禁

- 日期：2026-06-09
- 来源问题：进入 Gold 层前，字段级设计文档已经存在，但 Harness 还不能自动拦截缺少中文名、来源字段不存在、误用翻译草稿等问题。
- 根因：Silver 阶段的检查主要覆盖 Silver 字典、实表和空值画像，尚未把 Gold 设计文档当作可校验的事实源。
- 风险：Gold 事实表或维表可能引用不存在的 Silver 字段，或者把未经审核的中文名、指标名带入语义层，导致 Text2SQL Agent 使用错误口径。
- 规则：Gold 建表前必须运行 `check_gold_design.py`，检查字段中文名、英文类型、Silver 来源字段是否真实存在；Google 翻译和 LLM 翻译只能作为草稿，不能作为正式中文名事实源。
- 已落地规则文件：`docs/warehouse/gold/AGENTS.md`、`docs/warehouse/database_design/gold_database_design.md`
- 已落地检查：`scripts/quality/check_gold_design.py`
- 已落地测试：`tests/test_gold_design_quality.py`

## R016：Gold 建表后必须做物理表门禁

- 日期：2026-06-09
- 来源问题：Silver 阶段曾出现“文档存在但实表字段、注释、空值画像没有同步拦截”的问题；Gold 如果只校验设计文档，仍可能在落库后产生实表漂移。
- 根因：设计门禁只能证明文档可落地，不能证明 DuckDB 实际表已经按文档建成，也不能证明中文表注释和字段注释已写入 `meta`。
- 风险：Agent 查询 Gold 时可能拿到缺少中文注释的表，或者事实源文档与 DuckDB 实表分裂。
- 规则：Gold G0/G1 建成后必须运行 `check_gold_physical.py --batches G0,G1`，检查表存在、行数非空、字段集合一致、中文表注释和中文字段注释完整。
- 已落地规则文件：`docs/warehouse/gold/AGENTS.md`、`harness/config/harness_targets.yml`
- 已落地检查：`scripts/quality/check_gold_physical.py`
- 已落地测试：`tests/test_gold_build_quality.py`

## R017：每层数据变换都需要独立的空值画像门禁

- 日期：2026-06-09
- 来源问题：Codex 在构建 Gold 门禁时，补了 `check_gold_design.py`（设计门禁）和 `check_gold_physical.py`（物理表门禁），覆盖了结构正确性，但遗漏了数据正确性——没有补 `check_gold_null.py`。这意味着 Gold 表建完后，如果 JOIN/聚合/CASE WHEN 产生了非预期的全 NULL 字段，Harness 不会拦截。
- 根因：Silver 的 NULL 问题被 R014 归类为"TRY_CAST 格式转换失败"，`check_silver_null.py` 和 `silver_sparsity_baseline.yml` 的命名全部带了 "Silver"。Codex 将其理解为 Silver 特有的格式转换问题，没有抽象为"任何层的数据变换都需要空值画像 + 预期稀疏基线"的通用模式。**命名决定了认知边界——如果脚本叫 `check_layer_null.py`，Codex 大概率会复用。**
- 风险：Gold 层虽然不直接做 Bronze→Silver 的格式转换，但 LEFT JOIN 维表断裂、聚合空子集、CASE WHEN 分支遗漏同样会产生非预期 NULL。如果每层都等到踩坑才补门禁，项目会在每一层重复同样的"表建成了、数据有 bug、门禁没拦截"循环。
- 规则：当某个检查脚本在一个层验证有效后，必须在下一层落地前同步创建对应版本。具体到本项目的层结构：
  - `check_silver_null.py` → 建 Gold 前必须已有 `check_gold_null.py`
  - `silver_sparsity_baseline.yml` → 建 Gold 前必须已有 `gold_sparsity_baseline.yml`
  - Gold 空值基线的内容不同（JOIN 断裂而非 TRY_CAST），但机制完全一致
- 已落地检查：`scripts/quality/check_gold_null.py`
- 已落地基线：`harness/config/gold_sparsity_baseline.yml`
- 已接入门禁：`run_all_checks.py` + `harness_targets.yml`
