# Harness 演进路线图（Harness Evolution Roadmap）

> 从"项目级 Agent 工具链"演进到"企业级 Agent 数据平台治理底座"的完整路径。

---

## 1. Harness 成熟度模型

### 1.1 五级成熟度定义

| 等级 | 名称 | 核心特征 | 适用场景 |
|:----:|------|---------|---------|
| **L1** | 实验级 | 单次脚本，手动执行，无自动化 | 个人探索、POC |
| **L2** | 项目级 | 有固定脚本，有基本检查，依赖人工触发 | 小型项目、单人维护 |
| **L3** | 团队级 | 有检查体系、有门禁、有测试、有文档约束 | 多人协作、有 Agent 参与 |
| **L4** | 平台级 | 自动化触发、有审计追踪、有自愈能力、可跨项目复用 | 多项目、多 Agent、有合规要求 |
| **L5** | 企业级 | 全自动化治理、联邦式知识管理、自修复闭环、元治理 | 企业级数据平台、多团队多 Agent |

### 1.2 当前 TianShu Harness 评级

| 维度 | 当前水平 | 证据 | 评分 |
|------|---------|------|:--:|
| **检查体系** | L3+ | 9 个质量检查脚本 + 4 个测试模块 + pre-commit hook | 4 |
| **自动化触发** | L3 | pre-commit hook（半自动）+ run_all_checks.py（手动） | 3 |
| **文档约束** | L3+ | AGENTS.md 路由表 + 分层规则 + 契约文件 + 变更传播规则 | 4 |
| **测试覆盖** | L3 | 4 个测试模块（silver_dictionary, harness_quality, gold_design, gold_build）+ Text2SQL 评测 | 3 |
| **记忆治理** | L3- | 内容质量校验 + 生命周期状态，但缺少置信等级、版本管理、自动通知 | 2.5 |
| **审计追踪** | L2+ | 经验复盘 + 规则来源索引，但缺少结构化审计日志和自动影响分析 | 2.5 |
| **自愈能力** | L2 | 错误记忆修复流程已定义（Review Agent §2.8.4），但全手动执行 | 2 |
| **多 Agent 协作** | L3- | 3 个 Agent 角色分工（dev/review/text2sql），但缺少协作协议 | 2.5 |
| **元治理** | L1 | 无"治理系统的治理"机制 | 1 |
| **跨项目复用** | L2 | 架构抽象良好但耦合于 NYC 交通数据场景 | 2 |

**综合评级：L3（团队级），接近 L3+**

## 7. 关键决策点

1. **Phase 1 是否立即启动？** → 建议是。P0 风险（GR-001~GR-006）直接威胁记忆系统的可信度，越早修复成本越低。

2. **是否抽象 Harness 框架？** → 建议在 Phase 2 中评估。如果 TianShu 是唯一使用此框架的项目，不必过早抽象。如果计划在其他数据仓库项目中复用，则值得投资。

3. **是否需要 L5 企业级能力？** → 取决于 TianShu 的长期定位。如果是"NYC 交通数据的一次性分析项目"，L3 已足够。如果是"可复用的 Agent 数据平台方法论"，L4 是合理的长期目标。L5 仅在企业级多团队多 Agent 场景下有必要。
