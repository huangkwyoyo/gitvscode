# 治理风险登记册（Governance Risk Register）

> 本文档是 TianShu 治理体系的 P0/P1/P2 风险汇总，与 `docs/memory/风险清单.md`（工程风险）互补。
> 工程风险关注"数据会不会错"，治理风险关注"系统能不能长期演进不被污染"。

---

## P0 — 必须立即修复（阻碍长期演进的关键缺失）

### GR-001：记忆条目缺少事实置信等级（Fact Confidence Level）

- **风险类型**：记忆治理
- **现状**：记忆条目仅有生命周期状态（active/disputed/superseded/archived），没有事实置信等级。Agent 读取记忆时无法区分"已验证事实"和"推测性结论"。
- **影响**：Agent 可能将所有 active 条目同等对待，推测性结论被当作事实引用，形成错误决策链。
- **建议方案**：为每条记忆增加 `- 置信等级：` 字段，枚举值：`Verified Fact` / `Fact` / `Assumption` / `Hypothesis` / `Human Reviewed`。Agent 必须根据置信等级调整引用行为。
- **关联风险**：RISK-029

### GR-002：不存在自动检测错误记忆的机制

- **风险类型**：记忆治理
- **现状**：错误记忆的发现完全依赖人工发现或 Review Agent 事后审查。没有自动化的"记忆与事实源交叉验证"机制。
- **影响**：错误记忆可能在系统中潜伏数月，直到下游 Agent 产生错误输出才被发现。
- **建议方案**：在 `check_memory_update.py --check-content` 中增加"与 database_design 交叉验证"能力——检测记忆条目中引用的表名、字段名是否在 database_design 中真实存在。

### GR-003：Memory 写入与 Agent 引用之间缺少"冷却期"

- **风险类型**：记忆治理
- **现状**：记忆一旦写入（状态=active），立即对所有 Agent 可见。没有机制确保新记忆在生效前经过充分审查。
- **影响**：Agent A 写入错误记忆 → Agent B 在同一次会话中读取 → 错误立即传播。冷却期是"人类可以介入修正"的时间窗口。
- **建议方案**：新增 `proposed` 状态，所有新写入的记忆默认状态为 `proposed`，需经 Review Agent 或人工确认后才能转为 `active`。

### GR-004：AGENTS.md 路由表没有强制执行机制

- **风险类型**：知识治理
- **现状**：根 `AGENTS.md` 定义了详细的前置阅读路由表，但 Agent 是否真的读了这些文件没有验证机制。Agent 可以声称"已读"但实际跳过。
- **影响**：Agent 在缺少上下文的情况下开始工作，做出的决策可能违反已定义的规则。
- **建议方案**：在 Review Agent §2 中增加"前置阅读执行确认"检查项——要求 Agent 引用读完的文件中的具体内容作为证明。

### GR-005：Schema 变更缺少强制人工审批节点

- **风险类型**：Schema 治理
- **现状**：数据库设计文档是"最高事实源"，但 Dev Agent 可以直接修改它（场景 A/B）。Review Agent 是 AI，PR 审批取决于人类是否真的看了。
- **影响**：AI Agent → 修改 database_design → AI Review Agent 通过 → 人类匆忙合入 → 错误 schema 成为"事实"。双向污染闭环。
- **建议方案**：对 `docs/warehouse/database_design/` 的变更增加 `CODEOWNERS` 文件，要求特定人员审批。在 pre-commit hook 中增加"database_design 变更必须附带人类审批记录"的检查。

### GR-006：不存在"记忆消费端"的自动通知机制

- **风险类型**：记忆治理
- **现状**：当一条记忆被标记为 disputed 或 superseded 后，没有机制通知已经引用过该记忆的 Agent 或下游产物。
- **影响**：已经基于错误记忆生成的规则、检查脚本、测试可能继续存在，错误知识链不会自动断裂。
- **建议方案**：在 `规则来源索引.md` 中增加"被引用者"反向索引——每条规则记录哪些检查脚本/测试/AGENTS.md 引用了它。标记 disputed 时，自动输出受影响的下游清单。

## P1 — 建议尽快修复（影响可审计性和可追溯性）

### GR-007：记忆条目缺少结构化来源字段

- **风险类型**：记忆治理
- **现状**：来源追溯仅有 `- 来源追溯：` 一个自由文本字段。没有结构化的：`- 来源文档：`、`- 来源 PR：`、`- 来源人工确认：`、`- 来源数据：`。
- **影响**：事后审计时无法快速定位记忆的来源类型，只能阅读自由文本。自动验证也无法按来源类型分类处理。
- **建议方案**：在 `变更复盘模板.md` 中增加结构化来源字段，`check_memory_update.py` 相应增加结构化字段检测。

### GR-008：记忆条目缺少版本号管理

- **风险类型**：记忆治理
- **现状**：同一条规则（如 R001）只有一次写入，后续修正通过状态标记（superseded）指向新规则。缺少同编号下的 v1/v2/v3 演进记录。
- **影响**：无法追溯一条规则是如何演进的，也看不出每次修正的具体变化。
- **建议方案**：增加 `- 版本：v1` 字段，同一 R 编号可以有多个版本条目。旧版本标记为 superseded，新版本写入时在 `- 废弃原因：` 中说明 v1→v2 的具体变更。

### GR-009：缺少"记忆引用计数"机制

- **风险类型**：记忆治理
- **现状**：无法知道一条记忆被多少下游规则、检查脚本、测试引用。废弃记忆时无法评估影响范围。
- **影响**：废弃一条记忆可能导致下游断裂，但无法提前预警。
- **建议方案**：在 `规则来源索引.md` 中维护正向和反向引用计数。每次新增引用时更新。

### GR-010：契约文件（contracts/）没有版本锁定

- **风险类型**：知识治理
- **现状**：`contracts/` 下的 YAML 文件定义了 Agent 行为的硬约束，但没有版本号、没有变更日志、没有审批记录。
- **影响**：契约文件被修改后，所有 Agent 的行为立即改变。如果修改是错误的，影响范围是全域的。
- **建议方案**：在每个契约文件头部增加 `version:` 和 `last_reviewed:` 字段。修改契约文件必须通过 PR + 人工审批。

### GR-011：决策记录（ADR）没有定期复审机制

- **风险类型**：知识治理
- **现状**：6 份 ADR 全部标记为 "Accepted"，没有定期复审计划。决策的前提条件可能已经改变，但 ADR 不会自动提醒复审。
- **影响**：项目可能在过时决策的约束下运行，错失更好的方案。
- **建议方案**：在每份 ADR 中增加 "复审日期" 字段。在 `run_all_checks.py` 中增加 ADR 过期提醒（如超过 90 天未复审的 Accepted ADR 输出警告）。

### GR-012：未定义 "Human Review" 的完整流程

- **风险类型**：知识治理
- **现状**：项目中大量出现 `Human Review` 标记（字段含义、枚举值、指标口径），但没有定义：谁来 Review？多久内 Review？Review 后如何更新？不 Review 会怎样？
- **影响**：`Human Review` 可能成为"永远不 Review"的标记，关键字段长期处于不确定状态。
- **建议方案**：创建 `docs/governance/human_review_process.md`，定义 Human Review 的触发、指派、时限、升级和关闭流程。

### GR-013：缺少知识文档的生命周期管理

- **风险类型**：知识治理
- **现状**：docs/ 下的文档没有"创建日期"、"最后更新日期"、"复审周期"、"过期条件"等元数据。
- **影响**：无法判断一份文档是否过期。Agent 可能基于过期文档做出决策。
- **建议方案**：在每份知识文档头部增加 YAML frontmatter（日期、状态、复审周期）。在 `check_memory_update.py` 中增加文档过期检测。

### GR-014：缺少对 Agent 产出的"真实性抽样审计"

- **风险类型**：Schema 治理
- **现状**：Review Agent 审查每次变更，但没有机制对历史变更进行抽样复查——Agent 的审查可能漏过某些类型的错误。
- **影响**：系统性偏差可能在多次变更中积累，单次审查无法发现。
- **建议方案**：在 `run_all_checks.py` 中增加"历史变更抽样审计"模块，随机抽取历史变更进行交叉验证。

## P2 — 优化项（提升成熟度）

### GR-015：记忆条目缺少可信度数值评分

- **风险类型**：记忆治理
- **现状**：没有 0-100 的可信度评分机制。Agent 无法根据可信度调整记忆使用策略。
- **影响**：低可信度记忆和高可信度记忆被同等对待。
- **建议方案**：设计 Confidence Score 计算模型（来源可靠性 × 交叉验证次数 × 时间衰减因子）。Agent 根据分数决定是"直接引用"、"参考使用"还是"仅作提示"。

### GR-016：缺少记忆图谱（Memory Graph）

- **风险类型**：记忆治理
- **现状**：记忆条目是扁平的 R001-R022 列表，没有关联关系（如 R001→R020 都涉及金额治理，但无显式链接）。
- **影响**：Agent 无法通过关联关系发现相关记忆，可能遗漏重要上下文。
- **建议方案**：在 `规则来源索引.md` 中增加"关联规则"列，记录规则之间的依赖、互补、矛盾关系。

### GR-017：缺少自动化治理仪表板

- **风险类型**：治理可见性
- **现状**：治理状态分散在多个文件和脚本输出中，没有一个集中的健康度视图。
- **影响**：项目负责人无法快速了解当前治理状态（多少条记忆处于 disputed、多少字段标记 Human Review、多少 ADR 待复审）。
- **建议方案**：创建 `scripts/quality/governance_dashboard.py`，聚合所有治理指标，输出单页健康度报告。

### GR-018：Harness 配置文件缺少 schema 校验

- **风险类型**：配置治理
- **现状**：`harness_targets.yml` 等配置文件没有 JSON Schema 校验。配置错误只能在运行时发现。
- **影响**：配置漂移可能导致检查被静默跳过。
- **建议方案**：为 `harness_targets.yml` 定义 JSON Schema，在 `run_all_checks.py` 启动时自动校验。

### GR-019：缺少多 Agent 协作的"信息来源声明"规范

- **风险类型**：协作治理
- **现状**：当多个 Agent 协作时，没有规范要求 Agent 声明信息的来源和可信度。
- **影响**：Agent B 基于 Agent A 的输出继续工作，但不知道 Agent A 的结论是基于事实还是推测。
- **建议方案**：在根 `AGENTS.md` 中增加"信息来源声明"规范——Agent 输出必须标注每个结论的来源类型（Bronze 实际数据 / database_design / 经验记忆 / 推理）。

### GR-020：缺少对"治理系统自身"的质量保证

- **风险类型**：元治理
- **现状**：检查脚本、AGENTS.md、Review Agent 规则定义了治理行为，但没有机制验证治理系统自身是否正确运行。
- **影响**：治理系统可能被绕过、误配置或退化，但无人察觉。
- **建议方案**：创建"治理自检"模块——定期验证所有门禁是否真的在运行、Review Agent 是否真的在审查、路由表是否真的被读取。
