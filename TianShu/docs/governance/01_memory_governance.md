## 2. 设计目标

TianShu Memory 治理体系应实现：

1. **写入有门禁**：每条记忆在写入时通过内容质量校验 ✓（已实现）
2. **事实有等级**：Agent 知道每条记忆的可信程度 ✗（待建设）
3. **来源可追溯**：每条记忆能追溯到具体的人、文档、数据或 PR ✗（部分）
4. **版本可管理**：同一条规则的演进历史可追踪 ✗（待建设）
5. **错误可撤销**：错误记忆可以被发现、标记、废弃，并自动通知下游 ✗（部分）
6. **影响可评估**：废弃一条记忆前知道会影响哪些下游 ✗（待建设）
7. **可信度可量化**：Agent 根据可信度评分调整记忆使用策略 ✗（待建设）

## 3. 核心机制设计

### 3.1 事实置信等级（Fact Confidence Level）

#### 3.1.1 等级定义

每条记忆条目必须声明其事实置信等级：

| 等级 | 英文名 | 含义 | Agent 处理方式 |
|------|--------|------|---------------|
| **L5** | `Verified Fact` | 经多源交叉验证、人工审核确认的事实 | 直接引用，可作决策依据 |
| **L4** | `Fact` | 基于数据画像、DESCRIBE 结果或代码审查确认的事实 | 直接引用，标注来源 |
| **L3** | `Assumption` | 基于已有规则推导的合理假设，但未经独立验证 | 可引用，但必须标注"据推测" |
| **L2** | `Hypothesis` | 有依据的猜测，尚未被验证 | 仅作参考，不得作为唯一决策依据 |
| **L1** | `Human Reviewed` | 已经过人工确认，但具体证据链已不可追溯 | 可引用，但优先级低于 L4/L5 |

#### 3.1.2 等级判定规则

```
初始等级由写入者根据证据强度设定
  ↓
每次交叉验证通过 → 等级上升一级（最高 L5）
  ↓
每次发现与事实源冲突 → 等级下降一级
  ↓
下降至 L2 以下 → 自动触发 disputed 状态
```

#### 3.1.3 模板变更

在 `变更复盘模板.md` 中为每个条目增加：

```markdown
- 置信等级：L4（基于 DuckDB DESCRIBE 结果确认）
- 验证记录：
  - 2026-06-07：初始写入，基于数据画像 → L3
  - 2026-06-08：Review Agent 交叉验证通过 → L4
```

#### 3.1.4 Agent 引用规则

Agent 读取记忆时，必须根据置信等级调整行为：

| 场景 | L5 | L4 | L3 | L2 | L1 |
|------|:--:|:--:|:--:|:--:|:--:|
| 作为决策的唯一依据 | ✅ | ✅ | ❌ | ❌ | ⚠️ |
| 作为决策的参考之一 | ✅ | ✅ | ✅ | ✅ | ✅ |
| 写入 AGENTS.md 规则 | ✅ | ⚠️ | ❌ | ❌ | ❌ |
| 写入检查脚本 | ✅ | ✅ | ❌ | ❌ | ❌ |
| 写入测试用例 | ✅ | ✅ | ⚠️ | ❌ | ❌ |

### 3.2 结构化来源追踪（Structured Source Traceability）

#### 3.2.1 来源类型定义

每条记忆必须至少包含以下来源维度之一（可多个）：

| 来源类型 | 字段名 | 示例 | 必填 |
|---------|--------|------|:---:|
| 来源文档 | `- 来源文档：` | `docs/warehouse/database_design/silver_database_design.md` | 建议 |
| 来源 PR | `- 来源 PR：` | `PR #42` | 建议 |
| 来源 Commit | `- 来源 Commit：` | `commit 7d490d7` | 建议 |
| 来源数据 | `- 来源数据：` | `DESCRIBE bronze.parking_violations_all` | 建议 |
| 来源人工确认 | `- 来源人工确认：` | `张三（2026-06-07 确认枚举值口径）` | 建议 |
| 来源 ADR | `- 来源 ADR：` | `ADR-006` | 建议 |

#### 3.2.2 自动验证规则

`check_memory_update.py --check-content` 中增加结构化来源验证：

- **Error**：所有 6 个结构化来源字段均为空
- **Warn**：仅填充了 ≤1 个结构化来源字段
- **Info**：来源类型为"人工确认"但缺少确认人姓名

#### 3.2.3 模板变更

```markdown
## RXXX：规则标题

- 日期：YYYY-MM-DD
- 状态：active
- 置信等级：L4（Fact）
- 版本：v1
- 来源文档：docs/warehouse/database_design/gold_database_design.md
- 来源 Commit：7d490d7
- 来源数据：DESCRIBE gold.fact_parking_violations（date_key 范围检测）
- 来源人工确认：张三（2026-06-10 确认异常日期处理策略）
- 来源 ADR：无
- 来源问题：...（自由文本，保留）
- 根因：...
- 风险：...
- 规则：...
```

### 3.3 版本管理（Version Management）

#### 3.3.1 版本模型

同一条规则可以有多个版本。每个版本是一个独立的条目段落。

```markdown
## R001：Silver 不得凭空新增 Bronze 不存在的字段

### v3（当前版本）
- 日期：2026-07-01
- 状态：active
- 版本：v3
- 变更说明：扩展规则覆盖范围至 Gold 层
- ...

### v2（已替代）
- 日期：2026-06-15
- 状态：superseded
- 版本：v2
- 废弃原因：v3 扩展了覆盖范围，增加了 Gold 层约束

### v1（已替代）
- 日期：2026-06-07
- 状态：superseded
- 版本：v1
- 废弃原因：v2 增加了 derived 字段的具体校验规则
```

#### 3.3.2 版本规则

- 同一 R 编号最多保留 3 个历史版本（v1, v2, v3），更早版本移至 `harness/lessons/` 归档
- 只有最新版本的 `状态` 可以是 `active`
- 所有历史版本的状态必须是 `superseded` 或 `archived`
- 每个版本的 `废弃原因` 必须说明被哪个版本替代、替代原因

### 3.4 撤销与通知机制（Revocation & Notification）

#### 3.4.1 完整撤销流程

```
┌─────────────────────────────────────────────────────┐
│  Step 1: 发现（Discovery）                          │
│  - 人工发现 / Review Agent 审查 / 自动交叉验证触发   │
├─────────────────────────────────────────────────────┤
│  Step 2: 标记存疑（Flag as Disputed）                │
│  - 修改条目状态为 disputed                           │
│  - 添加存疑原因和时间戳                              │
│  - 自动查找所有引用该条目的下游（规则来源索引反向索引）│
├─────────────────────────────────────────────────────┤
│  Step 3: 通知（Notify）                              │
│  - 输出受影响下游清单                                │
│  - 标记所有引用者状态为"待复核"                       │
├─────────────────────────────────────────────────────┤
│  Step 4: 核实（Verify）                              │
│  - 人工确认错误                                      │
│  - 确定修正方案                                      │
├─────────────────────────────────────────────────────┤
│  Step 5: 修正（Fix）                                 │
│  - 创建新版本条目（如适用）或修改为 superseded        │
│  - 修正所有下游污染                                  │
│  - 更新规则来源索引                                  │
├─────────────────────────────────────────────────────┤
│  Step 6: 锁定（Lock）                                │
│  - 错误条目保留原文（防止同一错误再次写入）           │
│  - 在条目中增加"已知错误模式"标签，供未来检测使用     │
└─────────────────────────────────────────────────────┘
```

#### 3.4.2 下游影响自动分析

在 `规则来源索引.md` 中增加"被引用者"反向索引：

| 规则编号 | 被以下规则引用 | 被以下脚本引用 | 被以下测试引用 | 被以下 AGENTS.md 引用 |
|---------|-------------|-------------|-------------|-------------------|
| R022 | — | check_gold_date_quality.py | — | gold/AGENTS.md §11 |
| R020 | — | check_gold_physical.py | test_gold_build_quality.py | gold/AGENTS.md §9 |

标记 `disputed` 时，脚本自动输出受影响清单：
```
[WARN] R020 被标记为 disputed，受影响的下游：
  - 检查脚本：scripts/quality/check_gold_physical.py（需复核 G2 批次检查逻辑）
  - 测试用例：tests/test_gold_build_quality.py::test_gold_g2_build_creates_fact_tables
  - 规则文件：docs/warehouse/gold/AGENTS.md §9 金额字段关联规则
```

### 3.5 可信度数值评分（Confidence Score）

#### 3.5.1 评分模型

```
Confidence Score = 来源可靠性分 × 交叉验证分 × 时间衰减系数

其中：
- 来源可靠性分（0-100）：
  - 多源交叉验证（database_design + DESCRIBE + 人工）：100
  - 单源数据画像：70
  - 单源代码审查：60
  - 推理/推测：30

- 交叉验证分（0.8-1.2）：
  - 每次与事实源交叉验证通过：+0.1（上限 1.2）
  - 每次发现与事实源不一致：-0.2（下限 0.8）
  - 初始值：1.0

- 时间衰减系数（0.5-1.0）：
  - 0-30 天：1.0
  - 31-90 天：0.9
  - 91-180 天：0.75
  - 181-365 天：0.6
  - >365 天：0.5
```

#### 3.5.2 Agent 行为规则

| 分数范围 | Agent 行为 |
|---------|-----------|
| 80-100 | 可直接引用，无需额外验证 |
| 60-79 | 可引用，但建议在应用前做一次快速验证 |
| 40-59 | 可引用，但必须标注"可信度中等，建议人工确认" |
| 20-39 | 仅作参考，不得作为决策依据 |
| 0-19 | 自动视为 `disputed`，Agent 不得引用 |
