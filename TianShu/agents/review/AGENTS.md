# Review Agent 规则

> 从属于根 `AGENTS.md`。Review Agent 是 Harness 的守门人，负责发现问题并阻止不合规变更。

## 1. 核心原则

Review Agent 不负责"帮忙圆回来"，而是负责发现问题。审查结果是"通过"或"不通过"，不存在"勉强通过"。

## 2. 必查项目

### 2.1 文档完整性
- [ ] 数据库设计文档是否存在
- [ ] 字段字典是否存在
- [ ] 新增/变更的表是否有对应的设计文档

### 2.2 字段来源合规
- [ ] 每个 Silver 字段是否能追溯到 Bronze DESCRIBE 结果
- [ ] 派生字段是否标注了来源、计算逻辑、可信等级
- [ ] 是否出现 Bronze 中不存在的来源字段名

### 2.3 类型合规
- [ ] 金额字段是否使用了 DECIMAL
- [ ] 日期字段是否使用了 DATE/TIMESTAMP
- [ ] 字段类型列是否使用英文类型名（非"文本""整数"）

### 2.4 中英文并列
- [ ] 表名是否有中英文
- [ ] 字段名是否有中英文
- [ ] 主键/候选键是否有中文说明

### 2.5 一致性检查
- [ ] Markdown 规划文档字段数 == XLSX 字段字典字段数
- [ ] SQL 建表脚本字段与数据库设计文档一致
- [ ] DuckDB 实际 schema 与数据库设计文档一致

### 2.6 危险模式扫描
- [ ] 是否出现 `DATE::INT`
- [ ] 是否出现无序 `ROW_NUMBER() OVER ()`
- [ ] 是否出现无来源说明的 `amount` / `fine` / `payment` 字段
- [ ] Gold 是否跳过 Silver 直接引用 Bronze
- [ ] Join 关系是否有数据画像或 Meta 支撑
- [ ] 枚举值是否写死（应以 SELECT DISTINCT 为准）

### 2.7 变更流程合规
- [ ] 变更前是否运行了基线检查（`run_all_checks.py`）
- [ ] 变更后是否运行了全量 Harness（`run_all_checks.py` 全部通过）
- [ ] 是否同步更新了数据库设计文档（`docs/warehouse/database_design/`）
- [ ] 是否同步更新了字段字典（`docs/warehouse/data_dictionary/`，如适用）
- [ ] 是否同步更新了标准问题集（`harness/questions/gold_standard_questions.yml`，如适用）
- [ ] 是否有对应的风险清单更新或经验复盘写入（`docs/memory/`，如触发根 §13.4 条件）

### 2.8 记忆准确性（Memory Accuracy）

> 本节是错误记忆写入的关键防线——Memory Gate 只检查"写没写"，本节检查"写得对不对"。

#### 2.8.1 内容质量校验
- [ ] 运行 `python scripts/quality/check_memory_update.py --content-only`，确认无 ERROR 级问题
- [ ] 新增/修改的记忆条目是否包含所有基础必填字段（日期、来源问题、根因、风险、规则）
- [ ] 新增条目是否包含治理必填字段（置信等级、版本、状态）——旧条目待迁移为 INFO 级建议
- [ ] 新增条目是否至少填写了 1 项结构化来源字段（来源文档/PR/Commit/数据/人工确认）
- [ ] 置信等级是否合规——初始写入时不得高于 L3，除非有硬证据支撑
- [ ] 条目内容是否达标最低长度（≥ 8 行），非一行敷衍
- [ ] 是否存在与已有条目正文完全相同的复制粘贴
- [ ] 交叉验证：条目中引用的表名/字段名是否在 database_design 中真实存在

#### 2.8.2 来源追溯验证
- [ ] 每条新增/修改的记忆是否有可追溯的来源标记（ADR 编号 / git commit / 具体事故描述）
- [ ] 来源问题字段是否包含足够的实质描述（≥ 20 字）

#### 2.8.3 事实源交叉验证
- [ ] 记忆中的"事实"是否与 `docs/warehouse/database_design/` 冲突（若冲突，以 database_design 为准，记忆条目需标注冲突点）
- [ ] 记忆中的"规则"是否与已有 AGENTS.md 规则矛盾（若矛盾，条目需说明是替代还是并存）
- [ ] 记忆引用的表名、字段名是否在项目 schema 中真实存在

#### 2.8.4 发现错误记忆时的处理
1. 将该条目标记为 `disputed`（修改状态字段为 `disputed`，说明存疑原因），**同时降级置信等级至 L2 或更低**
2. 运行 `python scripts/quality/check_memory_update.py --check-content` 获取自动检测的下游影响清单
3. 通知项目负责人进行人工核实
4. 核实确认有误后：改为 `superseded`，填写废弃原因和替代规则编号；如有修正后的规则，创建新版条目（v2）
5. 检查 `docs/memory/规则来源索引.md` 中是否有引用该条目的规则 → 更新反向索引
6. 检查是否有基于该错误记忆创建的检查脚本/测试 → 修正或废弃
7. **禁止直接删除错误条目**——保留原文是为了防止同一错误被再次写入，也是置信演化链的一部分

## 3. 审查结果

通过：所有必查项目 ✅
不通过：任一必查项目 ❌ → 列出违规项 → 退回修改 → 修改后重新审查
