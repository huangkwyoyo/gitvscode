# 电信数仓模拟数据生成规范

## 目的

本文档用于约束本项目的电信业务模拟数据生成、入库和质量校验。目标不是随机造数，而是生成能支撑数仓建模、指标分析、SQL 生成、Agent 查询和业务解释的可信模拟数据。

适用范围：

- `ods`、`dwd`、`dws`、`ads` 四层 MySQL 表
- Python 模拟数据生成器
- ETL 清洗和汇总逻辑
- 后续 FastAPI、LangGraph Agent、SQL 查询接口使用的数据质量约束

## 基本原则

1. 模拟数据必须符合电信业务逻辑。
2. 主外键必须可关联，不允许凭空生成孤立明细。
3. 批量日期、业务时间、生命周期时间必须语义清晰。
4. ODS、DWD、DWS、ADS 的数据规模关系必须符合数仓分层逻辑。
5. 所有模拟值必须接近真实业务场景，禁止使用明显占位值。
6. 每次生成和入库后必须执行质量校验。

## 批量日期规范

### 字段定义

所有物理表必须包含 `data_time` 字段。

字段要求：

| 规则 | 要求 |
|---|---|
| 字段名 | `data_time` |
| 字段含义 | 批量数据日期 |
| 数据类型 | `DATE` |
| 值格式 | `yyyy-MM-dd` |
| 字段位置 | 每张表最后一个字段 |
| 是否业务时间 | 否 |

### 批量日期和业务时间的区别

`data_time` 只表示数据批次，不表示业务发生时间。

示例：

| 字段 | 含义 | 示例 |
|---|---|---|
| `data_time` | 批量日期 | `2026-05-31` |
| `activation_time` | 入网时间 | `2024-09-12 10:31:22` |
| `business_time` | 业务办理时间 | `2026-05-18 14:22:09` |
| `payment_time` | 支付时间 | `2026-05-18 14:25:41` |
| `billing_cycle_date` | 账单账期 | `2026-05-01` |

## 批次生成规范

本项目第一版固定使用五个全量批次：

| 批次顺序 | 批量日期 | 用途 |
|---:|---|---|
| 1 | `2025-05-30` | 同比基准日前一日 |
| 2 | `2025-05-31` | 同比基准日 |
| 3 | `2026-04-30` | 环比基准月末 |
| 4 | `2026-05-30` | 当前日前一日 |
| 5 | `2026-05-31` | 当前分析日 |

批次规则：

- 每个批次是全量快照，不是增量快照。
- 后一批次用户总量应大于前一批次。
- 允许用户离网，但新增用户数应大于离网用户数。
- 同一用户跨批次应保持基础身份稳定，例如姓名、手机号、客户 ID、账户 ID 不应频繁变化。
- 用户状态、套餐、欠费、使用量、投诉、风险等字段可随批次变化。

### 跨批次身份缓存

生成器通过 `data/generated/user_identity_cache.json` 文件持久化用户身份信息，确保批次间身份一致。

缓存规则：

- 首批次（batch 1）执行时，所有用户的身份信息写入缓存文件。
- 后续批次执行时，已有用户直接从缓存读取稳定字段（姓名、手机号、入网日期、地域、性别、年龄、付款方式），仅动态字段（套餐、风险分）重新评估。
- 新增用户（user_id 超出缓存范围的）生成新身份并追加到缓存。
- 缓存文件位于 `data/generated/` 目录，与模拟数据加载文件同级。
- 如需全量重跑（从零生成所有身份），删除缓存文件后重新执行。

稳定字段（缓存后跨批次不变）：

| 字段 | 说明 |
|---|---|
| `user_id` | 用户唯一标识（确定性生成） |
| `customer_id` | 客户唯一标识 |
| `account_id` | 账户唯一标识 |
| `account_code` | 账户编码 |
| `customer_name` | 客户姓名 |
| `service_number_id` | 服务号码（手机号） |
| `gender_type` | 性别 |
| `age_count` | 年龄 |
| `activation_date` | 入网日期 |
| 省/市/区县/镇/网点 | 地域信息 |
| `payment_timing_type` | 付款方式（后付/预付） |
| `service_network_type` | 网络类型 |

动态字段（每批次重新评估）：

| 字段 | 说明 |
|---|---|
| `product_id` / 套餐 | 用户可能更换套餐，30% 概率保留原套餐类型 |
| `risk_score` | 风险评分每批次随机浮动 |

### 跨批次离网逻辑

用户离网是渐进式的累积过程，非一次性随机。

离网规则：

- 首批次初始离网率约 3-6%（与城市等级相关）。
- 后续批次对仍活跃的用户按概率离网：基础离网率 2%，高风险用户（risk_score > 0.85）额外 +3%，低线城市（tier ≥ 3）额外 +1%。
- 离网日期落在上一批次和当前批次之间（距入网至少 90 天）。
- 一旦离网（`is_active_subscriber = 0`），后续批次不可复活。
- 离网用户仍保留在维度表中（用于历史分析），但不再产生新的业务事实。

## ID 生成规范

### 用户、客户、账户 ID

核心 ID 必须接近真实业务系统风格，不允许使用简单占位编号。

| 字段 | 规则 |
|---|---|
| `user_id` | 11 位数字风格，不得为 0，不得表示用户数 |
| `customer_id` | 与用户 ID 不同，保持客户主体唯一 |
| `account_id` | 与用户 ID、客户 ID 不同，保持账户主体唯一 |
| `account_item_id` | 与 `account_id` 不得相等 |

禁止示例：

| 禁止值 | 原因 |
|---|---|
| `0` | 不是有效用户 ID |
| `900000000001` | 过于机械，不符合当前模拟风格 |
| `700000000001` | 过于机械，不符合当前模拟风格 |
| `A000000000001` | 明显占位编号 |
| `CODE_676` | 明显占位编码 |

### 编码字段

编码字段必须体现业务来源或业务对象。

| 字段 | 推荐风格 |
|---|---|
| `account_code` | `ACCT` + 年月 + 序列 |
| `zhetno_code` | `HT` + 年月 + 序列 |
| `zictno_code` | `XM` + 年月 + 序列 |
| `acquisition_staff_id` | 企业员工号风格 |
| 部门编码 | 按省、市、区县、网点层级生成 |

禁止使用：

- `CODE_`
- `S000001`
- `D00001`
- `CH_CODE`
- `DEPT_`
- `模拟值`

## 名称生成规范

### 个人客户

个人客户姓名应使用中文姓名风格。

规则：

- 姓氏使用常见中文姓氏。
- 名字使用 1 到 2 个常见汉字。
- `account_name` 对个人账户直接使用姓名，不加“账户”后缀。

示例：

| 字段 | 合理值 | 不合理值 |
|---|---|---|
| `customer_name` | `文平贵` | `模拟客户0001` |
| `account_name` | `文平贵` | `文平贵账户` |

### 产品和套餐名称

产品、套餐、资费名称必须贴近真实电信业务。

合理示例：

- `5G畅享129元套餐`
- `5G畅享199元套餐`
- `融合宽带199元套餐`
- `校园流量39元套餐`
- `政企专线基础套餐`

禁止示例：

- `模拟primary_price_plan_name`
- `主资费0001`
- `套餐A`

## 产品层级规范

产品层级必须体现父子关系，不能不同层级使用相同值。

| 层级 | 示例 |
|---|---|
| 产品大类 | `MOBILE` |
| 二级产品 | `MOB5G` |
| 产品细类 | `MOB5G129` |

校验规则：

- 主产品细类和二级产品细类不得相等。
- 产品细类必须能回溯到二级产品。
- 5G 套餐用户的流量使用量和 ARPU 应整体高于低价或 4G 套餐用户。
- 融合套餐用户可同时体现移动、宽带、固话或家庭成员关系。

## 组织、渠道和员工规范

组织层级必须有真实层级含义。

推荐层级：

| 层级 | 含义 |
|---|---|
| 一级 | 省公司 |
| 二级 | 市公司 |
| 三级 | 区县分公司 |
| 四级 | 镇街网格或营业网点 |
| 员工 | 发展人、客户经理、投诉处理人 |

规则：

- 不同组织层级字段不能全部使用同一个值。
- 员工必须归属于某个组织或网点。
- 发展渠道、发展员工、发展部门之间必须能对应。
- 组织名称不能使用 `区县分公司0001` 这种机械模板。

## 地理层级规范

省、市、区县、镇街、网点必须从同一条行政层级链生成。

规则：

- 广东省下面不能出现广西南宁市。
- 市必须归属于省。
- 区县必须归属于市。
- 镇街和网点名称应基于区县生成。
- 地址、组织、渠道、网点字段应共享同一地理上下文。

示例：

| 省 | 市 | 区县 | 镇街 | 网点 |
|---|---|---|---|---|
| 广东省 | 广州市 | 天河区 | 天河区建设路街道 | 天河区建设路街道社区营业厅 |
| 江苏省 | 南京市 | 江宁区 | 江宁区中心镇 | 江宁区中心镇综合网格站 |

## 生命周期时间规范

### 基本规则

用户生命周期必须满足：

```text
activation_time < termination_time
```

所有业务时间必须满足：

```text
activation_time <= business_time <= termination_time
```

如果用户仍在网，生成和校验时可使用一个统一的远期逻辑离网时间参与判断，但业务字段中可保留为空或在网状态。

### 业务时间字段范围

以下字段都属于业务时间或业务日期，需要落在用户生命周期内：

- `billing_cycle_date`
- `billing_month_date`
- `business_time`
- `create_time`
- `payment_time`
- `recharge_time`
- `complaint_time`
- `activation_time`
- `termination_time`
- `data_date`
- `data_month_date`

### 订单时间规则

订单相关时间必须满足：

```text
activation_time <= create_time <= business_time <= payment_time <= termination_time
```

如果订单未支付，`payment_time` 可以为空，但不能早于创建时间。

## 账务与支付规范

账务数据必须与账户、套餐、用户状态关联。

规则：

- `billing_cycle_date` 不得早于 `activation_time`。
- 月租费用应来自套餐档位。
- 欠费用户更容易出现停机、投诉、流失风险。
- 缴费金额应能覆盖部分账单或欠费，不应完全随机。
- 税率、折扣率、欠费率、成功率等比例字段不应大量为 `0.000000`。

合理分布：

| 字段类型 | 推荐范围 |
|---|---|
| 税率 | 常见为 `0.060000`，可少量浮动 |
| 折扣率 | `0.500000` 到 `1.000000` |
| 欠费率 | `0.000000` 到 `0.300000`，高风险用户可更高 |
| 转化率 | `0.010000` 到 `0.300000` |

## 使用行为规范

用户使用行为必须和套餐、城市、年龄、网络类型相关。

规则：

- 5G 用户流量使用量整体高于 4G 用户。
- 高 ARPU 用户流量、语音、增值业务使用量更高。
- 校园用户流量高、语音低、套餐价格偏低。
- 政企用户账单金额较高，投诉和专线类业务更多。
- 欠费或停机用户近期使用量可能下降。

## 营销和投诉规范

### 营销触达

营销触达后只有部分用户转化。

规则：

- 高价值用户更容易收到高档套餐升级触达。
- 低流量套餐但高流量使用用户适合被推荐升档。
- 欠费、停机、投诉中的用户不应大量生成成功营销转化。

### 投诉事件

投诉应和用户体验、账务、网络、服务行为相关。

高投诉概率人群：

- 欠费争议用户
- 账单异常用户
- 使用量突然下降用户
- 网络流量高但体验差用户
- 近期订单失败或支付异常用户

## 分层数据规模规范

### ODS 到 DWD

ODS 是原始接入层，DWD 是清洗后的明细层。

规则：

- DWD 不得无来源扩增 ODS 明细。
- DWD 用户维度行数不得超过 ODS 用户来源快照对应批次行数。
- DWD 事实表应能追溯到 ODS 来源数据或模拟业务事件。

### DWS 和 ADS

DWS 是汇总宽表，ADS 是指标应用层。

规则：

- DWS 行数通常少于或等于对应 DWD 明细组合粒度。
- ADS 行数通常少于 DWS。
- ADS 指标必须有明确口径、来源表和推荐查询方式。

## 禁止值规范

生成器必须检查并禁止以下值或模式：

| 模式 | 说明 |
|---|---|
| `CODE_` | 占位编码 |
| `模拟` | 占位中文 |
| `S000001` | 占位员工 |
| `D00001` | 占位部门 |
| `A000000000001` | 占位账户 |
| `区县分公司0001` | 机械组织名称 |
| `0` 作为主键 | 无效主键 |
| 多个层级字段完全相同 | 层级生成失败 |

## 数据质量校验规范

每次生成、入库、重灌后必须执行质量校验。

### 表结构校验

| 校验项 | 要求 |
|---|---|
| `data_time` | 所有表必须存在 |
| `data_time` 类型 | 必须为 `DATE` |
| `data_time` 位置 | 必须为每张表最后一个字段 |
| 主键 | 所有表必须有主键 |
| 中文注释 | 关键字段必须有中文注释 |

### 主外键校验

| 校验项 | 要求 |
|---|---|
| 用户 ID | 不为空、不为 0、格式合法 |
| 客户 ID | 可关联客户维度 |
| 账户 ID | 可关联账户维度 |
| 产品 ID | 可关联产品维度 |
| 渠道 ID | 可关联渠道维度 |
| 员工 ID | 可关联员工维度 |

### 时间校验

| 校验项 | 要求 |
|---|---|
| 入网离网 | `activation_time < termination_time` |
| 账期 | 不早于入网时间 |
| 业务时间 | 位于生命周期内 |
| 支付时间 | 不早于订单创建时间 |
| 批量日期 | 等于指定批次日期 |

### 业务逻辑校验

| 校验项 | 要求 |
|---|---|
| 产品层级 | 细类和二级类不得相等 |
| 地址层级 | 省、市、区县不得错配 |
| ODS/DWD 行数 | DWD 不得超过对应 ODS 来源规模 |
| 欠费逻辑 | 欠费用户风险、停机、投诉概率更高 |
| 5G 逻辑 | 5G 用户流量和 ARPU 更高 |
| 营销逻辑 | 触达后只有部分用户转化 |

## 生成器实现要求

Python 生成器应按以下顺序执行：

1. 初始化批次日期和随机种子。
2. 加载跨批次用户身份缓存（`user_identity_cache.json`），首次运行跳过。
3. 生成地理、组织、渠道、员工基础维度。
4. 生成客户、用户、账户主数据（已有用户复用缓存，新用户写入缓存）。
5. 对已有活跃用户应用跨批次离网逻辑。
6. 生成产品、套餐和订购关系。
7. 生成账单、缴费、使用、订单、投诉、营销等事实数据。
8. 生成 DWD 维度和事实表。
9. 生成 DWS 汇总表。
10. 生成 ADS 指标表和 Agent 语义目录。
11. 写入 MySQL。
12. 保存跨批次用户身份缓存。
13. 执行全量质量校验。

运行方式：

```bash
# 单批次：指定日期
BATCH_DATE=2026-05-31 python src/telecom_dw/data_generator/generate_telecom_dw_data.py

# 全批次：从零开始按顺序执行所有批次（默认行为）
python src/telecom_dw/data_generator/generate_telecom_dw_data.py

# 全量重跑：删除缓存后重新生成
rm data/generated/user_identity_cache.json
python src/telecom_dw/data_generator/generate_telecom_dw_data.py
```

## 后续扩展规范

新增字段、表或指标时必须先补充以下内容：

- 字段中文含义
- 字段英文标准名
- 字段数据类型
- 是否维度字段
- 是否指标字段
- 是否主键或外键
- 是否枚举字段
- 生成规则
- 校验规则

新增模拟业务场景时必须说明：

- 涉及主题域
- 涉及 ODS 来源表
- DWD 明细承接方式
- DWS 汇总粒度
- ADS 指标或应用场景
- 与 Agent 查询的关系
