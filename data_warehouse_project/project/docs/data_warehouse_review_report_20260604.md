## 二、数据模型审查

### 2.1 ✅ 分层架构

四层 schema 设计（ods/dwd/dws/ads）清晰，符合企业数仓标准。各层职责明确，数据流向合理。

### 2.2 ✅ DWD 维度-事实表设计

10 张维度表 + 7 张事实表构成完整星型模型。主外键关联覆盖率达 100%：

- `dim_user → fact_usage_daily`: 100%
- `dim_account → fact_billing_monthly`: 100%
- 所有事实表均可追溯到维度表

### 2.3 ✅ 表中文注释

所有关键字段均包含中文注释（`COMMENT`），符合 Agent SQL 生成需求。

### 2.4 ⚠️ ODS 主键策略

ODS 层 31 张表使用 `synthetic_row_id`（自增 bigint）作为主键，而非业务主键。这在模拟场景可接受，但如果后续接入真实数据，建议改为业务自然键+批次日期的复合主键。

### 2.5 ⚠️ ADS Agent 目录表字段不足

| 表 | 问题 |
|---|---|
| `ads_agent_field_catalog` | 仅有 6 个字段（field_id, field_type, table_name, domain_name, is_metric, data_time），缺少 `field_name`、`field_description`、`business_meaning` 等语义说明字段。Agent 无法仅通过此表完成字段语义理解 |
| `ads_agent_metric_catalog` | `metric_name` 显示为乱码/占位值，难以用于实际 Agent 指标匹配 |

### 2.6 ⚠️ DWS 汇总粒度与命名不一致

| 表名 | 预期粒度 | 实际粒度 | 偏差 |
|---|---|---|---|
| `dws_user_day_summary` | 每用户每天一行 | 日级全用户聚合，仅 365 行 | 粒度粗化约 20,300x |
| `dws_user_month_summary` | 每用户每月一行 | 使用日级 `data_month_date`（如 2025-06-01），每月第一天集中全量用户 | 命名暗示月粒度但使用日期格式 |
| `dws_product_month_summary` | 每产品每月 | 仅 282 行（80 产品 × 多个月份），符合预期 | ✅ |

### 2.7 ✅ 表结构规范

- 所有 65 张表均包含 `data_time` 字段
- `data_time` 位置：**全部位于最后字段** ✅
- `data_time` 类型：**全部为 DATE** ✅

## 四、跨批次延续性审查

### 4.1 ✅ 用户总量趋势

| 批次 | 用户数 | 环比增长 |
|---|---|---|
| 2025-05-30 | 16,000 | — |
| 2025-05-31 | 16,200 | +200 |
| 2026-04-30 | 19,000 | +2,800 |
| 2026-05-30 | 20,000 | +1,000 |
| 2026-05-31 | 20,300 | +300 |

✅ 严格递增，符合规范。

### 4.2 ✅ 核心 ID 稳定性

| ID 字段 | 跨批次稳定性 |
|---|---|
| `customer_id` | ✅ 完全稳定（0 变化） |
| `account_id` | ✅ 完全稳定（0 变化） |

### 4.3 ❌ 严重：跨批次身份字段大规模变化

| 字段 | 变化用户数 | 占比 | 严重程度 |
|---|---|---|---|
| `customer_name` | 18,994 / 20,300 | **93.5%** | 🔴 严重 |
| `service_number_id`（手机号） | 18,990 / 20,300 | **93.5%** | 🔴 严重 |
| `activation_date`（入网日期） | 18,997 / 20,300 | **93.6%** | 🔴 严重 |

**问题分析**：数据生成器在 2025 年批次（batch 1-2）和 2026 年批次（batch 3-5）之间使用了不同的随机名称/手机号/入网日期。这违反了规范中"同一用户跨批次应保持基础身份稳定，例如姓名、手机号、客户 ID、账户 ID 不应频繁变化"的要求。

**影响**：Agent 基于用户 ID 查询时会得到矛盾的身份信息，影响用户画像分析的准确性。

**根因推测**：`generate_telecom_dw_data.py` 中每个批次独立生成用户属性，而非在首次生成后固化。需要修改生成器，使 batch 2-5 复用 batch 1 的基础身份数据。

### 4.4 ❌ 离网用户数为零

| 批次过渡 | 新增用户 | 离网用户 |
|---|---|---|
| 1 → 2 | +200 | **0** |
| 2 → 3 | +2,800 | **0** |
| 3 → 4 | +1,000 | **0** |
| 4 → 5 | +300 | **0** |

规范明确要求"允许用户离网，但新增用户数应大于离网用户数"。零离网意味着所有用户在 5 个批次期间 100% 留存，这完全不符合电信行业的真实情况（年离网率通常 10-40%）。

## 五、ADS/DWS 指标层排查

### 5.1 ⚠️ ADS 收入指标金额异常

`ads.ads_kpi_revenue_monthly` 中月份级别的收入在 34-175 元范围，这对于汇总指标而言过低。若 20,300 用户月均 ARPU 50 元，月度总收入应在 100 万元量级。

### 5.2 ⚠️ DWS 汇总表数据稀疏

`dws.dws_user_day_summary` 为每用户每天仅约 0.018 行（365 / 20,300），意味着大量用户-日期组合缺失数据。如果设计为轮询采样方式，应在表注释中说明。

## 六、修复优先级建议

| 优先级 | 问题 | 建议修复方式 |
|---|---|---|
| 🔴 P0 | 跨批次身份字段（name/phone/activation）大规模变化 | 修改生成器：batch 1 固化后，batch 2-5 复用同一用户的 `customer_name`、`service_number_id`、`activation_date` |
| 🔴 P0 | 离网用户数为零 | 在生成器中引入离网逻辑：每个批次过渡中约 1-3% 的用户离网 |
| 🟡 P1 | DWS 汇总粒度与命名不一致 | 将 `dws_user_day_summary` 改为真实每用户每日粒度，或重命名/补充注释说明当前为日聚合表 |
| 🟡 P1 | ADS 收入指标金额异常低 | 检查 ADS 生成逻辑：KPI 指标应为全局汇总（SUM），而非单行抽样值 |
| 🟢 P2 | ADS Agent 目录表语义字段不足 | 补充 `field_name`、`field_description`、`business_meaning` 等字段 |
| 🟢 P2 | ODS 主键为 synthetic_row_id | 非紧急，后续真实数据接入时改为业务键复合主键 |

## 七、总体评价

| 维度 | 评分 | 说明 |
|---|---|---|
| 架构设计 | ⭐⭐⭐⭐⭐ | 分层清晰，星型模型完整，文档齐全 |
| 单批次数据质量 | ⭐⭐⭐⭐⭐ | 主外键、时间逻辑、业务逻辑校验全部通过 |
| 跨批次延续性 | ⭐⭐ | 核心身份字段在批次间大规模漂移，离网逻辑缺失 |
| Agent 就绪度 | ⭐⭐⭐ | 语义目录表字段不足，需补充字段后可支撑 Agent 查询 |

**结论**：项目数据模型设计优秀，单批次数据质量控制良好。但跨批次数据延续性是短板——这表明数据生成器在设计时更多考虑了单批次验证，对多批次快照场景下的身份稳定性考虑不足。修复 P0 问题后，数据仓库可作为 Agent 训练的可靠基础。
