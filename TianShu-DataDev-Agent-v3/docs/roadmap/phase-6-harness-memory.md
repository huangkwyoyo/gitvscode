# Phase 6：Harness 评测集成与工程记忆

## 目标

构建 Prompt Harness 评测框架，建立结构化的 Engineering Memory 系统，实现系统行为的可观测性和可追溯性。Harness 评测与运行时主链路完全隔离，不干扰核心业务流程。

## 输入

- **Phase 1-4 的核心模块**：SQL 编译器、Spark 多智能体、CrossValidator、修复循环
- **评测用例集**：由团队定义的典型需求场景、边界场景、异常场景

## 输出

| 产出 | 说明 |
|------|------|
| **Prompt Harness 评测框架** | 可重复运行的评测系统，支持用例定义、自动执行、结果收集 |
| **8 维度评测报告** | 每次运行的标准化报告，包含 8 项评测维度的评分和明细 |
| **Engineering Memory 系统** | 结构化的工程知识记忆，支持查询、更新和版本管理 |
| **成本统计模块** | LLM Token 消耗、API 调用次数、响应延迟的统计与预警 |

## 模块职责

### Harness 评测框架

- **harness_runner/**：加载评测用例集，逐一执行，收集结果
  - 评测用例格式：YAML 定义（输入 RequirementIR + 预期输出约束 + 验证规则）
  - 执行模式：批量模式 / 单用例模式 / 回归模式
- **harness_reporter/**：生成标准化评测报告，包含 8 项评测维度

### 8 项 Harness 评测维度

| # | 维度 | 描述 | 度量方式 |
|---|------|------|----------|
| 1 | **语法正确性** | 生成的 SQL/Spark 代码语法是否正确 | 编译通过率 |
| 2 | **语义等价性** | SQL 和 Spark 输出结果是否语义等价 | 9 维度交叉验证通过率 |
| 3 | **安全性** | 代码是否通过 Static Validator 的安全检查 | 安全违规数/代码行数 |
| 4 | **性能效率** | 执行耗时是否在可接受范围内 | P50/P95 执行耗时 |
| 5 | **修复通过率** | 修复循环在 2 轮内的通过率 | 修复成功数/总修复触发数 |
| 6 | **回退频率** | 降级到 SQL 路径或 HUMAN_REVIEW 的频率 | 回退次数/总任务数 |
| 7 | **LLM Token 消耗** | 每次任务消耗的 Token 总量和成本 | Token 计数 + 预估成本 |
| 8 | **端到端成功率** | 从需求输入到结果输出的全链路成功率 | 成功交付数/总任务数 |

### Engineering Memory 系统

- **memory_store/**：结构化记忆的存储层
  - 存储介质：SQLite（开发）/ PostgreSQL（生产）
  - 记忆类型：
    - **需求模式记忆**：历史项目书中的常见需求模式和对应的成功实现策略
    - **修复经验记忆**：历史修复循环中有效的修复策略和无效的尝试
    - **失败案例记忆**：历史上的失败案例、根因分析和解决方案
    - **配置记忆**：各模块的最优配置参数和历史调优记录
- **memory_query/**：记忆查询接口，供 Phase 1-4 在决策时调用
  - 查询模式：相似度检索（基于 embedding） + 结构化查询（基于标签）
- **memory_version/**：记忆版本管理，记录每次记忆更新的时间戳和变更内容

### 成本统计模块

- **token_counter/**：统计 LLM API 调用的 Token 消耗
  - 按角色统计（Developer、Reviewer、Tester）
  - 按阶段统计（编译、生成、验证、修复）
  - 按任务统计（单次项目书的完整调用链）
- **cost_estimator/**：根据 Token 消耗和模型单价估算成本
- **latency_monitor/**：记录每次 API 调用的响应延迟，触发预警（超过阈值）

### 隔离策略

- **代码隔离**：Harness 评测框架放在独立的 `harness/` 包中，不导入运行时主链路
- **数据隔离**：Engineering Memory 使用独立的数据存储，不与运行时数据混淆
- **进程隔离**（可选）：Harness 评测可在独立进程中运行，不影响主服务
- **依赖隔离**：Harness 评测框架的第三方依赖（pytest、pandas 等）与运行时依赖分开管理

## 明确不做什么

- 不修改运行时主链路的业务逻辑
- 不依赖前端展示（评测报告为纯文档格式）
- 不涉及真实 LLM API（Token 消耗统计使用 Mock 数据开发）
- 不替代传统的单元测试和集成测试

## 风险

| 风险 | 缓解措施 |
|------|----------|
| Engineering Memory 数据膨胀 | 设置自动归档策略（90 天前的记忆归档到冷存储） |
| 评测结果波动大 | 固定随机种子 + 确定性执行模式，减少不确定性 |
| 记忆检索不准确 | embedding 模型选择 + 多路召回（相似度 + 标签 + 规则） |
| 成本统计与 LLM API 实际计费不一致 | 使用 API 返回的 `usage` 字段而非估算值 |

## 验收标准

1. [ ] Harness Runner 能加载 YAML 格式的评测用例并批量执行
2. [ ] 8 项评测维度全部实现，报告格式标准化
3. [ ] Engineering Memory 支持 4 种记忆类型的存储和查询
4. [ ] 记忆查询支持相似度检索和标签检索
5. [ ] Token 消耗统计按角色和阶段分别汇总
6. [ ] 成本估算模块能根据 Token 消耗和配置的单价计算预估成本
7. [ ] Harness 包与运行时主链路无直接导入依赖
8. [ ] 评测用例集包含至少 10 个典型场景

## 测试边界

- **测试范围**：Harness Runner 执行流程、Memory 存储/查询/版本管理、Token 统计、成本估算
- **不测试**：运行时核心业务逻辑、前端展示、真实 LLM API
- **隔离要求**：使用独立的测试数据库测试 Memory 存储，独立的 Mock 模块测试 Token 统计
- **异常测试**：评测用例格式错误、Memory 查询无结果、Token 统计溢出

## 与其他阶段的依赖

- **依赖 Phase 1-4**：Harness 评测框架依赖各阶段的稳定模块
- **被 Phase 7 依赖**：v1.0 验收需要 Harness 评测报告作为依据
