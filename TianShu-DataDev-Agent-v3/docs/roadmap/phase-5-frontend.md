# Phase 5：前端项目书工作台

## 目标

构建面向用户的前端交互界面，提供项目书创建、任务提交、结果查看和修复管理的完整工作台体验。用户可通过 Web 界面输入需求、查看 SQL/Spark 编译结果、监控验证状态和处理 HUMAN_REVIEW 请求。

## 输入

- **项目书需求**：用户在前端输入的结构化或半结构化数据分析需求
- **后端 API**：由 FastAPI 提供的数据接口，对接 Phase 1-4 的核心能力

## 输出

| 产出 | 说明 |
|------|------|
| **FastAPI 后端服务** | 提供 RESTful API，管理项目书 CRUD、任务提交、状态查询、结果展示 |
| **Web 前端界面** | 项目书工作台，包含 11 项功能需求的交互实现 |
| **API 规范文档** | OpenAPI 3.0 规范的接口文档 |

## 模块职责

### 11 项功能需求及实现计划

| # | 功能需求 | 实现计划 |
|---|----------|----------|
| 1 | **项目书创建/编辑** | 富文本编辑器 + 结构化表单（表名选择、过滤条件输入、聚合配置） |
| 2 | **需求解析预览** | 提交项目书后展示 LLM 解析出的 RequirementIR，用户确认或修正 |
| 3 | **SubIntent 列表** | 展示从 RequirementIR 提取的 SubIntent 列表，支持展开/折叠查看详情 |
| 4 | **SQL 代码展示** | 语法高亮的 SQL 代码展示面板，附带 ExecutionTrace 摘要 |
| 5 | **Spark 代码展示** | 语法高亮的 PySpark 代码展示面板，附带 SparkDeveloper/Reviewer/Tester 的输出 |
| 6 | **验证结果面板** | 9 维度比较结果的可视化仪表盘，PASS/FAIL 状态用颜色标识，差异行可展开 |
| 7 | **修复循环状态** | 展示当前修复轮次（1/2、2/2）、当前节点（分析中/修复中/验证中）、预计剩余时间 |
| 8 | **HUMAN_REVIEW 队列** | 待人工审查请求的列表，展示项目书摘要、差异根因、修复历史，支持审批/驳回操作 |
| 9 | **历史记录** | 按时间线展示历史项目书及其对应的 SQL/Spark 版本、验证结果、修复记录 |
| 10 | **快照管理** | 展示可用快照列表、版本、抽样时间、来源表、SHA-256，支持快照选择 |
| 11 | **系统状态面板** | 展示各模块健康状态、API 延迟、最近错误日志、组件版本信息 |

### 后端模块

- **api/routes/**：FastAPI 路由，按功能域划分（project_books、tasks、validations、repairs、snapshots、system）
- **api/schemas/**：Pydantic 请求/响应模型，对接 `contracts/` 中的数据结构
- **api/services/**：业务逻辑层，调用 Phase 1-4 的核心模块

### 前端模块

- **pages/**：页面级组件（项目书页、结果页、验证页、修复页、历史页、系统页）
- **components/**：可复用组件（代码展示器、状态指示器、差异比较表、时间线、面板容器）
- **stores/**：状态管理（项目书状态、任务状态、用户偏好）

## 技术选择

| 层次 | 技术选型 | 说明 |
|------|----------|------|
| 后端框架 | FastAPI | 异步支持好、自动生成 OpenAPI 文档、类型校验内置 |
| ORM/存储 | SQLite（开发）/ PostgreSQL（生产） | SQLAlchemy 异步 ORM |
| 前端框架 | 待定（React / Vue 3 / 其他） | 根据团队前端技术栈确定 |
| 代码展示 | react-syntax-highlighter / highlight.js | 支持 SQL 和 Python 语法高亮 |
| API 文档 | Swagger UI（FastAPI 内置） | 自动生成，方便前后端联调 |
| 状态管理 | 待定（Redux Toolkit / Pinia / 其他） | 与前端框架配套 |

## 交互流程

1. 用户打开工作台，进入"新建项目书"页
2. 通过结构化表单或粘贴文本输入需求
3. 提交后，后端解析需求生成 RequirementIR，返回预览
4. 用户确认后，后端按流水线（Phase 1 -> Phase 2 -> Phase 3）执行
5. 执行过程中，前端轮询状态并实时更新进度
6. 执行完成后，展示 SQL 和 Spark 代码（可切换/对比视图）
7. 验证结果面板显示 9 维度的 PASS/FAIL 状态
8. 若触发修复循环，显示当前修复轮次和进度
9. 若进入 HUMAN_REVIEW，通知用户处理
10. 用户可在历史页查看所有过往项目书和执行记录

## 明确不做什么

- 不实现 LLM 逻辑（仅调用后端 API）
- 不实现数据源管理（快照由 Phase 3 管理）
- 不实现用户认证和权限管理（v1.0 之后考虑）
- 不替代 Phase 1-4 的核心能力

## 风险

| 风险 | 缓解措施 |
|------|----------|
| 前端技术栈选择延迟 | 后端 API 先行，前端可用 Swagger UI 临时测试 |
| 实时状态更新的性能问题 | 使用 WebSocket 轮询替代短轮询，减少请求频率 |
| 代码展示大文件性能 | 分片加载 + 虚拟滚动 |
| 前后端 Schema 不一致 | 自动生成 TypeScript 类型定义（FastAPI -> openapi-typescript） |

## 验收标准

1. [ ] FastAPI 后端启动正常，所有接口返回正确状态码
2. [ ] 11 项功能需求中至少 8 项可用
3. [ ] 项目书创建 -> 提交 -> 结果查看的完整链路可用
4. [ ] 验证结果面板能正确展示 9 维度的 PASS/FAIL 状态
5. [ ] 修复循环状态展示正确（轮次、当前节点）
6. [ ] HUMAN_REVIEW 队列支持查看详情和审批/驳回
7. [ ] 历史记录页面支持时间线展示

## 测试边界

- **测试范围**：API 接口测试、前端组件测试、端到端流程测试
- **不测试**：后端核心业务逻辑（由 Phase 1-4 测试覆盖）、LLM 调用、数据源连接
- **隔离要求**：使用 Mock 后端数据，不依赖真实 Phase 1-4 模块
- **异常测试**：API 超时、后端服务不可用、无效输入、大文件加载

## 与其他阶段的依赖

- **依赖 Phase 1-4**：依赖各阶段产出的数据和 API 接口
- **被 Phase 7 依赖**：v1.0 验收需要前端作为用户交互入口
