## 1. 核心思想：角色转变

### 1.1 旧模式 vs 新模式

传统数据开发工具把 Agent 当作**生产执行者**——自动跑管道、直接产出数据、人只需"验收结果"。这种模式下，人是**旁观者**——管道在跑，人在等结果，出了问题才介入。

Data Dev Agent v2.0 把 Agent 当作**代码生成者 + 自测者**——它写代码、跑测试、出报告，但**不替人做上线决定**。人从"旁观者"变成**最终决策者**——审查代码、确认测试结果、拍板上不上线。

```
旧模式（v1.x）：
  人提需求 → Agent 自动执行管道 → 直接产出数据
  人 = 旁观者（被动接收结果）

新模式（v2.0）：
  人提需求 → Agent 写 SQL + Spark DSL 两份代码 → 自动验证 + 交叉验证
           → Agent 出测试报告 → 人审查 → 人拍板上线
  人 = 决策者（主动把控质量）
```

### 1.2 为什么这样设计

数据开发的特点是：**最终上不上线，始终是人做决定。** 从来没有一个数据团队会让 Agent 自动把代码部署到生产——在任何一个严肃的数据团队里，生产代码必须经过 code review。

既然如此，安全模型就不该基于"LLM 不可靠所以要隔离它"——而该基于"LLM 写的代码要经过人审查才能上线"。前者在防止风险的同时牺牲了 LLM 的价值，后者在释放 LLM 价值的同时用人审管控风险。

**核心权衡**：松开 LLM 的手（让它写代码），加四道保险（来源追溯 + 自动验证 + 交叉验证 + 人审闸门）。

### 1.3 角色对比

| 维度 | 旧架构（v1.x）Agent 角色 | 新架构（v2.0）Agent 角色 |
|------|------------------------|------------------------|
| **做什么** | 自动执行 8 层管道 | 分析需求 → 生成双份代码（SQL + Spark DSL）→ 自动验证 + 交叉验证 → 输出审查材料 |
| **代码来源** | 模板编译器确定性拼接 | LLM 生成草案，来源可追溯 |
| **决策权** | 管道自动决策（G3/G2、JOIN 路径） | LLM 提议方案，人确认 |
| **产出物** | 数据文件（Parquet/CSV） | SQL + Spark DSL + 测试结果 + 交叉验证报告 + 决策说明 |
| **上线** | 管道自动产出 = 上线 | 人审查通过 ≠ 上线，人还需执行部署 |
| **人的角色** | 旁观者（提需求、收结果） | 决策者（审查代码、拍板上线） |
| **验证机制** | 规则引擎单项校验 | 规则引擎 + SQL/Spark 结果交叉验证 |

## 2. 工作流

### 2.1 总览：Agent 做什么，人做什么

```
═══════════════════ Agent 的工作区 ═══════════════════

  需求输入（YAML 或自然语言）
    ↓
  [阶段 1] 需求分析   —— Agent 解析需求、匹配指标、确认数据可用性
    ↓
  [阶段 2] 方案设计   —— Agent 选表、选层级、设计 JOIN 路径
    ↓
  [阶段 3] 代码生成   —— Agent 生成两份代码：
                        ├─ SQL（DuckDB）实现
                        └─ Spark DSL 实现（同一逻辑，两种表达）
    ↓
  ═══════════ Agent 的确定性验证 ═══════════
    ↓
  [阶段 4] 自动验证   —— 规则引擎 7 项检查 + SQL/Spark 结果交叉验证
    ↓
  [阶段 5] 材料输出   —— 整合：双份代码 + 测试结果 + 交叉验证报告 + 决策说明 + 不确定项

═══════════════════ 人的决策区 ═══════════════════

  [阶段 6] 人审上线   —— 人审查全部材料 → 批准 / 修改后重审 / 拒绝
    ↓
  人通过 PR / CI/CD 部署到生产
```

**关键分界**：阶段 1-5 是 Agent 的工作——写代码、跑测试、出报告。阶段 6 是人的工作——审查、判断、拍板。Agent 不越过这条线。

### 2.2 各阶段详解

**阶段 1：需求分析**（Agent 主导）
- 解析输入（YAML 或自然语言），提取指标、维度、过滤条件
- 查已注册指标列表，精确匹配优先，模糊匹配由 LLM 消歧
- 确认所需数据在数据库设计文档中是否存在
- 输出：结构化需求分析报告（含匹配置信度、不确定项）

**阶段 2：方案设计**（Agent 主导，查确定性数据源）
- 查 ColumnBindingTable，确定每个指标的 G3/G2 可用性
- 决策数据源层级（G3 优先，G2 降级），附理由
- 设计 JOIN 路径（查白名单），确定增量策略
- 输出：设计方案（选表 + 选层级 + 选策略 + 理由）

**阶段 3：代码生成**（Agent 主导，双层边界——代码生成端）
- **生成 SQL 代码草案**（DuckDB 方言）：SELECT/JOIN/WHERE/GROUP BY/窗口函数/CTE
- **生成 Spark DSL 代码草案**（PySpark）：同一业务逻辑的独立实现
- 生成 Python 管道脚本（如需要多步 DAG）
- 生成 pytest 测试用例和调度配置
- **两份代码独立生成，逻辑等价但实现不同**——这是交叉验证的前提
- **所有表/字段引用标注来源**（格式见 §7.2）
- **生成的所有代码一律视为"不可信补丁"**，不得直接落地执行

**阶段 4：自动验证**（确定性规则引擎，不调 LLM，双层边界——数据执行端）
- 7 项规则检查 + 1 项交叉验证：
  - #1-6：表存在性、安全黑名单、表权限、JOIN 白名单、样本执行、结果质量
  - #7：**SQL vs Spark DSL 交叉验证**——两份代码分别对 dev 库执行，比较结果集
- 检查结果分三级：FAIL（阻止进入审查）/ WARN（标注警告）/ 通过
- 交叉验证不通过 → WARN，人审时必须调查差异原因

**阶段 5：材料输出**（确定性整合）
- 整合双份代码、测试结果、交叉验证报告、设计决策、不确定项清单
- 输出供人审查的完整材料包

**阶段 6：人审上线**（人主导）
- 人审查两份代码和测试结果
- 人确认交叉验证差异（如有）是否可接受
- 人确认所有"不确定"标注
- 人决定：批准上线 / 修改后重审 / 拒绝
- 批准后由人通过 PR 合入 → CI/CD 或手工部署

### 2.3 数据源层级决策

```
指标有 G3 汇总表？
  ├─ YES → 优先使用 G3（Agent 提议，附理由）
  └─ NO  → 指标有 G2 表达式？
            ├─ YES → 使用 G2 fact 表（Agent 提议，标注日期过滤契约）
            └─ NO  → Agent 告知用户该指标缺少数据来源
```

## 3. Agent 的权限边界：双层边界

v2.0 的核心安全设计是**双层边界**——区分"代码生成"和"数据执行"两个不同的风险层面。两层之间是防线 2（自动验证）——代码进入执行前必须穿过的唯一通道。

```
══════════════════════════════════════════════════════════
边界 1：代码生成（Code Generation Boundary）
  规则：LLM 可以生成代码，但产物 = "不可信补丁"
  验证：静态检查 → 单元测试 → 人审
  越过后果：❌ 未经验证的代码可能含幻觉引用
══════════════════════════════════════════════════════════
          ↓ 防线 2 自动验证（唯一通道）↓
══════════════════════════════════════════════════════════
边界 2：数据执行（Data Execution Boundary）
  规则：无论代码谁写的，都必须走 Validator → Executor
  硬禁止：绕过校验直接查库、直接执行 SQL、访问生产库
  越过后果：❌ 被防线 2 拦截（硬规则，不可绕过）
══════════════════════════════════════════════════════════
```

### 3.1 边界 1：代码生成边界

LLM 可以生成以下代码草案，但**所有产物一律视为"不可信补丁"**——在通过防线 2 验证之前，不得落地执行。

#### 3.1.1 可以生成什么

| 能力 | 说明 | 进入的验证链路 |
|------|------|--------------|
| 生成 SQL 代码草案 | SELECT、JOIN、WHERE、GROUP BY、窗口函数、CTE 等（DuckDB 方言） | Validator → 样本执行 → 交叉验证 |
| 生成 Spark DSL 代码草案 | PySpark 实现同一业务逻辑 | Schema 校验 → 交叉验证 |
| 生成 Python 管道脚本 | 多步 DAG 的完整实现 | 单元测试 → 静态检查 |
| 生成 YAML contracts | 数据契约定义 | Schema 校验 |
| 生成测试用例 | pytest 格式，覆盖正常和边界情况 | 必须实际跑通 |
| 生成文档/配置 | README、调度配置、注释 | 静态检查 |
| 推荐表名、字段名、JOIN 条件 | 根据需求选择最合适的表和关联路径 | 表存在性检查 → 来源追溯 |
| 分析需求、消歧指标名 | 将用户的自然语言描述映射到已注册指标 | 不确定时标注，进人审 |

#### 3.1.2 生成代码的验证链路

```
LLM 生成代码草案
  ↓
静态检查（安全黑名单、表/字段存在性、JOIN 白名单）
  ↓
单元测试（pytest，必须全部通过）
  ↓
样本执行（dev 库 LIMIT 1000，单份代码执行）
  ↓
交叉验证（SQL vs Spark DSL 结果对比）  ← 只有两份代码都通过才进入人审
  ↓
防线 3：人审闸门
```

**关键**：LLM 生成的 SQL 不是"可执行 SQL"，是"待验证的 SQL 草案"。它和模板编译器的产出一样，都必须通过防线 2 才能被执行。**约束在数据执行端，不在代码生成端。**

### 3.2 边界 2：数据执行边界

这是 Agent 的硬边界——不是"建议不要做"，是"技术上做不到"或"会被防线 2 拦截"。

#### 3.2.1 绝对不能做的事

| 禁止 | 原因 | 拦截机制 |
|------|------|---------|
| 绕过 Validator 直接执行 SQL | 未验证代码可能含幻觉引用或危险操作 | 防线 2——执行必须过 Validator |
| 绕过 Executor 直接查 DuckDB | 所有数据库访问必须走统一入口 | 架构约束——无其他 DB 连接路径 |
| 编造不存在的表或字段 | 数据完整性不可妥协 | 防线 2 检查项 #1：表/字段存在性 |
| 直接操作生产库 | Agent 没有生产凭据 | 环境隔离——Agent 只能连开发库 |
| 生成非只读 SQL（INSERT/UPDATE/DELETE/DROP 等） | 数据安全底线 | 防线 2 检查项 #2：安全黑名单 16 关键字 |
| 绕过人的审查直接上线 | 人是最終决策者 | Agent 没有部署权限 |
| 修改 `meta.metric_definitions` 或 schema | 元数据变更必须走审批 | 数据库写权限限制 |
| 访问 `bronze.*` / `silver.*` 表 | 原始和中间数据不可直接暴露 | 防线 2 检查项 #3：表访问权限 |

#### 3.2.2 唯一执行路径

```
SQL 草案（LLM 生成 或 模板编译器生成）
         ↓
    Validator（防线 2，7 项检查）
         ↓ 通过
    Executor（dev 库只读，LIMIT 1000）
         ↓
    测试结果 → 进入人审材料
```

**无论代码来源**——LLM 生成的、模板编译器生成的、人手写的——只要进入执行，都必须走这条路。没有例外。

### 3.3 Agent 必须做的事（自测者）

| 必须 | 说明 | 为什么 |
|------|------|--------|
| 每个表/字段引用标注来源 | 格式：`表名.列名（来源：文档名 §章节）` | 让人能快速验证引用的正确性 |
| 同一需求生成 SQL + Spark DSL 两份代码 | 两份实现独立生成，逻辑等价 | 交叉验证的前提——不同实现很难犯相同错误 |
| 生成代码后自动用样本数据跑测试 | LIMIT 1000，仅开发库 | 低级错误在 Agent 阶段就被拦截 |
| SQL 和 Spark DSL 结果交叉验证 | 行数、列名、值分布、抽样行对比 | 两份独立实现的相互校验，发现逻辑不一致 |
| 输出测试结果供人审查 | 含通过/失败、行数、空值率、执行时间、交叉验证对比 | 人是决策者，需要完整信息 |
| 不确定的映射明确标注 | 格式：`⚠️ 不确定：A 还是 B？请确认` | 人不该替 Agent 做它能做的判断，但 Agent 不该猜它不确定的事 |
| 引用 JOIN 白名单中的路径 | 不在白名单标注"需审批" | JOIN 路径是人审批过的，不可随意扩展 |
| 说明 G3/G2 层级选择理由 | 格式：`使用 G3 表 xxx——该表包含所需全部指标` | 让人理解设计意图 |

## 4. 安全模型：三道防线

安全模型从"硬门禁阻断"变为"层层过滤 + 人最终把关"。每一道防线解决不同层面的风险。

### 4.1 防线总览

```
防线 1：生成时约束（Agent 自控层）
  解决什么：减少幻觉进入代码
  怎么工作：LLM 只能引用已注册的表/字段，不确定必须标注
           LLM 生成两份独立代码（SQL + Spark DSL），不同实现难犯相同错误
  失败后果：不确定项进入"不确定清单"，人审时处理
  ↓
防线 2：自动验证（规则引擎层）
  解决什么：拦截确定性的低级错误 + 交叉验证逻辑一致性
  怎么工作：7 项规则检查（表存在性、安全黑名单、权限、白名单、样本执行、质量）
           + SQL/Spark DSL 结果交叉验证
  失败后果：FAIL 阻止进入人审，WARN 标注警告
  ↓
防线 3：人审闸门（人的判断层）
  解决什么：自动验证检查不了的——业务逻辑、方案合理性、指标映射准确性
  怎么工作：人审查全部材料（含交叉验证报告），逐条确认不确定项，最终决定
  失败后果：人拒绝，Agent 修改后重新提交
```

### 4.2 防线 2 的检查项

| # | 检查项 | 失败等级 | 说明 |
|---|--------|---------|------|
| 1 | 表/字段存在性 | FAIL | 对开发库执行 DESCRIBE，确认引用的表和列真实存在 |
| 2 | 安全关键字黑名单 | FAIL | 检测 INSERT/UPDATE/DELETE/DROP/TRUNCATE/ALTER 等 16 个关键字 |
| 3 | 表访问权限 | FAIL | 禁止 bronze.* / silver.* / 未注册表 |
| 4 | JOIN 白名单合规 | FAIL/UNCERTAIN | 在白名单中→通过；不在→标记"需审批" |
| 5 | 样本执行（SQL） | FAIL | 开发库执行 SQL（LIMIT 1000），检查是否报错 |
| 6 | 结果质量 | WARN | 空值率 >30%、行数 = 0、列缺失→标注警告 |
| 7 | **SQL vs Spark DSL 交叉验证** | WARN | 两份代码分别执行，结果集行数/列名/值分布对比；不一致→WARN，人审时调查 |

**交叉验证详解**（检查项 #7）：

```
同一需求
  ├─ SQL 草案（DuckDB）──→ dev 库执行 ──→ 结果集 A
  └─ Spark DSL 草案      ──→ dev 库执行 ──→ 结果集 B
                                              ↓
                                    对比：行数、列名、值范围、抽样行
                                              ↓
                              ┌─ 一致（在容差内）→ 通过
                              └─ 不一致          → WARN，人审时调查差异原因
```

交叉验证的价值在于：**SQL 和 Spark DSL 的表达方式、执行引擎、类型系统都不同，两份代码很难犯完全相同的错误。** 如果两份独立的代码产出了相同的结果，逻辑正确的置信度大幅提高。如果结果不一致，说明至少有一份代码有逻辑错误——人在审查时必须确定哪份是对的，或者两份都错。

如果 Spark 环境在自动验证时不可用：
- Spark DSL 代码仍然生成和审查
- 交叉验证标记为"待 Spark 环境执行"
- 人在上线前在 Spark 环境中手动验证

### 4.3 三道防线之间的关系

- **防线 1 失效时（幻觉进入代码）**：防线 2 的表存在性检查会拦截编造的表名和字段名。交叉验证（检查项 #7）还能发现两份代码之间的逻辑不一致。但如果 LLM 在 SQL 和 Spark DSL 中犯了相同的幻觉（如都编造了同一个不存在的表），防线 2 检查项 #1 会拦截——两份代码都引用不存在的表，都会失败。
- **防线 2 失效时（检查有漏洞）**：防线 3 的人审是最终保障。自动化检查不可能覆盖所有错误类型，人的判断力弥补了这一点。交叉验证报告让人的审查更高效——差异点是人审的重点关注区。
- **防线 3 失效时（人审不充分）**：三道防线设计承认人的审查也可能不完美，但这是所有代码审查都面临的问题。来源追溯机制降低了人审的难度和出错概率。交叉验证通过时，代码逻辑正确的置信度已经很高，人审的重点是业务合理性而非逐行 debug。

### 4.4 生产环境隔离

Agent 在开发环境运行，生产环境对 Agent 完全不可见：

| 环境 | Agent 权限 | 用途 |
|------|-----------|------|
| 开发库 | 只读 + LIMIT 1000 | 自动验证（阶段 4） |
| Spark 开发环境 | 只读 | SQL/Spark 交叉验证（阶段 4） |
| 生产库 | **无权限** | 由人部署后执行 |
| Spark 生产集群 | **无权限** | 由人部署后执行 |

## 5. 数据契约

### 5.1 输入

支持两种格式：**结构化 YAML**（精确，推荐）和**自然语言**（灵活，需要更多验证）。

### 5.2 输出：双份代码 + 审查材料

Agent 不直接输出数据文件。它输出供人审查的完整材料包：

| 产出物 | 说明 | 谁用 |
|--------|------|------|
| SQL 脚本（DuckDB） | 完整、参数化的 SQL，带来源注释 | 人审查后部署到生产执行 |
| Spark DSL 脚本（PySpark） | 同一业务逻辑的 Spark 实现，带来源注释 | 人审查后部署到 Spark 集群 |
| SQL vs Spark 交叉验证报告 | 两份代码的结果对比（行数/列/值分布/抽样） | 人审查时的核心参考——一致=高置信，不一致=重点查 |
| Python 管道脚本 | 多步 DAG 实现（如需要） | 同上 |
| 测试代码 | pytest 格式 | 人审查 + 后续回归测试 |
| 测试结果报告 | 自动验证的运行结果 | 人审查时的核心参考 |
| 设计决策说明 | 为什么选这些表、这样 JOIN、用 G3/G2 | 人审查时的决策参考 |
| 不确定项清单 | Agent 标注"需确认"的事项 | 人审时必须逐条确认 |
| 调度配置 | 可接入调度平台的 YAML | 人审查后部署 |

## 6. 失败处理

### 6.1 失败等级

| 等级 | 含义 | 处理 |
|------|------|------|
| **FAIL** | 硬约束违反，确定性错误 | 阻止进入人审，Agent 需修复后重新提交 |
| **WARN** | 可能有问题，但不一定是错误 | 标注警告，人审时重点检查 |
| **UNCERTAIN** | Agent 无法确定，需要人判断 | 列入不确定项清单，人审时逐条确认 |

### 6.2 常见场景

| 场景 | 等级 | 处理 |
|------|------|------|
| 引用的表不存在 | FAIL | Agent 检查来源文档，修正引用 |
| SQL 含禁止关键字 | FAIL | Agent 修改 SQL，移除危险操作 |
| SQL 执行超时 | FAIL | Agent 优化查询或缩小范围 |
| SQL vs Spark DSL 结果不一致 | WARN | 人审时调查差异——至少一份代码有逻辑错误 |
| 样本返回 0 行 | WARN | 可能日期范围无数据，人确认 |
| 空值率 >30% | WARN | 数据质量问题，人评估影响 |
| 指标映射歧义 | UNCERTAIN | Agent 列出候选，人选择 |
| JOIN 路径不在白名单 | UNCERTAIN | Agent 标注"需审批"，人决定 |

- 禁止在 FAIL 状态下进入人审
- 禁止伪造或省略来源引用

## 9. 架构决策记录

### 决策 1：角色转变（2026-06-16）
- **决策**：Agent 从"生产执行者"变为"代码生成者 + 自测者"，人从"旁观者"变为"最终决策者"
- **原因**：数据开发的现实是最终上线决定始终在人。让 Agent 写代码 + 自测，让人审查 + 拍板，既释放了 LLM 写代码的价值，又用人审管控了风险
- **影响**：LLM 可生成 SQL、推荐表名、写 Python 管道；安全模型从"硬规则阻断"变为"三道防线（约束→验证→人审）"

### 决策 2：三道防线（2026-06-16）
- **决策**：防线 1（生成时约束）→ 防线 2（自动验证）→ 防线 3（人审闸门）
- **原因**：单靠任一防线都不够——prompt 约束不完美、自动验证有盲区、人审有成本。三道防线互补：防线 1 减少幻觉，防线 2 拦截低级错误，防线 3 处理复杂判断

### 决策 3：来源追溯（2026-06-16）
- **决策**：所有表/字段引用必须标注来源
- **原因**：这是降低人审成本的关键——没有来源追溯，审查者需自己查文档验证每个引用；有来源追溯，只需验证"来源是否正确"。同等审查时间，后者能发现更多问题

### 决策 4：双层边界（2026-06-16）
- **决策**：将 Agent 权限边界分为两层——代码生成边界（Code Generation Boundary）和数据执行边界（Data Execution Boundary）
- **原因**：LLM 生成代码和代码被执行是两个不同风险层面的操作。分开定义边界后，LLM 的生成能力被充分释放（边界 1 宽松），但所有代码进入执行时必须过同一道门（边界 2 严格）。约束在数据执行端，不在代码生成端——这比"完全禁止 LLM 写 SQL"更精确、也更实用
- **影响**：AGENTS.md §3 从单一禁令重构为双层边界框架。防线 2（自动验证）是两层之间代码进入执行的唯一通道

### 决策 5：SQL + Spark DSL 双份代码交叉验证（2026-06-16）
- **决策**：LLM 对同一需求独立生成 SQL（DuckDB）和 Spark DSL（PySpark）两份实现，分别执行后对比结果集
- **原因**：两份代码使用不同的表达方式、执行引擎和类型系统，很难犯完全相同的逻辑错误。结果一致 → 置信度大幅提高；结果不一致 → 至少一份代码有错，人审时定位。这是一种"双重记账"式校验——不是检查代码写得对不对，而是检查两份独立代码是否算出了相同答案
- **影响**：阶段 3 代码生成变为双轨输出；防线 2 新增检查项 #7（交叉验证）；人审材料增加交叉验证报告。如果 Spark 环境暂时不可用，Spark DSL 仍生成和审查，交叉验证标记为"待 Spark 环境执行"
