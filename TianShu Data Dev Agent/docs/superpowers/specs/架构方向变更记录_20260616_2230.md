# 架构方向变更记录

> 日期：2026-06-16 | 状态：已确认
>
> **核心变更**：Agent 从"生产执行者"变成"代码生成者 + 自测者"，人从"旁观者"变成"最终决策者"。
>
> 本文档记录从 v1.x 到 v2.0 的架构方向变更——为什么变、变了什么、保留了什么。

---

## 一、为什么变

### 1.1 旧架构的矛盾

旧架构（v1.x）的核心规则是"LLM 不输出代码"——SQL 由模板编译器确定性拼接，LLM 只能输出不含表名的 Intent JSON。

这条规则的逻辑是：如果没有人审查 LLM 的输出，那 LLM 就不能生成代码——因为错误会直接进入生产。

**但问题是：数据开发中，从来没有人会让 Agent 的产出直接进入生产。** 生产代码必须经过 code review。最终上线决定始终在人。

旧架构的"无人审查"假设与实际流程中的"有人审查"现实之间存在根本矛盾。这个矛盾导致了一个次优结果：为了一个不存在的风险（无人审查直接上线），牺牲了 LLM 最有价值的能力（写代码）。

### 1.2 新架构的逻辑

如果实际流程是"有人审查后才上线"，那安全模型就应该基于"人审 + 自动验证"，而不是"完全隔离 LLM"。

这就是 v2.0 的核心逻辑：松开 LLM 的手（让它写代码），加上三道保险（来源追溯 + 自动验证 + 人审闸门）。

### 1.3 一句话总结

> 旧架构：因为怕 LLM 写错代码，所以不让它写。
> 新架构：让 LLM 写，但要求标注来源、自动测试，然后人审查。写错了在审查阶段会被发现。

## 二、变了什么

### 2.1 角色分配

| | 旧架构 | 新架构 |
|------|--------|--------|
| **Agent 角色** | 生产执行者——自动跑管道、产出数据 | 代码生成者 + 自测者——写代码、跑测试、出报告 |
| **人的角色** | 旁观者——提需求、收结果 | 最终决策者——审查代码、拍板上线 |
| **Agent 产出物** | 数据文件（Parquet/CSV） | 代码 + 测试结果 + 决策说明 |
| **上线方式** | 管道自动产出 = 上线 | 人审查通过 → 人部署到生产 |

### 2.2 LLM 边界

| | 旧 | 新 |
|------|----|----|
| LLM 可输出 | 仅结构化 JSON（不含表名/字段名） | 完整代码（SQL/Python/测试/配置） |
| LLM 可推荐 | 无 | 表名、字段名、JOIN 条件（标注来源） |
| LLM 参与范围 | L1（YAML fallback）+ L2（指标消歧） | 阶段 1-3（分析、设计、生成） |
| LLM 不参与 | L3-L8 | 阶段 4（自动验证） |

### 2.3 安全模型

| | 旧 | 新 |
|------|----|----|
| 机制 | 3 道硬门禁（L2/L5/L5） | 三道防线（约束→验证→人审） |
| 执行者 | 全部为确定性代码 | 防线 1（prompt）+ 防线 2（规则引擎）+ 防线 3（人） |
| 核心理念 | 不让 LLM 碰代码 = 安全 | LLM 可以写代码，但三道防线管控风险 |

### 2.4 工作流

| | 旧 | 新 |
|------|----|----|
| 结构 | 8 层管道（L1-L8） | 6 阶段工作流 |
| L3-L4 | LLM 不可逾越的编译区 | 不再存在——LLM 主导代码生成 |
| L5 | SQL 编译后校验 | 保留为防线 2 的自动验证 |

## 三、保留了什么

| 保留项 | 旧角色 | 新角色 |
|--------|--------|--------|
| ColumnBindingTable | 编译器的唯一数据源 | Agent 的参考手册 + 验证的校验基准 |
| JOIN 白名单 | 编译器查表取 JOIN 路径 | Agent 设计时的可选菜单 + 验证时的合规检查 |
| 安全黑名单（16 关键字） | L5 校验 SQL 文本 | 阶段 4 自动验证 |
| G3 优先降级策略 | 代码 `resolve_layer()` | Agent 设计时遵循 + 验证时检查 |
| 参数化查询 | L4 编译器生成 | Agent 生成的 SQL 使用 |
| DuckDB 只读连接 | L6 执行时 | 阶段 4 样本执行时 |
| 禁止 bronze/silver | L5 检查 | 阶段 4 自动验证 |
| 135 个已有测试 | 验证编译器确定性 | 验证防线 2 规则引擎正确性 |

---

## 四、废弃了什么

| 废弃项 | 原因 |
|--------|------|
| LLM 硬边界线（L2-L3 之间） | 边界移到阶段 3-4 之间 |
| 模板编译器优先设计 | LLM 现在直接生成 SQL。编译器代码保留为 fallback |
| 严格 8 层管道 | 简化为 6 阶段 |
| "确定性"作为核心承诺 | 由来源追溯 + 自动验证 + 人审等效保障 |
| Intent IR 不含表名的硬约束 | LLM 现在可输出表名，但需标注来源 |

## 五、影响评估

### 代码层面
- `layer4_generate.py`（模板编译器）：保留为安全 fallback，不作为唯一路径
- `layer5_validate.py`（安全校验）：保留为防线 2 规则引擎
- `layer3_plan.py`：确定性查表逻辑保留，作为 Agent 设计阶段的参考
- 其他 Layer 代码：保留为自动验证和审查输出的引擎

### 测试层面
- 135 个已有测试：保留，验证防线 2 的正确性
