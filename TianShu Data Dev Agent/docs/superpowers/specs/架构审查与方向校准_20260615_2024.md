# 架构审查与方向校准

> 日期：2026-06-15 | 状态：已确认 | 路线：A（保守演进）
>
> 审查范围：LLM 边界、管道层数、IR 设计、代码规模与模块边界

---

## 一、审查背景

四个具体架构顾虑触发本次审查：

1. **LLM 边界是否安全**——L1-L2 允许 LLM 参与，边界是否足够靠前？LLM 能否间接影响下游？
2. **管道层数是否过多**——8 层 vs Text2SQL Agent 的 3 层 IR，是否过度设计？
3. **IR 设计是否合理**——Requirement→Intent→SQLPlan→ExpressionRef 等 IR 层级是否需要精简？
4. **代码规模与模块边界**——6000 行管道代码，layer3_plan.py 近 1000 行，是否需要拆分？

**审查结论**：当前架构骨架正确。问题在文档缺失和模块边界不够清晰，不在架构设计本身。选择**路线 A（保守演进）**——不改架构骨架，针对性修复四个问题。

## 二、变更清单

### 2.1 LLM 边界文档化

**问题**：LLM 边界已在代码中正确实现，但缺少显式文档标注。

**变更**：

| 文件 | 变更 | 行数 |
|------|------|------|
| `AGENTS.md` §3.1 | "LLM 不能做什么"表格新增"代码证据"列，每条禁止项映射到具体代码位置 | +15 |
| `layer3_plan.py` 文件头 | 强化 LLM 硬边界注释——显式声明 L3-L8 禁止 LLM，最坏情况是 BLOCKED | +8 |
| Obsidian 架构设计文档 | 新增"LLM 防火墙"专题——LLM 可见 vs 不可见的信息分界图 | +30 |

**LLM 证据链**：

```
LLM 可见（Intent 中的字段）     LLM 不可见（被 L3 隔离）
─────────────────────────────    ─────────────────────────
metrics: ["trip_count"]          表名: gold.dws_daily_trip_summary
dimensions: ["date"]             列名: trip_count, trip_date
filters: {date_range: [...]}     JOIN: dim_date.date = fact.trip_date
                                 策略: G3_DIRECT vs G2_FACT
                                 方言: DuckDB vs Hive vs PG
```

安全保证：即使 LLM 产生幻觉（错误的指标名），最坏结果也是 L3 查 ColumnBindingTable 失败 → BLOCKED，不会生成错误的 SQL。

### 2.2 层数粒度的设计决策记录

**问题**：缺少文档解释为什么 8 层，与 Text2SQL 的 3-IR 对比时容易产生"过度设计"的直觉判断。

**变更**：

| 文件 | 变更 | 行数 |
|------|------|------|
| `PROJECT_STATUS.md` §六 | 新增"决策 7：为什么 8 层"，逐层解释不可合并的理由 | +25 |

**核心论据**：8 层是批量生产管道的**最小必要层数**。Text2SQL 的 3 层对交互查询是正确的，但它如果要做批量生产，也需要补上独立校验/评估/发布层。

| 合并提议 | 为什么不行 |
|---------|-----------|
| L1+L2 合并 | L1 是规则解析 YAML，L2 可能调用 LLM。合并后 LLM 边界模糊 |
| L3+L4 合并 | L3 是 IR 构造，L4 是方言编译。分离后可以独立测试方言 |
| L5+L4 合并 | L5 是安全校验。分离保证校验可以独立执行、独立审计 |
| L6+L7+L8 合并 | L7 评估可能产生 DIRTY 标记但仍需 L8 输出产物 |

### 2.3 IR 设计文档

**问题**：IR 类型分散定义在多个文件中，缺少一份文档解释每个 IR 为什么存在。

**变更**：

| 文件 | 变更 | 行数 |
|------|------|------|
| `PROJECT_STATUS.md` §三 | 扩展 IR 成熟度表为完整 IR 层级说明，新增"信息防火墙"视角 | +40 |

**IR 信息隔离链**：

```
Requirement IR     → 隔离 YAML 语法细节
Intent IR          → 隔离用户原始措辞（LLM 防火墙在此）
SQLPlan IR         → 隔离指标→列的映射逻辑
ExpressionRef IR   → 隔离表达式树结构（嵌入 SQLPlan）
SQL 文本           → 隔离方言特定语法
ExecutionResult    → 隔离数据库连接细节
Product 文件       → 隔离 Parquet/CSV 内部格式
```

每个 IR 只暴露下一层需要的信息，不多不少。

## 三、不变更清单

以下事项在本次审查中明确**不需要改变**：

| 事项 | 理由 |
|------|------|
| 管道层数（8 层） | 每层职责不可合并，是对批量生产的最小必要层数 |
| IR 类型数量 | 每个 IR 解决一个具体的隔离问题，删减会引入耦合 |
| L4 子编译器结构（5 个文件） | 注册表模式需要独立模块，拆分合理 |
| ColumnBindingTable 加载机制 | 当前 TianShu → static fallback 设计正确 |
| LLM 在 L1-L2 的参与方式 | 边界位置正确——LLM 只能输出 Intent，不接触物理层 |
