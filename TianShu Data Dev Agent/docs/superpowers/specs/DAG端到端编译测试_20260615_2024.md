### 1.3 范围

- **编译级（dry-run）**：只执行 L3-L5 编译管道，不连接数据库

- **不涉及增量逻辑**：不测试分区写入、水位线等运行时行为

## 2. 架构设计

### 2.1 测试文件

```
tests/test_pipeline_dag_e2e.py        # 新增，~350 行
tests/fixtures/dag_e2e/               # 新增目录
├── chain_5step.yml                   # 链式 DAG，5 步
├── diamond_4step.yml                 # 菱形 DAG，4 步
├── complex_6step.yml                 # 复杂 DAG，6 步
└── single_step.yml                   # 单步边界，1 步
```

### 2.2 测试类结构

```
tests/test_pipeline_dag_e2e.py
├── 辅助函数层
│   ├── _load_yaml_fixture(name) → PipelinePlan
│   ├── _compile_step(step) → str
│   └── _assert_sql_structure(sql, expected)
│
├── TestYAMLFxtureLoading          # 3 个用例：加载 3 种 DAG 模式
├── TestDAGCompileDryRun           # 5 个用例：全步骤编译 + 依赖引用 + 错误暴露
└── TestSQLStructureValidation     # 按 fixture 分组的 SQL 结构断言
```

### 2.3 YAML Fixture 格式

Fixture 是 PipelinePlan 的"输入数据"，不是 IR 的直接序列化。格式遵循简化约定：

```yaml
pipeline_id: "e2e_chain_5step"
pipeline_name: "链式5步管道-端到端测试"
target_dialect: "duckdb"

steps:
  - step_id: "extract_trip"
    step_name: "提取行程原始数据"
    operation: "CREATE_TABLE_AS_SELECT"
    target_table: "generated.tmp_trip_raw"
    depends_on: []
    sql_plan:
      primary_table: "gold.dws_daily_trip_summary"
      column_bindings:
        - { column_name: "city_id", alias: "city_id" }
        - { column_name: "trip_count", alias: "trip_count" }
      filter_bindings:
        - { column_name: "dt", operator: ">=", value: "2026-01-01" }
      group_by: []          # 可选
      join_tables: []        # 可选
```

关键设计决策：
- `sql_plan` 嵌套深度只到 `column_bindings` + `filter_bindings` + `group_by` + `join_tables`
- 步骤间通过 `depends_on` + `target_table` 引用
- 不包含 `expression_refs`（V1.0 表达式编译器仍在开发中）

### 2.4 SQL 结构验证矩阵

| 检查类别 | 检查项 | 验证方式 |
|---------|--------|---------|
| 子句存在性 | SELECT / FROM / WHERE / GROUP BY / JOIN | 关键字存在检查 |
| 操作语法 | CTAS / INSERT OVERWRITE / CREATE VIEW | 关键字开头匹配 |
| 标识符引用 | 表名 / 列别名 / 过滤列名 | 子字符串存在检查 |
| 依赖传递 | 上游 target_table 出现在下游 FROM | 跨步骤断言 |
| 安全黑名单 | 无 DROP / DELETE / TRUNCATE | 黑名单关键字检查 |

## 3. Fixture 定义

### 3.1 chain_5step.yml — 链式 DAG

```
extract_trip → aggregate_daily → enrich_city → tag_anomaly → final_report
```

- 覆盖 5 种操作：CTAS × 4 + SELECT_ONLY × 1
- 覆盖：简单 SELECT、GROUP BY 聚合、LEFT JOIN、WHERE 过滤
- 关键验证：步骤间表引用链（generated.tmp_trip_raw → generated.agg_trip_daily → generated.enriched_trip → generated.tagged_trip）

### 3.2 diamond_4step.yml — 菱形 DAG

```
extract_base → compute_city_agg ──→ merge_report
             → compute_date_agg ──→
```

- 覆盖：扇出 + 扇入
- compute_city_agg 和 compute_date_agg 并行依赖 extract_base
- merge_report 依赖两个分支汇聚
- 覆盖：CROSS JOIN

### 3.3 complex_6step.yml — 复杂 DAG

```
extract_raw → clean_and_filter → dim_city_stats ──→ incremental_merge
                                → dim_city_stats ──→ final_view
                                → dim_date_stats ──→ final_view
```

- 覆盖：多层依赖 + 多分支 + INSERT OVERWRITE_PARTITION + CREATE VIEW
- dim_city_stats 同时被 incremental_merge 和 final_view 依赖
- final_view 依赖两个分支（dim_city_stats + dim_date_stats）

### 3.4 single_step.yml — 边界测试

```
lone_view  (CREATE VIEW，零依赖)
```

- 覆盖：最小 DAG（1 步 0 依赖）+ CREATE VIEW

## 5. 关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 测试深度 | 编译级 dry-run | 不连数据库，纯静态验证——快速、确定性 |
| YAML vs Python fixture | YAML | 可读性好，跨语言可复用，便于非开发人员编写 |
| PipelinePlan 构造方式 | YAML → 手动构造 IR | 不修改 IR dataclass 的序列化逻辑——fixture 是"输入数据" |
| 辅助函数 `_assert_sql_structure` | 声明式 dict 驱动 | 避免每个测试用例写重复的 `assert 'SELECT' in sql` |
| 覆盖的操作类型 | 全部 5 种 StepOperation | SELECT_ONLY / CTAS / INSERT OVERWRITE / INSERT INTO / CREATE VIEW |
