## 审计摘要

### 当前实现

| 组件 | 位置 | 当前行为 |
|------|------|---------|
| `DecisionRecord` | `src/ir/types.py:648` | 简单 dataclass，`default_status="REQUEST_CHANGES"`，`options=[APPROVE/REQUEST_CHANGES/REJECT]` |
| `ReviewPackageManifest` | `src/ir/types.py:670` | `status: str = "pending_human_review"`，纯字符串，非枚举 |
| `publish_review_package()` | `src/agent/review_publisher.py:157` | 生成 decision.md（静态模板），永远 REQUEST_CHANGES |
| `verify_review_package()` | `src/agent/verification_engine.py:40` | 读取 decision.md 确认存在，**绝不修改** |
| `decision.md` | `generated/review_packages/*/` | 纯 Markdown，无结构化状态字段 |
| 测试覆盖 | 5 个文件共 ~15 个断言 | 验证默认非 APPROVE、M3 不改 decision、三选项存在 |

### 核心缺口

1. **decision.md 是一次性模板**——没有跨验证运行的状态保持
2. **无状态流转机制**——人改 decision.md 后，没有程序化校验前置条件
3. **无审计日志**——谁在何时做了什么决定，无迹可查
4. **无 SUPERSEDED 语义**——重新验证后旧批准自动失效？目前无处理
5. **ReviewPackageManifest.status 是自由字符串**——`"pending_human_review"` 仅文档约定，非枚举约束

## 1. 状态集合

```
                   ┌─────────────┐
                   │ PENDING_    │  ← 初始状态（M2 生成 Review Package 时）
                   │ REVIEW      │
                   └──────┬──────┘
                          │ 人执行 review_decision set
          ┌───────────────┼───────────────┐
          ▼               ▼               ▼
   ┌──────────┐   ┌──────────────┐   ┌──────────┐
   │ APPROVED │   │ REQUEST_     │   │ REJECTED │
   │          │   │ CHANGES      │   │          │
   └────┬─────┘   └──────┬───────┘   └──────────┘
        │                │
        │  M3 重新验证    │  人修改后重新生成
        ▼                ▼
   ┌──────────┐   ┌──────────────┐
   │ SUPERSEDED│   │ PENDING_     │  ← 新版 Review Package 重进审查
   │          │   │ REVIEW       │
   └──────────┘   └──────────────┘
```

### 状态定义

| 状态 | 含义 | 设置者 | 说明 |
|------|------|--------|------|
| `PENDING_REVIEW` | 待审查——Review Package 已生成，等人审 | Agent（M2/M3） | 初始状态，M2 生成时自动设置 |
| `APPROVED` | 已批准——人可以部署上线 | **仅人** | Agent 绝对不可设置 |
| `REQUEST_CHANGES` | 需修改——人审后发现需调整 | **仅人** | 附带修改意见 |
| `REJECTED` | 已拒绝——此需求/方案不通过 | **仅人** | 终态（或可重新提交） |
| `SUPERSEDED` | 已替代——新版 Review Package 覆盖了旧批准 | Agent（M2/M3） | 旧版 APPROVED 在新验证后自动置为此状态 |

**关于 REJECTED → PENDING_REVIEW 回流**：REJECTED 后如果需求调整重新提交，应生成**新的 `request_id`**（如 `trip_daily_v2`），旧 request_id 保持 REJECTED。不在同一 request_id 内回流，避免审计混乱。

## 2. 状态流转规则

### 流转矩阵

| 从 → 到 | PENDING_REVIEW | APPROVED | REQUEST_CHANGES | REJECTED | SUPERSEDED |
|----------|:---:|:---:|:---:|:---:|:---:|
| **PENDING_REVIEW** | — | 仅人 | 仅人 | 仅人 | — |
| **APPROVED** | — | — | — | — | Agent（M2/M3 重新生成） |
| **REQUEST_CHANGES** | Agent（M2 重新生成后） | 仅人 | 仅人 | 仅人 | — |
| **REJECTED** | —（需新 request_id） | — | — | — | — |
| **SUPERSEDED** | — | 仅人（重新批准新版） | 仅人 | 仅人 | — |

### 关键规则

1. **初始状态**：`PENDING_REVIEW`——M2 `publish_review_package()` 设置
2. **Agent 只能写两个状态**：
   - `PENDING_REVIEW`——生成新 Review Package 时
   - `SUPERSEDED`——M3 重新验证后，检测到旧状态为 APPROVED 时自动置为 SUPERSEDED
3. **人手动设置的状态**：`APPROVED`、`REQUEST_CHANGES`、`REJECTED`——Agent 在任何情况下都**不得**设置这三个状态
4. **APPROVED 前置条件**（CLI 强制校验）：
   - `verification.md` 必须存在且 `overall_status ∈ {PASS, WARN}`
   - 不存在 `FAIL` 状态的检查项
   - 所有 `PENDING` 项有人审确认注释
5. **SUPERSEDED 触发条件**：M3 重新验证时，若检测到当前 decision 状态为 `APPROVED`，自动将其标记为 `SUPERSEDED`——因为新版验证报告可能推翻旧结论

### 关于 FAIL 状态下能否 APPROVED 的建议

**建议：禁止。** 理由：

- FAIL 意味着防线 2（自动验证）发现了确定性问题——SQL 语法错误、表不存在、禁止关键字等。这些不是"人审可以 override 的业务判断"，而是"代码跑不通"的客观事实。
- 如果人在 FAIL 状态下强行 APPROVED，意味着人主动绕过自动验证——这违背三道防线的设计初衷。
- 人如果确实认为 FAIL 是误报（如 Validator bug），应先修复 Validator 或调整检查规则，而非在 FAIL 下强行批准。

**实现**：CLI `review_decision set APPROVED` 执行前校验 `overall_status != FAIL`，若为 FAIL 则拒绝并提示。

## 3. 审计日志设计

### 数据结构

```yaml
# decision_log.yml（位于 Review Package 根目录）
decisions:
  - actor: "human"              # human | agent
    actor_id: "huangkwyoyo"     # git user 或 agent 版本号
    timestamp: "2026-06-17T14:30:00+08:00"
    from_state: "PENDING_REVIEW"
    to_state: "REQUEST_CHANGES"
    reason: "SQL 中 trip_count 口径与业务定义不一致，需修正"
    verification_report_ref: "reports/verification.md"
    verification_overall_status: "PASS"
    commit_sha: "abc1234"       # 可选：关联的 git commit

  - actor: "agent"
    actor_id: "m3-verify-v2.0"
    timestamp: "2026-06-17T15:00:00+08:00"
    from_state: "APPROVED"
    to_state: "SUPERSEDED"
    reason: "M3 重新验证——新版 Review Package 覆盖旧批准"
    verification_report_ref: "reports/verification.md"
    verification_overall_status: "PASS"

  - actor: "human"
    actor_id: "huangkwyoyo"
    timestamp: "2026-06-17T16:00:00+08:00"
    from_state: "SUPERSEDED"
    to_state: "APPROVED"
    reason: "新版验证通过，口径已修正，批准上线"
    verification_report_ref: "reports/verification.md"
    verification_overall_status: "PASS"
```

### 设计要点

- **追加不修改**：每条决策是追加日志行，不修改历史记录
- **actor 明确**：`human` 或 `agent`，不模糊
- **from_state / to_state**：完整记录流转
- **reason 必填**：人对 APPROVED/REQUEST_CHANGES/REJECTED 必须写理由；Agent 可自动生成
- **verification_report_ref**：指向当时的验证报告，防止报告被覆盖后审计断链

## 4. 文件设计

### 方案：decision.md（人读） + decision.yml（机读） + decision_log.yml（审计）

```
generated/review_packages/{request_id}/
├── sql/main.sql
├── spark/main.py
├── tests/test_generated.py
├── reports/
│   ├── verification.md          # M3 验证报告（Agent 写）
│   └── cross_validation.md      # M3 交叉验证报告（Agent 写）
├── lineage/source_refs.yml
├── decision.md                  # 人读决策文件（Agent 生成模板，人填写）
├── decision.yml                 # 机读决策状态（结构化，Agent + CLI 读写）
└── decision_log.yml             # 审计日志（追加写入，不可删改）
```

### 职责分工

| 文件 | 格式 | 写入者 | 读取者 | 用途 |
|------|------|--------|--------|------|
| `decision.md` | Markdown | Agent 生成模板，人编辑 | 人 | 人类可读的审查意见和决策 |
| `decision.yml` | YAML | CLI（人触发），Agent（仅写 PENDING_REVIEW / SUPERSEDED） | CLI、Agent | 结构化状态——程序校验的数据源 |
| `decision_log.yml` | YAML 列表 | CLI（追加），Agent（追加） | CLI `audit` 子命令 | 完整审计轨迹 |

### decision.yml 结构

```yaml
# decision.yml——结构化人审状态
request_id: "trip_daily_report_m2"
current_state: "REQUEST_CHANGES"   # PENDING_REVIEW | APPROVED | REQUEST_CHANGES | REJECTED | SUPERSEDED
last_updated: "2026-06-17T14:30:00+08:00"
last_updated_by: "huangkwyoyo"
verification_report_ref: "reports/verification.md"
verification_overall_status: "PASS"
human_decision_note: "SQL 中 trip_count 口径需修正"
```

### 为什么保留 decision.md

`decision.md` 是给人看的——Markdown 格式适合写详细的审查意见、业务上下文和自由文本。`decision.yml` 是给程序看的——状态机需要结构化数据。两者互补，不替代。人审时如果有详细的业务分析（几百字的审查意见），放在 `decision.md` 中；状态流转只需要 `decision.yml` 中的结构化字段。

### Review Package Manifest 引用

`ReviewPackageManifest.status` 从自由字符串改为枚举 `DecisionStatus`，`files` 列表增加 `decision.yml` 和 `decision_log.yml`。

## 5. CLI 设计

### 入口

```
scripts/dev_agent/review_decision.py
```

### 子命令

#### `show`——查看当前决策状态

```powershell
python scripts/dev_agent/review_decision.py show \
    -p generated/review_packages/trip_daily_report_m2
```

输出：
```
Request ID:  trip_daily_report_m2
当前状态:    REQUEST_CHANGES
最后更新:    2026-06-17T14:30:00+08:00 (huangkwyoyo)
验证状态:    PASS
人审备注:    SQL 中 trip_count 口径需修正

审计日志（最近 5 条）:
  2026-06-17T14:30  human  PENDING_REVIEW → REQUEST_CHANGES
  2026-06-17T13:00  agent  PENDING_REVIEW（M2 生成）
```

#### `set`——人设置决策（Agent 禁止调用此子命令）

```powershell
# 批准
python scripts/dev_agent/review_decision.py set APPROVED \
    -p generated/review_packages/trip_daily_report_m2 \
    -m "口径已修正，验证通过，批准上线"

# 需修改
python scripts/dev_agent/review_decision.py set REQUEST_CHANGES \
    -p generated/review_packages/trip_daily_report_m2 \
    -m "SQL 中缺少日期过滤条件，请补充后重新提交"

# 拒绝
python scripts/dev_agent/review_decision.py set REJECTED \
    -p generated/review_packages/trip_daily_report_m2 \
    -m "此需求涉及的数据表尚未就绪，暂不实施"
```

**前置校验**（`set APPROVED` 时强制执行）：
1. `verification.md` 存在且 `overall_status != FAIL`
2. `--message` / `-m` 不为空
3. 当前状态不是 `REJECTED`（终态不可直接流转）
4. 当前状态为 `SUPERSEDED` 时，提示"此 Review Package 已替代旧批准版本"，需确认

#### `audit`——查看完整审计轨迹

```powershell
python scripts/dev_agent/review_decision.py audit \
    -p generated/review_packages/trip_daily_report_m2
```

输出完整 `decision_log.yml` 的历史记录，按时间正序。

### 安全约束

- `set` 子命令不接受任何非人工触发的方式（不接受 `--auto` 参数，不接受管道输入——`reason` 必须显式提供）
- 代码中不暴露 `set` 的函数接口给 Agent 调用——Agent 只能调用 `transition_to_pending_review()` 和 `transition_to_superseded()` 两个受限函数
- CLI 的 `set` 命令在 `review_decision.py` 中实现，`src/agent/` 不引用它

## 7. 风险评估

### 7.1 人审被绕过

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| CLI `set APPROVED` 被脚本化 | 🟡 中 | `reason` 必填且非空；审计日志记录 actor；git pre-commit hook 可校验 |
| Agent 代码中直接 import CLI 函数 | 🟢 低 | `set` 实现在 `scripts/dev_agent/review_decision.py`，`src/agent/` 不引用——架构隔离 |
| 人直接编辑 `decision.yml` 绕过 CLI | 🟡 中 | 无法完全防止（人始终有文件系统权限）。CLI 是推荐路径，但决策权在人——如果人选择绕过 CLI，那也是人的决定 |

### 7.2 报告过期但仍批准

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| M3 重新验证后，人仍基于旧报告批准 | 🟡 中 | `verify_review_package()` 自动检测并置旧 APPROVED 为 SUPERSEDED；CLI 通过 `verification_report_ref` 对比最新报告时间戳 |
| 人工改了 SQL 但未重新验证 | 🟡 中 | decision.yml 记录 `verification_report_ref` + `commit_sha`；CLI `set APPROVED` 前校验验证报告时间 > 代码最后修改时间 |

### 7.3 多次验证覆盖报告

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| M3 重跑覆盖了旧 `verification.md` | 🟢 低 | `verification.md` 是 M3 的一手产物，覆盖是正确的——最新验证应反映最新代码状态；旧状态通过 `decision_log.yml` 保留审计引用 |
| `decision_log.yml` 被删除或篡改 | 🟡 中 | 追加写入（不覆盖），建议入 git 版本控制 |

### 7.4 generated 产物是否应入 git

| 观点 | 理由 |
|------|------|
| **建议入 git** | `decision.yml` + `decision_log.yml` 是审计记录——谁在何时批准了什么，是合规要求的一部分，应与代码共同版本控制 |
| **不建议入 git** | `verification.md` / `cross_validation.md` 是 M3 自动生成的可复现产物——可从代码重新生成 |
| **推荐方案** | `decision.yml` + `decision_log.yml` **入 git**（审计记录，不可复现）；`verification.md` + `cross_validation.md` 入 `.gitignore`（可从代码重跑 M3 复现） |
