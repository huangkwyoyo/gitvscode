# 数据开发 Agent 设计文档

> 版本：v2.3 | 日期：2026-06-14 | 阶段：Phase 1-7 编译器全家福已交付 + 安全审计 P0-P2 全部修复（135 测试，零回归）

---

## 1. 项目定位

TianShu Data Dev Agent 是建立在 TianShu 数据仓库（DuckDB + NYC 交通数据）之上的**确定性数据生产管道系统**。

**不是自由聊天机器人。不是 Text2SQL 查询工具。是数据生产系统。**

核心原则：
- **LLM 只做"翻译"**（自然语言 → 结构化 JSON），不做"工程决策"
- **SQL 由编译器生成**，不由 LLM 或人手写
- **ColumnBindingTable 是中枢神经**，所有字段映射通过它确定
- **安全策略不可绕过**，由纯规则引擎强制执行

## 2. 架构总览

```
         ┌──────────────────────────────────────────┐
         │  1. 需求解析层（Requirement Layer）         │  YAML 规则解析器
         │     输入：需求说明书 YAML                    │  LLM 仅作 fallback
         │     输出：结构化 Requirement 对象            │
         └──────────────┬───────────────────────────┘
                        ↓
         ┌──────────────────────────────────────────┐
         │  2. 意图理解层（Intent Layer）              │  LLM 辅助：指标消歧
         │     输入：Requirement 对象                  │  输出严格受限 JSON
         │     输出：Intent 对象（WHAT，不含 HOW）      │  ← LLM 最后出现的地方
         └──────────────┬───────────────────────────┘
                        ↓
  ═══════════════════════════════════════════════════════
              LLM 边界 —— 以下全部确定性代码
  ═══════════════════════════════════════════════════════
                        ↓
         ┌──────────────────────────────────────────┐
         │  3. 规划层（Planning Layer）               │  纯确定性：查表 + 规则
         │     输入：Intent 对象                      │  构造 JoinGraph
         │     输出：完整 SQLPlan IR                  │  决定 G3/G2 层级
         └──────────────┬───────────────────────────┘
                        ↓
         ┌──────────────────────────────────────────┐
         │  4. 生成层（Generation Layer）             │  纯确定性：模板编译器
         │     输入：SQLPlan IR                      │  所有标识符来自 CBTable
         │     输出：完整 SQL 文本 + 参数列表          │  支持别名映射
         └──────────────┬───────────────────────────┘
                        ↓
         ┌──────────────────────────────────────────┐
         │  5. 校验层（Validation Layer）             │  纯规则引擎
         │     输入：SQL 文本 + SQLPlan               │  安全黑名单 + 语义检查
         │     输出：校验报告（通过/拒绝）             │  JOIN 白名单 + 日期合规
         └──────────────┬───────────────────────────┘
                        ↓
         ┌──────────────────────────────────────────┐
         │  6. 执行层（Execution Layer）              │  DuckDB 只读连接
         │     输入：已验证的 SQL + 参数               │  参数化查询防注入
         │     输出：DataFrame + 执行元数据            │  30s 超时
         └──────────────┬───────────────────────────┘
                        ↓
         ┌──────────────────────────────────────────┐
         │  7. 评估层（Evaluation Layer）             │  纯统计检查
         │     输入：DataFrame + 预期约束             │  空值率、行数、列完整性
         │     输出：评估报告                         │
         └──────────────┬───────────────────────────┘
                        ↓
         ┌──────────────────────────────────────────┐
         │  8. 产出层（Product Layer）                │  结果文件 + 报告 + 任务配置
         │     输入：DataFrame + 评估报告 + SQLPlan   │  Parquet / CSV / Markdown
         │     输出：generated/ 目录中的产物           │
         └──────────────────────────────────────────┘
```

## 3. LLM 职责边界（不可突破）

### 3.1 LLM 允许做的事

| 层 | 允许 | 机制 |
|----|------|------|
| Layer 1 | YAML schema 不匹配时辅助解析 | 规则解析器优先，LLM 仅 fallback |
| Layer 2 | 指标消歧（用户说"行程量"→ 匹配已注册的 `trip_count`） | LLM 只能输出 Intent JSON |
| Layer 2 | 识别维度和过滤条件 | 输出中不含表名/字段名 |
| Layer 2 | 判断是否需要跨域查询 | 标记 domain 为 cross_domain |

### 3.2 LLM 绝对禁止做的事

| 禁止行为 | 为什么禁止 | 替代方案 |
|---------|-----------|---------|
| 输出/推荐表名 | LLM 可能编造不存在的表 | Layer 3 通过 metric_registry 查表 |
| 输出/推荐字段名 | LLM 可能编造不存在的字段 | ColumnBindingTable 预编译映射 |
| 决定 G3 或 G2 | 性能和安全决策，需查契约 | `resolve_layer()` 函数决定 |
| 写 JOIN 条件 | JOIN 必须来自白名单 | JoinGraph 数据结构构造 |
| 生成 SQL 或其片段 | SQL 必须是确定性编译的 | Layer 4 模板编译器 |
| 绕过安全策略 | 安全策略是硬约束 | Layer 5 纯规则引擎 |

## 4. 核心数据结构

### 4.1 ColumnBindingTable（系统中枢神经）

管道启动时从 `meta.metric_definitions` 动态加载，结合静态绑定作为 fallback。运行时不可变，LLM 不可访问。

当前注册 **10 个指标**，覆盖 **4 个业务域**：

| 域 | 指标名 | G3 列 | G2 表达式 | G2 表 |
|----|--------|-------|----------|-------|
| traffic | trip_count | dws_daily_trip_summary.trip_count | COUNT(*) | fact_trips |
| traffic | total_fare_amount | dws_daily_trip_summary.total_fare_amount | SUM(fare_amount) | fact_trips |
| traffic | total_tip_amount | —（G3 不含此列） | SUM(tip_amount) | fact_trips |
| traffic | total_distance_miles | dws_daily_trip_summary.total_distance_miles | SUM(trip_miles) | fact_trips |
| violation | parking_violation_count | dws_daily_parking_summary.violation_count | COUNT(*) | fact_parking_violations |
| violation | standard_fine_total | dws_daily_parking_summary.standard_fine_total | SUM(standard_fine_amount) | dim_violation_type |
| safety | crash_count | dws_daily_crash_summary.crash_count | COUNT(*) | fact_crashes |
| safety | persons_killed | dws_daily_crash_summary.persons_killed | SUM(persons_killed) | fact_crashes |
| safety | persons_injured | dws_daily_crash_summary.persons_injured | SUM(persons_injured) | fact_crashes |
| supply | tif_payment_amount | —（无 G3 汇总表） | SUM(total_payment_amount) | fact_tif_payments |

维度绑定（date 维度在多张表中的列名）：

| 表 | date 维度的列名 | 类型 | 需 JOIN dim_date |
|----|----------------|------|------------------|
| gold.dws_daily_trip_summary | trip_date | DATE | 否 |
| gold.dws_daily_parking_summary | issue_date | DATE | 否 |
| gold.dws_daily_crash_summary | crash_date | DATE | 否 |
| gold.fact_trips | pickup_date_key | INTEGER | **是** |
| gold.fact_parking_violations | issue_date_key | INTEGER | **是** |
| gold.fact_crashes | crash_date_key | INTEGER | **是** |
| gold.fact_tif_payments | payment_date_key | INTEGER | **是** |

**G2 日期过滤契约**：当使用 G2 fact 表时，日期列是整数代理键（`_key` 后缀），管道自动检测并：
1. 自动 JOIN `gold.dim_date`
2. 过滤条件切换到 `dim_date.date`（实际 DATE 类型）
3. Layer 5 校验 `requires_date_dim` 确保 dim_date 已正确 JOIN

### 4.2 JoinGraph（多表关系表达）

```yaml
# 单表查询（Phase 1 最常见）
join_graph:
  primary:
    table: "gold.dws_daily_trip_summary"
    alias: "dws_daily_trip_summary"       # 单表无 JOIN 时不使用别名
  joins: []

# 跨域 JOIN（trip ↔ crash 通过日期关联）
join_graph:
  primary:
    table: "gold.dws_daily_trip_summary"
    alias: "dws_daily_trip_summary"
  joins:
    - table: "gold.dws_daily_crash_summary"
      alias: "dws_daily_crash_summary"
      type: "LEFT JOIN"
      condition:
        left: "dws_daily_trip_summary.trip_date"     # left_key 属于左表
        right: "dws_daily_crash_summary.crash_date"   # right_key 属于右表
      constraint_ref: "sql_safety_policy.yml#join_whitelist"

# G2 降级 JOIN（fact 表 + dim_date）
join_graph:
  primary:
    table: "gold.fact_trips"
    alias: "fact_trips"
  joins:
    - table: "gold.dim_date"
      alias: "dim_date"
      type: "INNER JOIN"
      condition:
        left: "fact_trips.pickup_date_key"
        right: "dim_date.date_key"
      constraint_ref: "sql_safety_policy.yml#join_whitelist"
```

**Phase 1 约束**：`joins` 最多 2 个元素（跨域 JOIN + dim_date JOIN）。

**Phase 2 扩展**：`joins` 可增加到 N 个，支持 fact + dim + enrichment 多步依赖，支持管道内部步骤间的表引用（`$step.xxx.target_table`）。

### 4.3 SQLPlan（Layer 3 → Layer 4 的唯一接口）

```yaml
plan_id: "plan_20260614_170000_1234"
plan_name: "2026年Q1每日行程量与事故量趋势对比"
source_layer: "g3"                       # g3 | g2
domain: "cross_domain"                   # traffic | violation | safety | supply | cross_domain

join_graph: {...}                        # 见上节

column_bindings:                         # 指标列绑定
  - metric_name: "trip_count"
    fully_qualified: "gold.dws_daily_trip_summary.trip_count"
    alias: "行程量"
    domain: "traffic"
  - metric_name: "crash_count"
    fully_qualified: "gold.dws_daily_crash_summary.crash_count"
    alias: "事故数量"
    domain: "safety"

dimension_bindings:                      # 维度列绑定
  - dim_name: "date"
    fully_qualified: "gold.dws_daily_trip_summary.trip_date"
    alias: "日期"

filter_bindings:                         # 过滤条件绑定（参数化）
  - filter_type: "date_range"
    fully_qualified: "gold.dws_daily_trip_summary.trip_date"
    operator: "BETWEEN"
    value: ["2026-01-01", "2026-03-31"]

group_by: ["gold.dws_daily_trip_summary.trip_date"]
order_by: [{"column": "gold.dws_daily_trip_summary.trip_date", "direction": "ASC"}]

execution_constraints:
  read_only: true
  query_timeout_seconds: 30
  max_result_rows: 100000
  allowed_tables: ["gold.dws_daily_trip_summary", "gold.dws_daily_crash_summary"]
  requires_date_dim: false              # G3 表直接含 DATE 列，无需 dim_date

warnings: []
is_valid: true
```

### 4.4 Intent（Layer 2 → Layer 3 的唯一接口，LLM 的最终输出）

```yaml
description: "按天统计2026年Q1的行程量和事故量，生成每日对比趋势"
metrics_requested:
  - registered_name: "trip_count"        # 必须精确匹配已注册指标名
    fuzzy_matched: false
    user_name: "行程量"
  - registered_name: "crash_count"
    fuzzy_matched: false
    user_name: "事故数量"
dimensions:
  - name: "date"
    alias: "日期"
filters:
  date_range: ["2026-01-01", "2026-03-31"]
group_by: ["date"]
domain: "cross_domain"
confidence:
  metric_match: "high"
  dimension_match: "high"
```

**关键**：Intent 里没有表名、没有字段名、没有 JOIN。这些由 Layer 3 确定性构造。

## 5. 管道层接口定义

### Layer 1 → Layer 2

```
输入：YAML 文件路径
输出：Requirement 对象
      {name, description, metrics[{name}], dimensions[{name}],
       filters{date_range[]}, group_by[], order_by[], output{format}}
机制：规则解析器优先；需 LLM 辅助时仅输出 JSON
```

### Layer 2 → Layer 3

```
输入：Requirement 对象
输出：Intent 对象
      {metrics_requested[{registered_name, fuzzy_matched, user_name}],
       dimensions[{name, alias}], filters{date_range[]},
       group_by[], domain, confidence{}}
机制：精确匹配 → 模糊匹配 → 反问澄清（若全部失败）
```

### Layer 3 → Layer 4

```
输入：Intent 对象
输出：SQLPlan 对象
      {join_graph, column_bindings[], dimension_bindings[],
       filter_bindings[], group_by[], order_by[],
       execution_constraints, source_layer, domain}
机制：
  1. resolve_layer() → G3 优先，缺指标降级 G2
  2. determine_primary_table() → 查 metric_registry
  3. 检测 needs_dim_date（G2 fact 表日期列以 _key 结尾）
  4. build_join_graph() → 单表/跨域/dim_date
  5. build_column_bindings() → 查 ColumnBindingTable
  6. build_filter_bindings() → 参数化，G2 时切换到 dim_date.date
  7. build_execution_constraints() → 从 join_graph 推算 requires_date_dim
```

### Layer 4 → Layer 5

```
输入：SQLPlan 对象
输出：(sql_text, params[]) —— 完整 SQL 文本 + 参数化查询参数
机制：
  0. build_alias_map() → 多表 JOIN 时构建表名→别名映射
  1. compile_select() → 所有列经 _resolve_column_ref() 别名解析
  2. compile_from() → 单表无别名，多表有别名
  3. compile_where() → 经别名解析，参数化占位符
  4. compile_group_by() → G3 跳过（已预聚合），G2 使用别名
  5. compile_order_by() → 经别名解析
  6. compile_limit() → 可选
约束：所有标识符来自 ColumnBindingTable，绝对不拼接用户输入
```

#### 5.1 Layer 4 的核心工作原理：JSON → SQL 的字符串拼接

一个常见的问题：**LLM 不输出 SQL，只输出 JSON，复杂的 SQL 靠"字符串拼接"真的能实现吗？**

**答案是能。** `compile_sql()` 的本质就是将 SQLPlan IR（JSON 对象）序列化为 SQL 文本的**模板拼接器**。

**原理**：一条 SELECT 语句的各个组成部分天然可拆分，每个部分对应 SQLPlan JSON 中的一个字段：

| SQL 子句 | 来自 SQLPlan 的 JSON 字段 | 拼接方式 |
|---------|--------------------------|---------|
| `SELECT col1, col2` | `column_bindings[].alias` | `", ".join()` |
| `FROM schema.table AS t1` | `join_graph.primary.table` + `.alias` | f-string 模板 |
| `LEFT JOIN schema.t2 AS t2 ON ...` | `join_graph.joins[]` 每个元素 | 循环拼模板行 |
| `WHERE t1.date BETWEEN ? AND ?` | `filter_bindings[]` 每个元素 | 参数化占位符 `?` |
| `GROUP BY t1.date` | `group_by[]` | `", ".join()` |
| `ORDER BY t1.date ASC` | `order_by[]` | `", ".join()` + 方向 |
| `LIMIT 1000` | `limit` | f-string |

最终一行 `"\n".join()` 把所有片段拼成完整 SQL：

```python
sql_text = "\n".join([
    "SELECT",       select_clause,
    from_clause,    where_clause,
    group_clause,   order_clause,
    limit_clause
])
```

**为什么拼接是安全的？** 每个被拼入 SQL 字符串的片段，其来源都是确定性的——不是 LLM 生成的：

- **列名/表名** → ColumnBindingTable 查表（确定性映射）
- **JOIN 条件** → `sql_safety_policy.yml` 白名单（人工审核过的契约）
- **过滤值** → 参数化占位符 `?`（用户输入通过参数绑定，不拼入 SQL）
- **别名映射** → `alias_map`（从 JoinGraph 确定性推导）

**编译器不需要"理解" SQL 语义**——它只需要把 JSON 字段填进对应的模板位置。这就像填空题：模板是固定的，空格答案来自 JSON，拼出来就是标准 SQL。同样的 JSON 永远拼出同样的 SQL，这正是"确定性"的来源。

**即使是最复杂的场景**（窗口函数、CTE、增量 MERGE），本质仍然是 JSON 字段 → 模板字符串的机械拼接。编译器不需要"创作"任何东西——它只是忠实地把 IR 中的结构化信息序列化为目标数据库方言的文本。

### Layer 5 → Layer 6

```
输入：SQL 文本 + SQLPlan
输出：ValidationReport {passed, safety_checks[], table_checks[],
                         join_checks[], date_compliance[], issues[], warnings[]}
检查项：
  1. 禁止操作黑名单（INSERT/UPDATE/DELETE/DROP/TRUNCATE 等）
  2. 只读前缀检查（必须以 SELECT/WITH/EXPLAIN 等开头）
  3. 表引用合法性（禁止 bronze.* / silver.*）
  4. 完全限定名检查（所有表名必须 schema.table 格式）
  5. JOIN 白名单检查（跨表 JOIN 必须在核准列表中）
  6. 日期合规检查（G2 必须 JOIN dim_date，通过 requires_date_dim 验证）
```

### Layer 6 → Layer 7

```
输入：已验证的 SQL + params
输出：ExecutionResult {success, dataframe, row_count, column_count,
                         execution_time_ms, error_message}
约束：DuckDB read_only=True，超时 30s，禁止多语句，参数化查询
```

### Layer 7 → Layer 8

```
输入：DataFrame + SQLPlan（预期约束）
输出：EvaluationReport {checks[{name, passed, detail}], status, warnings[]}
检查项：
  - 行数范围（max_result_rows）
  - 空值率（< 30%）
  - 列完整性（所有预期列存在）
  - 指标一致性（列数 ≥ metrics 数量）
```

### Layer 8

```
输入：DataFrame + EvaluationReport + SQLPlan
输出物：
  - generated/results/{plan_name}_{timestamp}.parquet
  - generated/reports/{plan_id}_report_{timestamp}.md
  - generated/tasks/{plan_id}_task_{timestamp}.yml
```

## 6. 失败策略

### 6.1 失败等级

| 等级 | 含义 | 行为 |
|------|------|------|
| **FAIL** | 硬约束违反，不可恢复 | 立即终止管道，不生成产出物 |
| **BLOCKED** | 需人工介入 | 暂停管道，输出阻塞报告 |
| **DIRTY** | 结果可能有问题 | 完成但标记 DIRTY，附加警告 |

### 6.2 每层失败映射

| 层 | FAIL | BLOCKED | DIRTY |
|----|------|---------|-------|
| Layer 1 | YAML 解析完全失败 | 必填字段缺失 | 可选字段缺失 |
| Layer 2 | 全部指标不匹配 | 指标歧义需反问 | 置信度=medium |
| Layer 3 | 所有指标无 G3/G2 | G2 表达式未注册 | — |
| Layer 4 | ColumnBindingTable 缺少必要绑定 | — | — |
| Layer 5 | 安全黑名单命中 | — | G2 降级警告 |
| Layer 6 | 执行超时/锁定 | DB 被占用 | 返回 0 行 |
| Layer 7 | 关键列缺失 | 行数异常 | 空值率超阈值 |
| Layer 8 | 文件写入失败 | 磁盘不足 | — |

### 6.3 重试与降级

- **重试**：仅 Layer 6 因超时/锁失败时重试 1 次（间隔 5s）
- **G3→G2 降级**：G3 汇总表不含所需指标 → 自动降级到 G2 fact 表
- **G2→BLOCKED**：G2 也无可用的表达式 → 阻塞，告知用户
- **跨域→BLOCKED**：跨表 JOIN 不在白名单 → 阻塞

### 9.2 编译器架构模式统一

所有编译器共享相同的架构基因：

```
IR dataclass → 注册表分发 {dialect: {type: handler}} → SQL 字符串
                 └── 未注册 → fallback 到 DuckDB
```

| 编译器 | 注册表键 | 粒度 |
|--------|---------|------|
| P3 表达式 | `{dialect: {ExpressionType: handler}}` | 每种表达式类型独立函数 |
| P4 窗口函数 | `{dialect: {WindowFunctionName: handler}}` | 默认统一模板（12 种共享） |
| P5 CTE | 无需注册表（所有方言 WITH 语法一致） | 回调模式避免循环导入 |
| P6+P7 操作 | `{dialect: {ExecutionStrategy: handler}}` | 每种策略独立函数 |

### 9.3 方言差异矩阵

| 功能 | DuckDB | Hive | PostgreSQL |
|------|--------|------|------------|
| 日期差 | `DATEDIFF('day', s, e)` | `DATEDIFF(e, s)` | `(e - s)` |
| 日期截断 | `DATE_TRUNC('month', col)` | `TRUNC(col, 'month')` | fallback DuckDB |
| 日期格式化 | `STRFTIME(col, fmt)` | `DATE_FORMAT(col, fmt)` | `TO_CHAR(col, fmt)` |
| 类型转换 | `CAST(x AS INT)` | fallback DuckDB | `x::int` |
| 字符串拼接 | `CONCAT(a, b)` | fallback DuckDB | `a \|\| b` |
| 全量覆盖 | `CREATE OR REPLACE TABLE AS` | `INSERT OVERWRITE TABLE` | `BEGIN; DELETE; INSERT; COMMIT` |
| 分区覆盖 | 降级为全量覆盖 | `INSERT OVERWRITE ... PARTITION (col)` | 降级为全量覆盖 |
| 分区追加 | `INSERT INTO` | `INSERT INTO ... PARTITION (col)` | `INSERT INTO` |
| 视图创建 | `CREATE OR REPLACE VIEW` | `CREATE VIEW` | `CREATE OR REPLACE VIEW` |

### 9.5 关键架构决策（P3-P7）

| 决策 | 阶段 | 内容 |
|------|------|------|
| 注册表模式 | P3 | `{方言: {类型: handler}}` 二级字典替代 if/elif 链 |
| 两遍编译 | P3 | Pass 1 构建 alias 索引 → Pass 2 递归展开（深度上限 10） |
| 回调模式 | P5 | `compile_cte_clause(defs, dialect, compile_sql_fn)` 打破循环导入 |
| C4 递归约束 | P5 | CTE 体内不允许嵌套 CTE——编译时硬阻断 |
| 策略降级 | P7 | INSERT_OVERWRITE 缺分区列 → 自动降级为 FULL_OVERWRITE |

### 9.6 安全审计——编译器安全加固（2026-06-14）

在 P3-P7 全家福交付后，对全部代码进行安全审计，发现并修复 3 个 P0、3 个 P1、2 个 P2 问题：

#### P0 级修复（严重安全缺陷——可导致错误 SQL 通过校验）

| 问题 | 位置 | 缺陷 | 修复 |
|------|------|------|------|
| **P0-1 G2 日期过滤违规** | `layer3_plan.py:839` | `_is_date_key_column()` 后缀启发式不可靠；`requires_date_dim` 依赖 `join_graph` 内容形成假阴性校验 | 废除启发式，改为硬安全门禁：G2+日期过滤→强制 `needs_dim_date=True`，缺 JOIN 路径→拒绝编译（`is_valid=False`） |
| **P0-2 跨域 JOIN 表顺序不稳定** | `layer3_plan.py:647` | `set()` 迭代顺序非确定性，多表场景下表选择可能漂移 | `set()` → `sorted(tables)` 字母序确定性排序 |
| **P0-3 多表别名列引用** | `layer4_generate.py:69` | `_resolve_column_ref()` 无失败检测，未替换的 `schema.table.` 前缀在 DuckDB 运行时崩溃 | 添加防御性校验：别名模式下残留全限定表名→抛出 `SQLCompileError` |

#### P1 级修复（高优先级缺陷——环境依赖/事实源漂移）

| 问题 | 位置 | 缺陷 | 修复 |
|------|------|------|------|
| **P1-1 ColumnBinding 硬编码** | `column_binding.py:68` | 10 个指标静态写死，无加载来源追踪 | 新增 `get_load_status()` 诊断函数，追踪 `tianShu`/`static_fallback`/`not_attempted` 三种状态 |
| **P1-2 Windows CLI 编码崩溃** | `run_pipeline.py:90` | GBK 控制台 `print()` 触发 `UnicodeEncodeError` | 模块加载时自动 `_wrap_stdio_utf8()`（stdout+stderr），仅在 `PYTHONIOENCODING` 未设置时生效 |

#### P2 级修复（基础设施加固）

| 问题 | 位置 | 缺陷 | 修复 |
|------|------|------|------|
| **P2-1 Quality check 覆盖太浅** | `check_pipeline.py` | 仅 7 项检查（依赖/目录/静态绑定数）——不覆盖安全路径 | 新增 5 项：G2 dim_date JOIN 覆盖、JOIN 白名单完整性、合约结构验证、SQL 编译 dry-run、ColumnBinding 加载状态 |
| **P2-2 缺少 .gitignore** | 项目根目录 | `generated/` 运行时产物可被误提交 | 新建 `.gitignore`：排除 `generated/results/`、`generated/reports/`、`__pycache__/` 等 |

#### 安全审计原则

| 原则 | 说明 |
|------|------|
| **默认拒绝** | 不确定时不猜测——拒绝执行比执行错误好 |
| **确定性优先** | 相同输入必须产生相同输出——禁止非确定性集合 |
| **失败可检测** | 静默失败是最危险的失败——必须抛出明确错误 |
| **状态可追踪** | 关键运行状态必须可查询——不能假定一切正常 |
| **环境无关** | 不依赖用户环境配置 |
| **检查即门禁** | 自检脚本覆盖关键安全路径 |
