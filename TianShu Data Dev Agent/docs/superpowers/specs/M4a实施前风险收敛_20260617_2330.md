## 3. SUPERSEDED 是否应推迟到 M4b

### 关键冲突

```
当前测试：
  test_verify_does_not_modify_decision_file
    → assert before == after（decision.md 在 M3 前后逐字节一致）

SUPERSEDED 逻辑：
  if decision.yml.current_state == APPROVED:
      decision.yml.current_state = SUPERSEDED
      decision_log.yml.append(...)
```

### 分析

`test_verify_does_not_modify_decision_file` 当前断言 `decision.md` 在 M3 前后逐字节一致。这个测试的**设计意图**是：Agent 不能偷偷把 decision 从 REQUEST_CHANGES 改成 APPROVE。

如果 M3 修改的是 `decision.yml`（新文件）而非 `decision.md`，这个测试**仍然通过**——它只检查 `decision.md` 是否被修改。

SUPERSEDED 写入的是 `decision.yml` + `decision_log.yml`——**两个新文件，不是 `decision.md`**。

### 结论：SUPERSEDED **可以且应该**纳入 M4a

理由：
1. 不破坏 `test_verify_does_not_modify_decision_file`——该测试只检查 `decision.md`
2. SUPERSEDED 是 M3 的核心职责："重新验证后旧批准自动失效"——没有它，M4a 只是格式变化，不提供实质价值
3. 新测试覆盖 SUPERSEDED 路径即可，不与旧测试冲突
4. 推迟到 M4b 意味着 M4a 交付后，M3 重新验证时**旧 APPROVED 状态仍然有效**——这是安全漏洞

### 需要调整的测试

|------|------|
| `test_verify_does_not_modify_decision_file` | 不变——仍断言 `decision.md` 不被修改 |
| `test_decision_remains_request_changes_after_verify` | 修改——从检查 `decision.md` 文本 → 检查 `decision.yml` 的 `current_state` |

## 4. 是否应新增 `reports/verification_summary.yml`

### 当前问题

`verification.md` 是 Markdown 自由文本——当前唯一持久化的验证结果是 `VerificationEngineResult` dataclass 的字段，但这些字段只存在于 Python 内存中，不落盘。CLI（`verify_review_package.py`）打印 `overall_status` 到 stdout 后就丢失了。

未来 M4b CLI `set APPROVED` 需要检查 `overall_status != FAIL`。如果只能解析 `verification.md` 文本，存在以下风险：

| 风险 | 严重度 |
|------|--------|
| 人手动编辑 verification.md 后，程序解析到错误状态 | 🟡 中——人可能手误改了状态行 |
| Markdown 格式不稳定——未来改报告格式会导致解析逻辑失效 | 🟡 中——`_build_verification_report()` 是内部函数，格式可能随需求变化 |
| 正则/字符串解析脆弱——`总体状态：PASS` 的匹配模式依赖中文冒号 | 🟢 低——但仍是隐式契约 |

### 使用 YAML summary 的好处

```yaml
# reports/verification_summary.yml（M3 写入，人不应编辑）
overall_status: "PASS"         # PASS | WARN | FAIL | PENDING | SKIPPED
sql_static_status: "PASS"
sql_sample_status: "PASS"
spark_static_status: "PASS"
spark_sample_status: "SKIPPED"
cross_validation_status: "SKIPPED"
failures: []
warnings:
  - "Spark sample run 状态为 SKIPPED"
  - "交叉验证状态为 SKIPPED"
verified_at: "2026-06-17T15:00:00+08:00"
m3_version: "v2.0"
```

| 优势 | 说明 |
|------|------|
| **结构化校验** | CLI 直接 `yaml.safe_load()` 读取 `overall_status`，无需解析 Markdown |
| **人不应编辑** | 文件头注释"此文件由 M3 自动生成，请勿手动编辑" |
| **格式稳定** | 无论 `verification.md` Markdown 怎么变，`verification_summary.yml` 的 schema 不变 |
| **零额外成本** | `verify_review_package()` 已经计算了所有状态值——只需在写入 `verification.md` 的同时写入一份 YAML |

### 建议：M4a 纳入 `verification_summary.yml`

这是最小的增量——`verify_review_package()` 已经持有所有数据（`VerificationEngineResult`），只需多写一个文件。没有它，M4b CLI `set APPROVED` 的前置校验就缺少结构化数据源。推迟到 M4b 会导致 M4b 要同时实现 YAML summary + CLI，增加 M4b 的复杂度。


**`generated/review_packages/` 已在 git 中，且 .gitignore 未忽略它。**

### 5.2 如果只想提交 decision.yml / decision_log.yml

技术上可行：

```gitignore
# 忽略 Review Package 的运行产物
generated/review_packages/*/sql/
generated/review_packages/*/spark/
generated/review_packages/*/tests/
generated/review_packages/*/reports/

generated/review_packages/*/decision.md

# 但保留决策文件
!generated/review_packages/*/decision.yml
!generated/review_packages/*/decision_log.yml
```

**问题**：这种模式脆弱——每新增一个需忽略的子目录就要加一行。且 `decision.md` 是人在上面写审查意见的，忽略它意味着人的审查意见不入版本控制。

### 5.3 建议：整体忽略 Review Package，用 golden fixture 替代

```
方案：
  1. .gitignore 新增 generated/review_packages/
  2. 保留 tests/fixtures/review_package_golden/ 目录
     （提交 M2 生成的 golden 基线——不含人审决策，仅做格式验证）
  3. 真正的 decision.yml / decision_log.yml 在开发者本地生成
```

**但**：如果 `decision.yml` 和 `decision_log.yml` 要入 git（作为审计记录），这个方案不适用——因为本地生成的审计日志无法通过 golden fixture 替代。

### 5.4 最终建议：分两步走

| 阶段 | 操作 | 说明 |
|------|------|------|
| **M4a** | 保持现状——`generated/review_packages/` 继续全部跟踪 | 避免 M4a 范围膨胀。`trip_daily_report_m2` 是唯一一个 Review Package，它的 `decision.yml` 和 `decision_log.yml` 直接提交即可 |
| **M4b** | 重新评估——如果 Review Package 数量增长，改为 golden fixture 模式 | 届时 `review_packages/` 整体忽略，`tests/fixtures/` 保留 golden 基线 |

**M4a 不做 gitignore 变更。** 当前 `trip_daily_report_m2` 的 7 个文件已在 git 中，新增 2 个文件（`decision.yml` + `decision_log.yml`）直接提交。

## 附录：关键设计决策记录

### A. 为什么 decision.md 保留 "REQUEST_CHANGES" 作为当前结论文案

`decision.md` 是给人看的第一印象。"PENDING_REVIEW" 是状态机的技术用语，对人不直观。**当前结论** 文案保持 "当前不是 APPROVE，需要人工审查"——决策选项列表增加 SUPERSEDED 即可。结构化状态在 `decision.yml` 中为 `PENDING_REVIEW`。

### B. 为什么 M4a 就引入 SUPERSEDED

没有 SUPERSEDED 的 M4a 是一个"有状态但无保障"的系统——人可以 APPROVE，但 M3 重新验证后旧批准依然有效。这会创造虚假安全感。SUPERSEDED 是状态机唯一由 Agent 触发的保护性流转，是 M4a 提供的实质安全价值。

### C. verification_summary.yml 不替代 verification.md

`verification.md` 是给人看的完整报告——包含检查明细、WARN/FAIL 列表、人审提示。`verification_summary.yml` 只包含程序校验需要的结构化字段。两者互补，不互相替代。
