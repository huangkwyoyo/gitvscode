## 1. 设计动机：为什么角色要变

### 1.1 旧架构的根本问题

旧架构（v1.x）把 Data Dev Agent 设计为**确定性数据生产管道**：

```
人提 YAML → Agent 自动跑 8 层管道 → 直接产出数据文件
```

这个设计的隐含假设是：**管道自动产出 = 上线**。既然没有人审查，LLM 就不能参与代码生成——它只能输出"指标名 + 维度名 + 过滤值"，SQL 由模板编译器确定性拼接。

这个假设的问题在于：**数据开发团队从来不会让 Agent 自动部署代码到生产。** 生产代码必须经过 code review，最终上线决定始终在人。

### 1.2 新架构的核心权衡

既然有人审查，安全模型就不该基于"LLM 不可靠所以要隔离它"——而该基于"LLM 写的代码需要人审查才能上线"。

**前者在防止风险的同时牺牲了 LLM 的价值。后者在释放 LLM 价值的同时用人审管控风险。**

```
旧权衡：安全 ← 隔离 LLM → 牺牲 LLM 写代码的价值
新权衡：价值 ← 释放 LLM → 三道防线管控风险 → 安全
```

### 1.3 角色转变

| | 旧架构 | 新架构 |
|------|--------|--------|
| **Agent 角色** | 生产执行者——自动跑管道、产出数据 | 代码生成者 + 自测者——写代码、跑测试、出报告 |
| **人的角色** | 旁观者——提需求、收结果 | 最终决策者——审查代码、拍板上线 |
| **Agent 的权限** | 执行管道（设计意图含写 generated.*，但 Layer 6 实际为 read_only=True） | 写代码 + 读开发库（验证用） |
| **人的权限** | 提需求 | 审查 + 拍板 + 部署到生产 |

## 2. 架构总览

```
═══════════════════ Agent 的工作区 ═══════════════════

  需求输入（YAML 或自然语言）
    ↓
  [阶段 1] 需求分析      Agent 解析需求、匹配指标
    ↓
  [阶段 2] 方案设计      Agent 选表、选层级、设计 JOIN
    ↓
  [阶段 3] 代码生成      Agent 写 SQL/Python/测试/配置（标注来源）
    ↓
  ═══════════ Agent 的自测边界 ═══════════
    ↓
  [阶段 4] 自动验证      规则引擎：6 项检查 + 样本执行
    ↓
  [阶段 5] 材料输出      整合代码 + 测试 + 决策 + 不确定项

═══════════════════ 人的决策区 ═══════════════════

  [阶段 6] 人审上线      人审查 → 批准 / 修改 / 拒绝
    ↓
  人部署到生产
```

**关键分界**：
- 阶段 3-4 之间：LLM 参与到此为止。自动验证是确定性代码，不调 LLM。
- 阶段 5-6 之间：Agent 工作到此为止。之后的决定在人。

## 3. 安全模型：三道防线

每道防线解决不同层面的风险，三道互补。

```
防线 1：生成时约束（Agent 自控层 / fail-fast）
  定位：防御性 fail-fast 检查，减少明显危险草案进入后续流程。
        不是安全信任边界——生成端检查通过不代表代码安全、已验证或可执行。
  解决：减少幻觉进入代码
  机制：LLM 只能引用已注册的表/字段，不确定必须标注
        dual_code_generator 调用共享 spark_safety 分析器做 fail-fast 拦截
  失效时：防线 2 的表存在性检查会拦截编造的表名

防线 2：自动验证（规则引擎层 / 权威安全检查）
  定位：任何代码进入 sample run 前的唯一权威安全检查。
        Validator（checker.py）是安全信任边界——和生成端共享同一
        spark_safety 规则事实源，但 Validator 的裁决才是最终安全结论。
  解决：拦截确定性的低级错误
  机制：7 项规则检查（表存在性、安全黑名单、权限、白名单、样本执行、质量）
        + SQL/Spark 交叉验证 + AST-based Spark 安全分析
  失效时：防线 3 的人审会发现逻辑错误和方案问题

防线 3：人审闸门（人的判断层）
  解决：自动验证覆盖不了的业务判断
  机制：人审查全部材料，逐条确认不确定项，最终拍板
  失效时：这是最后一道防线——人的审查也可能不完美，但来源追溯降低了审查难度
```

**重要边界声明（2026-06-19 新增）**：三道防线用于降低风险，不构成上线充分条件。当前验证为 PARTIAL 级别——仅 SQL 单引擎样本执行（LIMIT 1000），Spark 侧为 SKIPPED/PENDING。即使未来实现 DUAL_ENGINE_SAMPLE（双引擎样本一致），也只代表样本集一致，不代表全量数据一致、业务正确或生产就绪。APPROVED = 人审通过，QUERY_LOGIC_APPROVED——不是自动化正确性证明。上线仍需人独立评估部署风险。

## 4. 关键设计决策

### 决策 1：角色转变——从执行者到生成者+自测者
Agent 不再自动执行生产管道，而是生成代码并自测，由人决定上线。这解决了旧架构的根本矛盾——在一个"最终上线决定在人"的场景里，用了"无人审查"的安全模型。

### 决策 2：三道防线代替单一硬门禁
单靠任一道防线都不够。三道互补——prompt 约束减少幻觉，自动验证拦截低级错误，人审处理复杂判断。防线之间的"失效传递"是设计的一部分：如果防线 1 漏了，防线 2 能接住；防线 2 漏了，防线 3 能接住。

### 决策 3：来源追溯作为人审效率的杠杆
这是降低人审成本最关键的设计。没有来源追溯，审查者需要自己查文档——审查 100 行 SQL 可能需要 20 分钟。有来源追溯，只需验证"注释中的来源是否正确"——同样的 SQL 可能只需 5 分钟。效率提升 4 倍意味着人可以审查更多代码，或同样的时间审得更细。

### 决策 4：保留旧的确定性代码作为防线 2
`scripts/pipeline/` 中的 135 个测试覆盖的确定性代码不删除——它们成为防线 2（自动验证）的规则引擎。LLM 生成的代码可能随着模型版本变化而变化，但自动验证的行为始终确定。这保证了安全底线的稳定性。

### 5.4 关键约束

- LLM 产物即使未来接入，也只能是**不可信草案**
- 所有代码必须经过 Validator → sample run → 人审
- **不能自动 APPROVE**——Agent 只能写入 PENDING_REVIEW 和 SUPERSEDED
- **不能把 PENDING/SKIPPED 写成 PASS**
- **交叉验证仅证明样本一致**——不证明全量或生产行为一致
- v1 pipeline 不删除、不重构
