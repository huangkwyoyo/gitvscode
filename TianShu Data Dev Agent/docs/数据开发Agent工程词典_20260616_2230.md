## §1 核心概念：角色转变与双层边界

### 1. 旧角色 vs 新角色

这是理解整个 v2.0 架构的入口概念。

**旧架构（v1.x）的角色分配**：
```
人：提需求 YAML → 等结果
Agent：自动执行 8 层管道 → 产出数据文件
```
Agent 是**生产执行者**——它自动跑管道，人只需提交需求和接收结果。人是**旁观者**——管道在跑，人在等，出了问题才介入。

**新架构（v2.0）的角色分配**：
```
Agent：分析需求 → 写 SQL + Spark DSL 两份代码 → 跑测试 → 交叉验证 → 出报告
人：审查代码和测试结果 → 拍板决定是否上线
```
Agent 是**代码生成者 + 自测者**——它写代码、跑测试、出报告，但不替人做上线决定。人是**最终决策者**——审查代码、确认测试结果、拍板上不上线。

### 2. 为什么角色要变

旧架构的设计假设是"无人值守自动生产"——在这个假设下，禁止 LLM 输出代码是正确的（没人审查，LLM 的错误直接进生产）。

但这个假设本身是错的。数据开发的现实是：**从来没有一个团队会让 Agent 自动把代码部署到生产。** 生产代码必须经过 code review，最终上线决定始终在人。

既然如此，安全模型就不该基于"LLM 不可靠所以要隔离它"——而该基于"LLM 写的代码要经过人审查才能上线"。前者在防止风险的同时牺牲了 LLM 的价值（LLM 只能输出"指标名+维度名"，不能写代码），后者在释放 LLM 价值的同时用人审管控风险。

**核心权衡**：松开 LLM 的手（让它写代码），加四道保险（来源追溯 + 自动验证 + 交叉验证 + 人审闸门）。

### 3. 角色对比表

| 维度 | 旧架构 Agent | 新架构 Agent |
|------|------------|------------|
| **角色** | 生产执行者 | 代码生成者 + 自测者 |
| **做什么** | 自动执行 8 层管道 | 分析需求 → 生成双份代码（SQL + Spark DSL）→ 自动验证 + 交叉验证 → 输出审查材料 |
| **代码来源** | 模板编译器确定性拼接 | LLM 生成草案，来源可追溯 |
| **决策权** | 管道自动决策 | LLM 提议方案，人确认 |
| **产出物** | 数据文件（Parquet/CSV） | SQL + Spark DSL（PySpark）+ 交叉验证报告 + 测试结果 + 决策说明 |
| **上线** | 管道产出 = 上线 | 人审查通过 ≠ 上线，人还需执行部署 |
| **验证机制** | 规则引擎单项校验 | 规则引擎 7 项检查 + SQL/Spark 结果交叉验证 |

| 维度 | 旧架构中的人 | 新架构中的人 |
|------|-----------|-----------|
| **角色** | 旁观者 | 最终决策者 |
| **做什么** | 提需求、收结果 | 审查双份代码、确认交叉验证、拍板上线 |
| **什么时候介入** | 出问题时 | 每次上线前 |
| **需要什么能力** | 能描述需求即可 | 需要审查代码、理解设计决策 |

### 4. 双层边界（Double Boundary）

**是什么**：v2.0 的核心安全设计。将 Agent 的权限分为两层——**代码生成边界**（宽松，LLM 自由发挥）和**数据执行边界**（严格，唯一校验通道）。两层之间是防线 2（自动验证）——代码进入执行前必须穿过的唯一通道。

```
══════════════════════════════════════════════
边界 1：代码生成（宽松）
  LLM 可以生成 SQL、Spark DSL、Python、YAML、测试
  所有产物 = "不可信补丁"
══════════════════════════════════════════════
          ↓ 防线 2 自动验证（唯一通道）↓
══════════════════════════════════════════════
边界 2：数据执行（严格）
  无论代码谁写的，都必须走 Validator → Executor
  硬禁止：绕过校验、访问生产库、生成非只读 SQL
══════════════════════════════════════════════
```

**为什么需要双层边界**：旧架构因为 LLM 不生成代码，所以只需一条"执行边界"。v2.0 中 LLM 大量生成代码，必须区分"生成代码"和"执行代码"两个风险层面——在生成端释放 LLM 能力（边界 1 宽松），在执行端收紧管控（边界 2 严格）。**安全的信任边界在 Validator（数据执行端），生成端只做 fail-fast 防御性检查——生成端检查通过不代表代码安全、已验证或可执行。**

### 5. 代码生成边界（Code Generation Boundary）

**是什么**：双层边界的第一层。LLM 可以生成 SQL 草案、Spark DSL（PySpark）草案、Python 管道、YAML contracts、测试用例、文档配置等一切代码产物。

**核心规则**：所有产物一律视为"不可信补丁"——在通过防线 2 验证之前，不得落地执行。

**为什么边界 1 宽松**：LLM 的价值在于写代码。如果连生成都禁止，就等于回到 v1.x 的"LLM 只输出结构化意图"——浪费了 LLM 最大的能力。宽松的生成边界 + 严格的执行边界，比"完全禁止生成"更实用。

### 6. 数据执行边界（Data Execution Boundary）

**是什么**：双层边界的第二层。无论代码来源（LLM 生成、模板编译器、人手写），只要进入执行，必须走唯一路径：

```
SQL/PySpark 草案 → Validator（7 项检查）→ Executor（dev 只读，LIMIT 1000）→ 结果
```

**硬禁止**：
- 绕过 Validator 直接执行任何 SQL
- 绕过 Executor 直接查 DuckDB
- 访问生产数据库
- 生成非只读 SQL（INSERT/UPDATE/DELETE/DROP 等）

**为什么边界 2 严格**：数据执行的后果不可逆。生产库的误操作可能造成数据损失。宽松的生成边界意味着更严格的执行边界——越放手让 LLM 生成，执行端的门禁就要越紧。

### 7. 不可信补丁（Untrusted Patch）

**是什么**：LLM 生成的所有代码的初始状态标签。

**不可信补丁的含义**：
- 代码逻辑未经任何验证——可能正确，也可能含幻觉
- 引用的表和字段可能不存在
- JOIN 条件可能不合理
- 需要经过完整验证链路才能升格为"待审代码"

**验证链路**：不可信补丁 → 静态检查 → 单元测试 → 样本执行 → 交叉验证 → 防线 2 全部通过 → 升格为待审代码 → 进入人审。

**为什么需要这个标签**：防止"LLM 写的代码看起来很对所以直接执行"的心理陷阱。所有 LLM 产物默认不可信，必须验证——这是 Agent 和人的共同认知基础。

## §2 来源追溯

### 8. 来源追溯（Source Tracing）

**是什么**：Agent 生成的代码中，每个表名、字段名、JOIN 条件都必须附带来源注释——指向具体的数据库设计文档章节或元数据条目。

**为什么是角色转变的关键支撑**：没有来源追溯，人审时需要自己查文档来验证每个表名是否正确——这是"让审查者帮 Agent 做 Agent 该做的事"。有来源追溯，人审时只需验证"注释中的来源是否正确"——同样时间能审查更多代码、发现更多问题。

**格式**：
```python
# 来源：gold_database_design.md §3.2 dws_daily_trip_summary 表定义
TRIP_TABLE = "gold.dws_daily_trip_summary"
```

**三个作用**：
1. **降低人审成本**：从"自己查文档验证"变为"验证来源标注"
2. **暴露 LLM 幻觉**：如果 Agent 标注的来源章节不存在或内容不匹配，人一眼就能发现
3. **代码自文档化**：后人接手时能快速找到设计依据

## §3 Agent 生成代码的参考数据源

### 9. ColumnBindingTable（列绑定表）

**是什么**：指标名→物理列的映射表。在 v2.0 中，角色从"编译器的唯一数据源"变为"Agent 生成代码时的参考手册 + 自动验证时的校验基准"。

**当前注册 10 个指标，4 个业务域**：traffic（4 个）、violation（2 个）、safety（3 个）、supply（1 个）。

**新角色**：Agent 在阶段 2（方案设计）查此表，确定每个指标的 G3/G2 可用性。在阶段 4（自动验证）用此表校验 Agent 生成的代码中引用的列是否真实存在。

### 10. G3 / G2 / dim_date JOIN

**是什么**：数据源层级。G3 是预聚合汇总表（优先），G2 是明细事实表（降级）。G2 的日期列是整数代理键，必须通过 `dim_date` 表过滤日期。

**新角色**：Agent 在设计方案时选择 G3 还是 G2（阶段 2），并在决策说明中解释理由。自动验证（阶段 4）检查 Agent 是否正确处理了 G2 的 dim_date JOIN 契约。

### 11. JOIN 白名单

**是什么**：已审批的表关联路径清单。定义了哪些表之间可以 JOIN，以什么条件 JOIN。

**新角色**：Agent 在方案设计时以此为可选菜单。自动验证时检查 Agent 生成的 JOIN 是否在白名单中。不在白名单的 JOIN 标记为"需审批"——不确定项进入人审。

## §4 自动验证（Agent 的自测能力）

### 12. 自动验证

**是什么**：Agent 在生成代码后自动执行的 **7 项规则检查 + 1 项交叉验证**。纯确定性代码，不调 LLM。

**7 项规则检查**：

| # | 检查项 | 失败等级 | 说明 |
|---|--------|---------|------|
| 1 | 表/字段存在性 | FAIL | 对开发库执行 DESCRIBE，确认引用的表和列真实存在 |
| 2 | 安全关键字黑名单 | FAIL | 检测 INSERT/UPDATE/DELETE/DROP/TRUNCATE/ALTER 等 16 个关键字 |
| 3 | 表访问权限 | FAIL | 禁止 bronze.* / silver.* / 未注册表 |
| 4 | JOIN 白名单合规 | FAIL/UNCERTAIN | 在白名单中→通过；不在→标记"需审批" |
| 5 | 样本执行（SQL） | FAIL | 开发库执行 SQL（LIMIT 1000），检查是否报错 |
| 6 | 结果质量 | WARN | 空值率 >30%、行数 = 0、列缺失→标注警告 |
| 7 | **SQL vs Spark DSL 交叉验证** | WARN | 两份代码分别执行，对比结果集——不一致时人审调查 |

**为什么需要**：一个只生成代码但不测试的 Agent，等于把验证工作转移给了人。自动验证让 Agent 在交给人之前先拦截低级错误，人审时可以专注于业务逻辑——这才是人最擅长的。

### 13. 安全黑名单

**是什么**：16 个永久禁止的 SQL 关键字（INSERT/UPDATE/DELETE/DROP 等）。自动验证阶段 4 的检查项 #2。

**为什么仍然需要**：即使有来源追溯和人审，黑名单仍是高效的自动化安全网——正则扫描秒级完成，零遗漏。

### 14. 参数化查询

**是什么**：用户输入值通过 `?` 占位符绑定，不直接拼入 SQL。

**为什么仍然需要**：防止 SQL 注入是基础安全要求，无论 SQL 是谁生成的。

### 15. 交叉验证（Cross-Validation）

**是什么**：防线 2 的检查项 #7。对同一需求，LLM **独立生成** SQL（DuckDB）和 Spark DSL（PySpark）两份代码草案，分别对 dev 库执行，然后对比结果集。

```
同一需求
  ├─ SQL 草案（DuckDB）──→ dev 库执行 ──→ 结果集 A
  └─ Spark DSL（PySpark）──→ dev 库执行 ──→ 结果集 B
                                            ↓
                          对比：行数、列名、值范围、抽样行
                                            ↓
                    ┌─ 一致（在容差内）→ 通过，样本集逻辑一致
                    └─ 不一致          → WARN，人审时定位差异原因
```

**为什么有效**：SQL 和 PySpark 使用不同的表达方式、执行引擎和类型系统。两份独立代码很难犯完全相同的逻辑错误。这是一种"双重记账"式校验——不是检查代码写得对不对，而是检查两份独立代码是否算出了相同答案。

**三种结果**：
- **一致**（行数相等、列名匹配、值分布在容差内）→ 两份代码在样本集（LIMIT 1000）上逻辑一致。注意：样本一致不代表全量数据一致、业务正确或生产就绪。交叉验证报告标注"一致"
- **不一致**（行数不匹配、列名不同、值差异显著）→ 至少一份代码有逻辑错误，人审时必须确定哪份是对的
- **仅 SQL 执行**（Spark 环境暂时不可用）→ Spark DSL 仍生成和审查，交叉验证标记"待 Spark 环境执行"

### 16. Spark DSL（PySpark）

**是什么**：Agent 为同一需求生成的第二份代码实现。用 PySpark（Python API）编写，与 SQL 版本逻辑等价但实现独立。

**与 SQL 版本的关系**：
- **相同**：同一需求、同一数据源、同一业务逻辑
- **不同**：表达方式（声明式 SQL vs 过程式 PySpark）、执行引擎（DuckDB vs Spark）、类型系统
- **独立生成**：LLM 分别生成两份代码，不能互相复制——这是交叉验证有效的前提

**代码示例**（PySpark）：
```python
# 来源：gold_database_design.md §3.2 dws_daily_trip_summary 表定义
# 指标来源：meta.metric_definitions — trip_count, total_fare_amount
from pyspark.sql import SparkSession
from pyspark.sql.functions import col

df = (
    spark.table("gold.dws_daily_trip_summary")
    .select("trip_date", "trip_count", "total_fare_amount")
    .filter(col("trip_date").between(start_date, end_date))
    .orderBy("trip_date")
)
```

## §5 人审闸门（人的决策权）

### 17. 人审闸门

**是什么**：工作流的最后阶段（阶段 6）。人审查 Agent 的全部产出，做出最终决策。

**为什么是三道防线的最后一道**：自动验证覆盖的是"确定性的低级错误"（语法、表存在性、安全违规）。它覆盖不了的是"需要业务判断的高层决策"（方案是否合理、指标映射是否准确）——这靠人的判断力。

**交叉验证如何辅助人审**：
- 两份代码结果一致 → 人审时逻辑层面的顾虑大幅降低，重点审查业务合理性
- 两份代码结果不一致 → 人审时必须逐项对比差异，判断哪份正确（或两份都错）
- 差异报告是人的"调查线索"——不需要逐行 debug 代码，先看差异在哪

### 18. 审查材料

**是什么**：Agent 在阶段 5 整合的输出包，包括需求分析报告、设计方案、**双份代码（SQL + PySpark，含来源标注）**、自动验证报告（含 7 项检查结果）、**交叉验证报告**、不确定项清单。

**设计原则**：材料包的目的是让人的审查决策有据可依——不是只看代码，而是看到完整的"需求→方案→双份代码→两份测试结果→交叉验证"全链路。

### 19. 不确定项（UNCERTAIN）

**是什么**：Agent 在生成过程中标注的、自己无法确定的事项。例如：指标名有多个候选、JOIN 路径不在白名单中、交叉验证结果不一致。

**为什么是 Agent"诚实"的标志而非"失败"**：Agent 在不确定时主动标注，比假装确定生成错误代码要好。不确定项是 Agent 和人之间的"对话接口"——Agent 说"这里我不确定，你来看"，人确认或纠正。

## §6 废弃概念

以下概念在旧架构中是核心，在 v2.0 中已废弃或降级：

| 旧概念 | 旧角色 | 为什么废弃 |
|--------|--------|-----------|
| 编译器优先设计 | SQL 由模板编译器从 IR 拼接 | LLM 现在直接生成 SQL。编译器代码保留作为 fallback |
| LLM 硬边界线 | L2 和 L3 之间的不可逾越线 | 替换为双层边界（代码生成 ← → 数据执行）。约束在执行端 |
| 8 层管道 | L1-L8 严格线性 | 简化为 6 阶段。旧代码保留为防线 2 规则引擎 + fallback 编译器（含写操作编译器 `compile_operation()`，可编译 CTAS/INSERT/VIEW，但当前未接入可写执行器） |
| 确定性保证 | 同样输入 100% 同样 SQL | 不再作为核心承诺。来源追溯 + 7 项自动验证 + 交叉验证 + 人审提供等效保障 |
| 单份代码验证 | 只生成 SQL，只验证 SQL | 替换为双份代码交叉验证——两份独立实现相互校验 |

## §7 快速命令

| 操作 | 命令 |
|------|------|
| 运行全部测试 | `python -m pytest tests/ -v` |
| 运行 dry-run | `python scripts/pipeline/run_pipeline.py -r fixtures/requirements/trip_daily_report.yml --dry-run` |
| 查看帮助 | `python scripts/pipeline/run_pipeline.py --help` |
