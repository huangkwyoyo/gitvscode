# Data Dev Agent 工作流

> 版本：v2.0 | 日期：2026-06-16
>
> **核心思想**：Agent 是代码生成者 + 自测者，人是最终决策者。本文档说明每个阶段 Agent 做什么、人做什么，以及为什么这样分工。

---

## 一、角色分工：一条线分两边

整个工作流被一条清晰的线分成两个区域：

```
═══════════════════ Agent 的工作区 ═══════════════════

  阶段 1：需求分析      Agent 解析需求、匹配指标
  阶段 2：方案设计      Agent 选表、选层级、设计 JOIN
  阶段 3：代码生成      Agent 生成两份代码：SQL（DuckDB）+ Spark DSL（PySpark）
  阶段 4：自动验证      Agent 跑 7 项规则引擎检查 + SQL/Spark 交叉验证
  阶段 5：材料输出      Agent 整合双份代码 + 测试 + 交叉验证报告 + 决策说明

═══════════════════ 人的决策区 ═══════════════════

  阶段 6：人审上线      人审查所有材料（含交叉验证报告）→ 拍板决定
```

**Agent 不越线**——它不会替人做上线决定。
**人不越线**——人不需要帮 Agent 写代码或跑测试。

这条线是 v2.0 架构最核心的设计：把"写代码 + 验证"交给 Agent，把"判断 + 决策"留给人。

## 二、逐阶段详解：Agent 做什么，人看什么

### 阶段 1：需求分析

| | 内容 |
|------|------|
| **Agent 做什么** | 解析输入（YAML 或自然语言）→ 识别指标/维度/过滤条件 → 查已注册指标列表匹配 → 确认数据可用性 |
| **Agent 输出什么** | 需求分析报告：`{指标匹配结果, 匹配置信度, 数据可用性, 不确定项}` |
| **人看什么** | 指标映射是否正确？有没有漏掉的需求？不确定项是否合理？ |
| **常见问题** | "这个指标名我理解成 A，对吗？"——Agent 不确定时会主动标注 |

**Agent 的决策逻辑**：
```
指标名在已注册列表中？
  ├─ 精确匹配 → 置信度 high
  ├─ LLM 模糊匹配有单一高置信候选 → 置信度 medium，标注供人确认
  └─ 多个候选或无匹配 → 置信度 low，标注 UNCERTAIN
```

---

### 阶段 2：方案设计

| | 内容 |
|------|------|
| **Agent 做什么** | 查 ColumnBindingTable → 确定每个指标的 G3/G2 可用性 → 选择数据源层级 → 设计 JOIN 路径 → 确定增量策略 |
| **Agent 输出什么** | 设计方案：`{主表, JOIN 路径, 层级选择, 增量策略, 每项附理由}` |
| **人看什么** | 层级选择是否合理（G3 优先？）？JOIN 路径是否在白名单中？降级理由是否充分？ |
| **常见问题** | "为什么选了 G2 而不是 G3？"——Agent 在决策说明中解释了原因，人确认即可 |

**Agent 查的数据源**（只读，Agent 不能修改）：
- `meta.metric_definitions`——指标注册表
- ColumnBindingTable——指标→物理列映射
- JOIN 白名单——已审批的表关联路径
- 数据库设计文档——表结构事实源

---

### 阶段 3：代码生成

| | 内容 |
|------|------|
| **Agent 做什么** | 根据设计方案**独立生成两份代码**——SQL 草案（DuckDB）+ Spark DSL 草案（PySpark）→ 生成 Python 管道（如需 DAG）→ 生成测试用例 → 生成调度配置 → **所有表/字段引用标注来源** |
| **Agent 输出什么** | 双份代码包：`{SQL 脚本, Spark DSL（PySpark）脚本, Python 管道, 测试代码, 调度配置}` |
| **人看什么** | 两份代码逻辑是否等价？来源标注是否准确？风格是否合规？ |
| **重要约束** | 两份代码必须**独立生成**——LLM 分别实现同一逻辑，不能一份复制另一份。不同表达方式（声明式 vs 过程式）和不同引擎（DuckDB vs Spark）使两份代码很难犯相同错误——这是交叉验证有效的前提 |
| **常见问题** | "SQL 和 PySpark 实现的是同一个逻辑吗？"——交叉验证（阶段 4）会用结果回答 |

**SQL 草案示例**（DuckDB）：
```sql
-- 来源：gold_database_design.md §3.2 dws_daily_trip_summary 表定义
-- 指标来源：meta.metric_definitions — trip_count, total_fare_amount
SELECT
    trip_date,        -- gold.dws_daily_trip_summary.trip_date (DATE)
    trip_count,       -- gold.dws_daily_trip_summary.trip_count (INTEGER)
    total_fare_amount -- gold.dws_daily_trip_summary.total_fare_amount (DECIMAL)
FROM gold.dws_daily_trip_summary
WHERE trip_date BETWEEN ? AND ?
ORDER BY trip_date ASC
```

**Spark DSL 草案示例**（PySpark，同一需求）：
```python
# 来源：gold_database_design.md §3.2 dws_daily_trip_summary 表定义
# 指标来源：meta.metric_definitions — trip_count, total_fare_amount
from pyspark.sql import SparkSession
from pyspark.sql.functions import col

df = (
    spark.table("gold.dws_daily_trip_summary")
    .select("trip_date", "trip_count", "total_fare_amount")
    .filter(col("trip_date").between(start_date, end_date))
    .orderBy("trip_date")
)
```

### 阶段 4：自动验证

| | 内容 |
|------|------|
| **Agent 做什么** | 确定性规则引擎（不调 LLM）执行 7 项检查——表存在性、安全黑名单、表权限、JOIN 白名单、样本执行、结果质量、**交叉验证** |
| **Agent 输出什么** | 验证报告：`{检查项, 结果(通过/FAIL/WARN), 详细信息, 交叉验证对比结果}` |
| **人看什么** | 有没有 FAIL？WARN 的具体内容？交叉验证是否一致？不一致时的差异在哪里？ |
| **重要** | 这是 Agent 的最后一步。之后的阶段 5 只是整合材料，阶段 6 就交给人了 |

**7 项检查详解**：

| # | 检查项 | 怎么检查 | 为什么重要 | 失败等级 |
|---|--------|---------|-----------|---------|
| 1 | 表/字段存在性 | 在开发库执行 DESCRIBE | 拦截 LLM 幻觉编造的表/字段 | FAIL |
| 2 | 安全黑名单 | 正则扫描 SQL 中的 INSERT/UPDATE/DELETE/DROP 等 16 个关键字 | 防止破坏性 SQL 进入审查 | FAIL |
| 3 | 表访问权限 | 检查 FROM/JOIN 中的表是否在白名单中（不含 bronze/silver） | 防止访问原始和中间数据 | FAIL |
| 4 | JOIN 白名单 | 检查所有 JOIN 路径是否在已审批白名单中 | JOIN 路径是人审批过的，不可随意扩展 | FAIL/UNCERTAIN |
| 5 | 样本执行（SQL） | 在开发库用 LIMIT 1000 执行 SQL | 拦截语法错误、类型不匹配等运行时错误 | FAIL |
| 6 | 结果质量 | 检查行数、空值率、列完整性 | 数据质量过低时提醒人注意 | WARN |
| 7 | **交叉验证** | SQL 和 PySpark 两份代码分别对 dev 库执行，对比行数/列名/值分布/抽样行 | 两份独立实现相互校验——不同实现很难犯相同逻辑错误 | WARN |

**交叉验证的三种结果**：

```
SQL 草案     ──→ dev 库执行 ──→ 结果集 A
PySpark 草案 ──→ dev 库执行 ──→ 结果集 B
                                   ↓
              对比：行数 | 列名 | 值范围 | 抽样行
                                   ↓
    ┌─ ✅ 一致 → 交叉验证通过，两份代码样本集逻辑一致（不代表全量一致或业务正确）
    ├─ ⚠️ 不一致 → WARN，人审时必须调查（至少一份代码有逻辑错误）
    └─ ⏸️ 仅 SQL 执行（Spark 不可用）→ 标记"待 Spark 环境执行"，无法提供双引擎背书
```

---

### 阶段 5：材料输出

| | 内容 |
|------|------|
| **Agent 做什么** | 将前 4 个阶段的产出整合为审查材料包 |
| **Agent 输出什么** | 完整审查材料：双份代码 + 验证报告 + 交叉验证报告 + 设计决策 + 不确定项清单 |
| **人看什么** | 这是人审的"案卷"——所有信息汇总在一处 |

**审查材料包结构**：
```
审查材料
├── 1. 需求分析报告           ← 阶段 1 输出
├── 2. 设计方案               ← 阶段 2 输出（含层级/JOIN/策略决策和理由）
├── 3. 代码                   ← 阶段 3 输出（含来源标注）
│   ├── query.sql             ← SQL 草案（DuckDB）
│   ├── query.py              ← Spark DSL 草案（PySpark，同一逻辑）
│   ├── pipeline.py           ← Python 管道（如有 DAG）
│   └── test_query.py         ← pytest 测试
├── 4. 自动验证报告           ← 阶段 4 输出（含 7 项检查结果）
│   └── 交叉验证对比           ← SQL vs PySpark 结果对比
└── 5. 不确定项清单           ← Agent 标注"需人确认"的所有事项
```

---

### 阶段 6：人审上线

| | 内容 |
|------|------|
| **人做什么** | 审查全部材料 → 逐条确认不确定项 → 判断是否批准上线 |
| **人输出什么** | 决策：批准 / 修改后重审 / 拒绝 |
| **Agent 做什么** | 等待。不主动干预 |

**人的审查清单**：

| # | 检查项 | 关注什么 |
|---|--------|---------|
| 1 | 指标映射 | Agent 理解的指标是否对应用户的需求？ |
| 2 | 方案合理性 | 层级选择（G3/G2）是否合理？JOIN 路径是否必要？ |
| 3 | 代码质量 | SQL 和 PySpark 两份代码的逻辑是否正确？是否等价？性能是否可接受？ |
| 4 | 来源追溯 | 表/字段引用的来源标注是否准确？ |
| 5 | 测试结果 | 样本执行是否通过？结果质量是否有 WARN？ |
| 6 | **交叉验证报告** | SQL 和 PySpark 结果是否一致？不一致时差异在哪？哪份代码正确？ |
| 7 | 不确定项 | 逐条确认 Agent 标注的所有 UNCERTAIN 项 |

**交叉验证如何影响决策**：

| 交叉验证结果 | 人的处理方式 |
|-------------|------------|
| ✅ 一致 | 两份代码样本集逻辑一致（LIMIT 1000），重点审查业务合理性、方案选择、来源标注。不代表全量数据一致或生产就绪 |
| ⚠️ 不一致 | 必须定位差异原因——逐项对比行数/列名/值，判断哪份代码正确（或两份都错），要求 Agent 修改后重新提交 |
| ⏸️ 仅 SQL 执行 | SQL 侧正常审查，Spark DSL 代码审阅逻辑，标注"上线前需在 Spark 环境验证" |

**决策选项**：
- **批准上线**：双份代码合入 Git → CI/CD 或手工部署到生产（SQL→DuckDB/Spark SQL, PySpark→Spark 集群）
- **修改后重审**：指出需要修改的地方 → Agent 修改 → 重新走阶段 4-6
- **拒绝**：需求无法通过现有数据满足，或方案存在根本性问题

## 三、Agent 的"自测"能力

### 3.1 Agent 为什么需要自测

一个只生成代码但不测试的 Agent 对人没有价值——人需要自己去跑测试来验证代码是否正确，这等于把工作转移给了人。

Agent 的自测能力解决了这个问题：代码生成后立即用开发库的样本数据执行，低级错误在 Agent 阶段就被拦截，人审时可以专注于业务逻辑和方案合理性——这才是人最擅长、自动化最难替代的判断。

### 3.2 自测的范围和局限

**自测能发现的**：语法错误、表/字段不存在、类型不匹配、安全违规、JOIN 路径不合法、结果为空、空值率过高、**SQL 与 PySpark 逻辑不一致**。

**自测不能发现的**：
- 两份代码犯相同的业务逻辑错误（如 SQL 和 PySpark 都用了相同的错误 JOIN 条件——但这种情况极其罕见，因为两种表达方式不同）
- 方案选择不当（如应用 G3 却降级到了 G2）
- JOIN 基数正确性——LIMIT 1000 样本无法验证全量 JOIN 行为
- 生产性能——样本执行不反映生产环境资源消耗
- 分区/幂等/回滚正确性——当前不验证部署链路
- 全量数据行为——LIMIT 1000 不代表全量

**交叉验证的补充作用**：传统的单份代码测试只能发现"代码跑不跑得通"。交叉验证额外发现"两份代码算得一样不一样"——这是一种传统测试做不到的逻辑一致性检查。但注意：即使交叉验证 CONSISTENT（两份代码样本一致），也不代表全量数据一致或业务正确。

**这正是人审存在的理由**——自动验证覆盖不了的，靠人的判断力补上。

## 四、快速参考

| 问题 | 答案 |
|------|------|
| Agent 能不能直接部署到生产？ | 不能。Agent 没有生产凭据。 |
| 为什么生成两份代码？ | SQL + PySpark 独立实现，交叉验证——两份代码很难犯相同逻辑错误。 |
| 交叉验证不一致怎么办？ | WARN，人审时必须调查——至少一份代码有错。 |
| 人审通过后谁部署？ | 人通过 PR 合入 → CI/CD 或手工部署。 |
| Agent 不确定某个映射怎么办？ | 标注 UNCERTAIN，列出候选，等人确认。 |
| 自动验证 FAIL 了怎么办？ | Agent 修复后重新提交，不会进入人审。 |
| 旧架构的代码还在吗？ | 在。`scripts/pipeline/` 中的代码保留为防线 2 的规则引擎。 |
| Spark 环境不可用怎么办？ | PySpark 代码仍生成和审查，交叉验证标记"待执行"，人上线前在 Spark 环境手动验证。 |
